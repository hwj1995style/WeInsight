# 项目协作规则

除上级目录或会话中已生效的规则外，本项目补充以下 Git 标准动作。

## Git 交付标准动作

- 每个独立任务使用一个 `codex/<task-name>` 分支，不直接在 `main` 上开发。
- 开始新任务前确认工作区干净，并从最新的 `origin/main` 创建任务分支；不同任务的改动不混入同一个分支或 PR。
- 用户要求“提交并推送”时，默认依次执行：相关测试、检查变更范围、提交、推送当前任务分支、创建或更新 Draft PR。此授权不包含合并到 `main`。
- 用户要求“合并到 main”时，默认依次执行：确认 PR 和检查状态、将 PR 转为 Ready、合并、同步本地 `main`、删除本地和远程的已合并任务分支、执行 `git fetch --prune` 并验证结果。
- PR 合并后，分支清理属于标准收尾动作，无需保留已合并的 `codex/*` 任务分支。
- 绝不删除未合并分支、受保护分支、包含未推送提交的分支，或其他 worktree 正在使用的分支；遇到这些情况应停止清理并说明原因。
- 不使用 force push，不覆盖远程 `main`，不以删除分支代替正常合并。
- 长期分支仅在有明确持续维护目的时保留，并在文档或 PR 中注明负责人和用途。

详细检查清单见 `docs/operations/Git分支交付与清理标准流程.md`。
