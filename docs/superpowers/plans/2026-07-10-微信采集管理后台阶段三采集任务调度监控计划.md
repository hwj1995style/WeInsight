# WeInsight 管理后台阶段三采集任务调度监控 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 交付可创建、停止、软删除和监控的持久化采集任务，以及安全运行真实 Runner 的采集 Worker、微信健康探测、结构化日志和 SSE。

**Architecture:** MySQL 保存任务、next_run、运行租约和事件，APScheduler 只负责周期唤醒。采集 Worker 主线程执行 RPA，后台调度线程刷新心跳和轻量健康检查；Web 只写任务和读取事件。

**Tech Stack:** Python 3.12、APScheduler 3.10.4、SQLAlchemy、PyMySQL、FastAPI、HTMX、ECharts、pytest、现有 GroupPollingRunner/ArticlePollingRunner/Fake RPA。

## Global Constraints

- 必须先完成阶段二计划及控制面迁移。
- 任务时区固定为 `Asia/Shanghai`。
- 公众号任务间隔至少 600 秒，微信群任务间隔至少 30 秒。
- 一个任务只能绑定同一链路的一个或多个目标。
- 同一目标不能属于时间范围重叠的 scheduled、active、stop_requested 任务。
- 任务级 `interval_seconds` 是托管调度唯一频率口径；名单默认频率只用于表单预填。
- 全局先领取 group，再领取 article；微信 UI 同时只能有一个 owner。
- 漏掉多个周期时只合并补跑一次，不创建历史运行风暴。
- 停止必须在目标之间或公众号安全检查点生效，不强杀 RPA。
- 微信健康非 ok 时不领取新运行；恢复后仍只补跑一次。
- 托管 Worker 存活时，旧 scheduler 和未获取 UI 锁的人工真实采集命令不得运行。
- Web 不得直接调用本计划中的 Worker、Runner 或 RPA 工厂。

- [ ] **Step 4: 实现事务 Repo**

create 必须在同一事务中按 `source_id` 升序调用 `MysqlSourceWriteGuard.lock_for_job_target(connection, source_type, source_id)`，对每个配置行执行 `FOR SHARE` 锁并校验 enabled，再检查 overlap、插入 job、插入所有 target snapshot；禁止先在事务外读取 enabled 再写 target，也禁止按请求原始顺序加多行锁。这样任务目标创建与名单停用、改名、删除使用同一行锁协议并保持稳定锁顺序。状态更新 SQL 必须包含 `WHERE id=:id AND version=:expected_version` 并执行 `version=version+1`。

公开接口：

```python
@dataclass(frozen=True)
class CollectorTickResult:
    status: str
    run_id: int | None
    pipeline_type: PipelineType | None
    executed_target_count: int


```

`ManagedCollectorWorker` 的精确方法签名为 `run_tick(self, now: datetime) -> CollectorTickResult`、`execute_run(self, run: ClaimedCollectionRun, now: datetime) -> CollectorTickResult`、`heartbeat(self, now: datetime) -> None`。

每个 job target 使用单目标 provider 创建现有 Runner，确保 group 停止可以在一个群完成后检查。target result 映射到 target_run；全部成功为 success，部分失败为 partial_success，全部失败为 failed，停止剩余目标为 cancelled。

所有阶段三新增或改造的采集历史 writer 必须沿用阶段二锁协议：在写 raw、route cache、cursor/progress、collect log 的同一事务内，先调用 `MysqlSourceWriteGuard.lock_for_history_write(connection, source_type, source_name)`。现有 MySQL writer 已内置该 guard；若 Worker 新增绕过这些 Repo 的直写路径，必须在该直写事务中显式调用 guard，禁止先锁后另开事务写入。

- [ ] **Step 6: 实现 Worker 进程入口**

`collector_main.py` 使用 `BackgroundScheduler` 注册 heartbeat、健康检查和过期 run 恢复，真实 RPA 由主线程显式循环执行：

```python
background = BackgroundScheduler(timezone="Asia/Shanghai")
background.add_job(worker.heartbeat_now, "interval", seconds=config.workers.heartbeat_seconds, max_instances=1, coalesce=True)
background.add_job(worker.health_check_now, "interval", seconds=config.wechat.check_login_interval_seconds, max_instances=1, coalesce=True)
background.add_job(worker.recover_expired_now, "interval", seconds=config.workers.heartbeat_seconds, max_instances=1, coalesce=True)
background.start()
try:
    while not stop_event.is_set():
        worker.run_tick(datetime.now())
        if args.once:
            break
        stop_event.wait(config.workers.schedule_tick_seconds)
finally:
    background.shutdown(wait=False)
```

入口解析 `--config` 和 `--once`。`runtime_factory.py` 在 `collector_mode=fake` 时只构造现有 Fake RPA，在 real 时构造 wxauto Client；模式值无效立即退出 2。SIGINT/CTRL_C 设置 stop_event，停止领取新运行并等待当前安全检查点。
