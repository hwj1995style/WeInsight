# 微信信息采集分析系统第九阶段真实账户POC执行 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在第八阶段真实 POC 前置材料齐备后，执行真实账户受控 POC 执行与 3 账号小规模扩容，并给出是否继续扩到最多 20 个的判断。

**Architecture:** 第九阶段从 1 个实际授权公众号/订阅号和 1 个实际授权核心群的真实执行开始，先完成环境核验、单链路执行、双链路交错小窗口，再按验收结果决定是否扩到 3 个账号。所有微信 PC UI 操作仍通过 `wechat_ui_lock` 串行交错，群链路优先，article 链路可中断、可关闭、可回滚。

**Tech Stack:** Python 3.12、pytest、MySQL 8.4、微信 PC 4.1.8.107、PowerShell、Markdown 运维记录、现有 `app.main` CLI。

## Global Constraints

1. 微信 PC 版本固定为微信 PC 4.1.8.107。
2. 微信自动更新必须保持关闭。
3. 第一轮只允许 1 个实际授权公众号/订阅号。
4. 第一轮只允许 1 个实际授权核心群。
5. 所有真实执行都必须有人值守。
6. 不注册 Windows 计划任务。
7. 不启用无人值守。
8. 不启用后台常驻 article 调度。
9. AI 继续 dry-run，输出必须包含 `model_called=0`。
10. 公众号/订阅号只采集当天发布数据，并按 `article_hash` 去重。
11. 文章正文只允许运行时临时读取，不长期保存文章正文。
12. 所有涉及微信 PC UI 的操作必须获取 `wechat_ui_lock`。
13. 核心群等待超过阈值时，立即暂停 article 链路。
14. CLI、日志、截图和日报不得输出聊天正文、文章全文、HTML 或具体文章链接。
15. 通过 1 个账号和 1 个核心群验收后，才允许扩到 3 个账号。
16. 3 个账号稳定前，最多 20 个只做后续评估，不做实际扩容。
17. 所有中文文档按 UTF-8 保存，并通过 `rg -n "\?\?" README.md docs app tests sql` 检查。
