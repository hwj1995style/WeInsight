# 结构化日志降噪、对象展示与分级保留实施计划

## 1. 实施顺序

1. 扩展日志只读模型和仓储查询，关联目标名称并增加对象筛选。
2. 更新日志页面，在主区域展示对象名称和对象类型。
3. 删除正常路径中的高频逐目标开始、完成及运行领取事件写入。
4. 增加日志保留配置、分级清理仓储和维护服务。
5. 将清理周期接入 Pipeline Worker，先运行 dry-run，再启用实际清理。
6. 执行全量测试、真实数据库验证和浏览器验收。

## 2. Task 1：对象名称查询与筛选

- 先扩展仓储测试，覆盖微信群、公众号、运行级事件、系统事件和缺失关联降级。
- 为 `RuntimeEvent` 增加目标名称快照。
- 日志 count/data SQL 使用相同 LEFT JOIN 和对象精确筛选条件。
- 增加最近三个月对象筛选选项查询。
- 保持现有三个月读取边界。

## 3. Task 2：日志页面改造

- 筛选区增加对象下拉框。
- 关联对象列优先显示“微信群/公众号 + 名称”。
- ID 链接作为次要信息保留。
- 页面说明区分运行过程与结构化日志用途。
- 增加 HTML 转义、筛选参数和无目标降级测试。

## 4. Task 3：写入降噪

- 修改 Collector Worker 测试，明确正常目标不再写开始/完成事件。
- 修改运行领取路径测试，明确不再写 `collection_run_claimed`。
- 保留运行完成汇总、失败、租约、misfire 和审计事件。
- 回归运行与目标状态、指标和幂等行为。

## 5. Task 4：分级保留

- 扩展配置模型与环境变量覆盖测试。
- 新增保留仓储：dry-run 计数、按规则选取主键、小批量删除和数据库互斥。
- 新增维护服务：周期判断、每类最多 20 批、失败不中断主循环。
- 上海时区日历月边界覆盖月末和闰年。
- 将维护服务接入 Pipeline Worker，不向事件表写清理事件。

## 6. Task 5：验证与发布

- 对当前数据库执行 dry-run，记录各保留规则预计删除数量。
- 校验仅命中事件表，不影响运行实例和业务结果。
- 启用实际清理并核对前后计数、保留边界和重要事件。
- 运行完整 pytest。
- 重启 Web、Pipeline Worker，检查心跳和错误。
- 使用 Chrome + Playwright 验证对象筛选、名称展示、桌面与窄屏、控制台健康并截图。

## 7. 回滚

- 配置关闭事件清理即可停止后续删除。
- 恢复三个高频事件写入即可恢复旧行为。
- 查询和模板可独立回滚，不影响已保存事件。
- 被清理的冗余 INFO 不恢复；运行实例、目标运行和业务结果始终保留。
