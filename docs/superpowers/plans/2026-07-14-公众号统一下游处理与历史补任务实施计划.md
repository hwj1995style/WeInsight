# 公众号统一下游处理与历史补任务实施计划

> **For agentic workers:** Implement task-by-task with TDD. Every task must finish with focused tests, self-review, and a commit before the next task starts.

**Goal:** 在公众号状态页提供统一“下游处理”开关和安全、幂等的历史补处理入口，并在发布后开启九个目标公众号、补齐 2026-07-10 至当天的缺失任务。

**Architecture:** 复用 `downstream_clean_enabled` 作为业务统一开关，保留 `clean_article` 与 `analyze_article` 两阶段任务。新增专用 backfill 仓储在单事务内基于 raw/clean/analysis/task 状态创建或恢复任务，服务层负责范围、日期、模式和一箱蛋排除，Web 层只做管理员表单与 PRG 展示。

**Tech Stack:** Python 3、FastAPI、Jinja2、SQLAlchemy text、MySQL、pytest、Playwright、原生 CSS。

## Global Constraints

- 开关不修改 WeRSS 采集字段 `enabled` 或上游状态。
- “一箱蛋”必须在服务层和仓储层双重排除。
- 默认日期为最近 7 个自然日，单次最多 31 个自然日，结束日期不得晚于 Asia/Shanghai 当天。
- 默认 `missing_only` 只补缺失环节；已有 analysis 即跳过，即使没有报价。
- `force_analyze` 仅处理已有 clean 的文章，不重跑 clean，不提前删除 analysis 或报价。
- `pending`/`running` 不重复排队；`failed` 和结果缺失的 `success` 可恢复 pending。
- 任务幂等依赖现有 `uk_article_task_ref(task_type, ref_type, ref_id)`，所有写入必须兼容并发提交。
- HTTP 请求只排队，不同步执行正文读取、清洗、分析或报价提取。
- 不输出文章正文、URL、完整文章哈希列表、raw JSON 或数据库凭据。
- 所有 POST 必须通过现有管理员 Session、CSRF 和单值字段校验。
