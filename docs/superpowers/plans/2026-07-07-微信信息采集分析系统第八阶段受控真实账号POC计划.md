# 微信信息采集分析系统第八阶段受控真实账号POC Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不扩大风险面的前提下，用 1 个实际授权公众号/订阅号和 1 个核心群完成有人值守的真实 POC 验证，再决定是否扩到 3 个账号和更长窗口。

**Architecture:** 第八阶段只做受控真实账号 POC，不进入无人值守生产。所有真实微信 UI 操作继续通过 `wechat_ui_lock` 串行互斥；群链路优先，article 链路可中断、可关闭、可回滚。AI 仍保持 dry-run，不调用外部模型，不参与真实结论生成。

**Tech Stack:** Python 3.12、pytest、MySQL 8.4、微信 PC 4.1.8.107、PowerShell、Markdown 运维文档、现有 `app.main` CLI。

## Global Constraints

1. 第一轮只允许 1 个实际授权公众号/订阅号。
2. 第一轮只允许 1 个核心群。
3. 手动命令触发，有人值守。
4. 不注册 Windows 计划任务。
5. 不启用后台常驻 article 调度。
6. AI 仍保持 dry-run，输出必须包含 `model_called=0`。
7. 公众号/订阅号只采集当天发布数据，并按 `article_hash` 去重。
8. 文章正文只允许运行时临时读取，不长期保存文章正文。
9. 所有涉及微信 PC UI 的操作必须获取 `wechat_ui_lock`。
10. 核心群等待超过阈值时，立即暂停 article 链路。
11. CLI、日志、截图和日报不得输出聊天正文、文章全文、HTML 或具体文章链接。
12. 所有中文文档按 UTF-8 保存，并通过 `rg -n "\?\?" README.md docs app tests sql` 检查。
