# WeInsight 微信采集管理后台总路线 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 按五个独立验收阶段交付局域网单管理员的微信采集管理后台，并保持现有 group/article 双链路、RPA 安全边界和 421 项测试基线。

**Architecture:** Web、采集 Worker、后处理 Worker 三进程共用 MySQL。总路线只规定顺序和阶段闸门；每个阶段的文件、接口、测试、命令和提交步骤由对应子计划定义。

**Tech Stack:** Python 3.12、FastAPI、Uvicorn、Jinja2、HTMX、ECharts、SQLAlchemy、PyMySQL、MySQL、APScheduler、Playwright、pytest、Windows Task Scheduler。

## Global Constraints

- 设计基线：`docs/superpowers/specs/2026-07-10-微信采集管理后台设计.md`。
- 只允许受控局域网访问；生产使用 HTTPS，不提供公网入口。
- 默认管理员为 `admin/admin123456`；首次登录不强制改密，但必须持续显示默认密码提醒。
- 不引入 Redis、Celery、Node 构建链或新的分布式控制面。
- Web 不得导入真实 RPA Client，不得获取 `wechat_ui_lock`，不得在请求线程执行采集或后处理。
- Web 不得查询或返回群 raw 原文、文章正文、HTML、原始文章链接或故障截图内容。
- 故障截图只显示经 `runtime.screenshot_dir` 校验的采集主机名和本机绝对路径。
- 微信群与公众号任务必须分离；所有微信 UI 操作继续串行执行并保持微信群优先。
- 微信群任务间隔不得小于 30 秒；公众号任务间隔不得小于 600 秒。
- 核心群不超过 5 个，公众号/订阅号不超过 20 个。
- 所有数据库变更使用增量迁移；禁止删除、清空或重建现有业务表。
- 每个阶段必须保持当前 421 项测试通过，并增加该阶段的测试。
- 当前工作区没有 `.git` 元数据；执行计划前应在正式 Git checkout 中工作，计划中的 commit 步骤不得在无 Git 元数据目录中伪造成功。

## 子计划与依赖顺序

### Plan 1：Web 与鉴权基础

文件：`docs/superpowers/plans/2026-07-10-微信采集管理后台阶段一Web鉴权基础计划.md`

交付：

1. Web/Session 配置与依赖。
2. 管理员和 Session 数据表。
3. Argon2id 密码、登录锁定、CSRF 和主动改密。
4. FastAPI 应用、登录页、默认密码持续提醒和基础布局。

阶段闸门：未登录请求被阻断；默认账号可登录；首次登录不强制改密；写请求无 CSRF 时被拒绝；现有测试全部通过。

### Plan 2：名单管理与安全只读页面

文件：`docs/superpowers/plans/2026-07-10-微信采集管理后台阶段二名单与安全查询计划.md`

依赖：Plan 1 完成。

交付：

1. 采集任务控制面全部数据表。
2. 群和公众号配置按 ID 管理。
3. 新增、编辑、启用、停用、安全删除和禁止危险改名。
4. 仅 clean/analysis 的群明细、文章分析、报价明细和现有日报只读页面。
5. 基础总览查询，不启动真实 Worker。

阶段闸门：被任务引用的名单无法删除；Web 数据库查询无群 raw；文章正文、原始链接和截图内容不出现在响应；分页上限生效。

### Plan 3：采集任务、调度、健康与运行日志

文件：`docs/superpowers/plans/2026-07-10-微信采集管理后台阶段三采集任务调度监控计划.md`

依赖：Plan 2 完成。

交付：

1. 任务状态机、时间窗口、跨午夜和漏跑合并算法。
2. 任务创建、重叠校验、停止、软删除和运行租约。
3. 采集 Worker、群优先、公众号检查点停止和 UI 锁互斥。
4. 微信定时健康探测、Worker 心跳和托管模式互斥。
5. 运行实例、目标进度、结构化事件、SSE 和任务页面。

阶段闸门：Fake RPA 下重复运行和并发 UI 锁均为 0；停止只在安全检查点生效；微信异常阻止新运行；SSE 可断点续读。

### Plan 4：后处理与日报双触发

文件：`docs/superpowers/plans/2026-07-10-微信采集管理后台阶段四后处理与日报计划.md`

依赖：Plan 3 完成。

交付：

1. 日报生命周期字段和日报请求表。
2. 后处理 Worker 自动消费 group/article clean、parse、analysis。
3. 手动生成当日 provisional 日报。
4. 次日 `00:10` 生成前一自然日 final 日报。
5. 日报请求、状态、图表和安全 Markdown 下载页面。

阶段闸门：手动生成可幂等更新；自动补偿不产生重复请求；单对象失败不阻止其他对象；后处理失败不修改采集任务或另一链路。

### Plan 5：Windows 部署、回归与受控上线

文件：`docs/superpowers/plans/2026-07-10-微信采集管理后台阶段五部署与受控上线计划.md`

依赖：Plan 4 完成。

交付：

1. Web、采集 Worker、后处理 Worker 的 Windows 启停与任务计划脚本。
2. HTTPS、Windows 防火墙、生产 MySQL 最小权限和日志目录手册。
3. 旧 scheduler 注销与托管模式切换。
4. 浏览器 E2E、Fake RPA 全量验收、单群单公众号真实 POC。
5. 自动调度和最多 5 群/20 公众号的分批扩容闸门。

阶段闸门：只有真实 POC 验收 Go 后才能自动调度；只有单目标和小窗口稳定后才能扩容；任一 UI 锁、健康、失败率或核心群等待指标不合格立即回退。

## 执行原则

1. 严格按 Plan 1 → Plan 5 顺序执行。
2. 每个子计划完成后运行该计划的局部测试和完整 `pytest -q`。
3. 未通过阶段闸门时停止，不跳到下一阶段。
4. 真实微信 UI 验证只在 Plan 5 人工值守步骤执行。
5. 计划实施过程中如需改变数据模型、权限边界、部署方式或测试策略，先更新设计文档并重新确认。
