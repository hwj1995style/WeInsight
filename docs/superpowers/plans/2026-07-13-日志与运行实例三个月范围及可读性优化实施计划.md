# 日志与运行实例三个月范围及可读性优化实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 日志与运行实例只允许查看上海时区最近 3 个日历月的数据，并将日志重构为中文运维摘要表。

**Architecture:** `RuntimeMonitorService` 负责从可注入时钟计算唯一可见起点并执行详情访问控制；MySQL 仓储在分页前应用该起点；SSE 在建立连接前校验运行并在每次轮询中限制事件时间。日志原始数据保持不变，服务层生成安全、可测试的运维展示模型，模板只负责渲染。

**Tech Stack:** Python 3.12、FastAPI、SQLAlchemy、MySQL 8、Jinja2、原生 `<details>`、pytest、Playwright CLI。

## Global Constraints

- 仅限制“日志”和“运行实例”，不限制任务列表。
- 以 `Asia/Shanghai` 当前时刻向前滚动 3 个日历月，不使用固定 90 天。
- 日志列表、运行列表、运行详情、详情事件和 SSE 使用同一边界。
- 超期运行详情固定显示“该记录已超出可查看范围”，且不泄露记录内容。
- 历史数据保留，不执行删除、归档或迁移。
- 日志采用运维摘要表；英文事件、阶段、Worker ID 和安全指标原文默认折叠。
- 未识别事件显示“未分类事件”，页面不得因未知类型或非法指标失败。


映射表和指标解析放在服务模块的纯函数中，继续调用现有安全输出函数。`technical_details` 只包含已经脱敏和截断的 `event_type/stage/worker_id/metrics_summary`，不携带未清洗原值。
