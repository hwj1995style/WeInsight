# WeInsight 采集任务启动与停止控制设计

## 1. 背景与现状

采集任务列表目前只提供“查看详情”，管理员必须进入详情页才能停止任务；详情页已有安全停止入口，但没有将已停止任务恢复调度的能力。任务状态机已经区分 `scheduled`、`active`、`stop_requested`、`stopped`、`completed` 和 `deleted`，采集 Worker 也会在领取前、目标之间及链路阶段检查停止请求，因此新增控制应复用现有状态机，不能直接启动线程或强杀 RPA。

本次功能在列表和详情页增加可见、可审计的启动与停止操作，使人工任务可以在保留原时间窗口、频率、目标快照和运行历史的前提下暂停并恢复。

## 2. 目标与非目标

### 2.1 目标

- 在采集任务列表直接显示合法的“启动”或“停止”按钮。
- `scheduled` 和 `active` 任务继续使用现有安全停止语义。
- `stopped` 任务可以恢复调度，并重新计算下一次合法运行时间。
- 启动、停止均使用 CSRF、乐观锁版本和管理员身份审计。
- 系统托管的公众号全局任务不允许人工启动、停止或删除。
- 详情页与列表页使用同一后端状态转换，不产生两套逻辑。

### 2.2 非目标

- “启动”不表示立即创建一次运行实例，也不跳过任务时间网格。
- 不支持从 `stop_requested` 撤销停止；必须等待 Worker 到达安全检查点并转为 `stopped`。
- 不支持重新启动 `completed`、`deleted` 或已超过有效期且没有未来运行时间的任务。
- 不修改任务目标、有效期、每日窗口、频率或历史运行数据。
- 不启动或停止采集 Worker 进程本身。

## 3. 影响范围

- 任务服务协议与状态校验：`app/services/collection_job_service.py`。
- MySQL 任务仓储与审计事件：`app/storage/collection_job_repo.py`。
- 审计事件保留和中文展示：`app/storage/event_retention_repo.py`、`app/services/runtime_monitor_service.py`。
- Web 路由和表单校验：`app/web/routes/jobs.py`。
- 任务列表与详情模板：`app/web/templates/jobs/index.html`、`detail.html`。
- Repository、Service、Web、事件保留和监控展示测试。
- 不需要数据库表结构迁移。

## 4. 状态与权限模型

### 4.1 状态转换

```text
scheduled --停止--> stopped
active --无运行实例时停止--> stopped
active --有运行实例时停止--> stop_requested --安全检查点--> stopped
stopped --启动--> scheduled（尚未到整体开始时间）
stopped --启动--> active（已进入整体有效期）
```

启动时根据原 `ScheduleSpec` 计算从当前时刻开始的下一次合法运行时间：

- 当前时间早于 `effective_start_at` 时，恢复为 `scheduled`。
- 当前时间已进入有效期且仍存在未来时间网格时，恢复为 `active`。
- 有效期已结束或不存在未来运行时间时拒绝启动，任务保持 `stopped`。
- `scheduled` 或 `active` 的重复启动请求保持幂等，不重复写事件或递增版本。
- `stop_requested`、`completed`、`deleted` 的启动请求返回状态冲突。

### 4.2 权限与并发

- 继续使用管理员 Session 与 CSRF 校验。
- 启动和停止表单必须提交任务 `version`，数据库在行锁下校验乐观锁。
- `managed_key = article_global` 的系统任务在 Service、Repository 和模板三层禁止人工变更。
- 状态转换和审计事件写入同一数据库事务。

## 5. 后端实现方案

### 5.1 启动任务

新增 `CollectionJobService.start_job` 与 Repository `start_job`：

1. 校验任务 ID、版本、操作者和时区时间。
2. Service 拒绝系统托管任务。
3. Repository 使用 `SELECT ... FOR UPDATE` 锁定任务并再次检查系统托管标识。
4. 仅 `stopped` 状态进入恢复逻辑；重复启动活动任务直接返回当前状态。
5. 依据原调度配置计算下一合法时间并更新 `status`、`next_run_at`。
6. 清空上一次 `stop_requested_at`、`stop_requested_by`，递增 `version`。
7. 写入 `job_started` 管理员审计事件。

### 5.2 停止任务

复用现有 `request_stop`：

- 尚未实际运行的 `scheduled` 或 `active` 任务直接转为 `stopped`。
- 存在运行实例时转为 `stop_requested`，等待 Worker 安全检查点。
- 不增加强制停止或进程终止入口。

### 5.3 列表数据与动作返回

- `CollectionJobSummary` 增加 `managed_key`，模板不再通过任务名称猜测系统任务。
- 新增 `POST /jobs/{job_id}/start`，与停止、删除共用严格表单解析和错误映射。
- 列表表单可提交固定值 `return_to=list`，成功后返回任务列表；详情页保持原有详情反馈。
- `return_to` 只接受 `list` 或省略，不允许任意 URL，避免开放重定向。

### 5.4 审计事件

- `job_started` 纳入长期保留的任务审计事件。
- 运行监控页面将其显示为“已启动采集任务”。

## 6. 前端影响

### 6.1 任务列表

- 非系统任务处于 `scheduled`、`active` 时显示“停止”。
- 非系统任务处于 `stopped` 时显示“启动”。
- `stop_requested`、`completed`、`deleted` 只显示“查看详情”。
- 系统托管任务只显示“查看详情”。
- 操作按钮使用现有 `row-actions`、紧凑按钮和危险操作样式，不引入新的视觉体系。

### 6.2 任务详情

- `stopped` 状态同时显示“启动任务”和“删除任务”。
- 启动说明明确为恢复原调度计划，而不是立即执行。
- 停止中继续显示安全检查点说明，不提供撤销按钮。

## 7. 测试与回归方案

### 7.1 自动化测试

- Service 校验启动参数并拒绝系统托管任务。
- Repository 验证 `stopped` 到 `scheduled/active` 的转换、下一运行时间、版本和事件。
- Repository 验证重复启动幂等、非法状态、版本冲突和过期调度。
- 列表按状态和 `managed_key` 只渲染合法按钮。
- 列表启动/停止成功后返回列表，并提交唯一 CSRF 与版本字段。
- 详情页停止、启动、删除动作组合符合状态机。
- 启动路径拒绝 GET、重复字段、未知字段、伪造 `return_to` 和系统任务。
- `job_started` 纳入审计保留与中文事件名称。
- 运行现有任务创建、调度领取、安全停止、软删除和全量 Web 回归。

### 7.2 浏览器验收

目标流程：`/jobs` → 停止一个人工活动任务 → 状态变为停止中或已停止 → 在 `stopped` 后点击启动 → 返回列表并显示运行中及下一运行时间。

同时检查：

- 页面身份、非空渲染、无框架错误覆盖层。
- 控制台无相关 error/warning。
- 系统任务没有人工启动/停止按钮。
- 桌面和移动视口下操作列不裁切、不重叠。

## 8. 风险与分阶段落地建议

### 8.1 风险

- 将“启动”误解为立即执行；通过按钮附近说明和下一运行时间反馈降低歧义。
- 停止请求尚未完成时再次启动可能与正在运行的 RPA 冲突；因此禁止 `stop_requested -> active`。
- 任务有效期结束后没有未来运行时间；后端必须拒绝并保留原状态。
- 列表数据若缺少 `managed_key` 可能暴露系统任务控制；改为数据库字段驱动并保留 Repository 二次校验。
- 并发管理员操作可能提交旧版本；使用现有乐观锁并刷新当前详情。

### 8.2 落地顺序

1. 先补 Service、Repository 和 Web 失败测试。
2. 实现原子启动状态转换及审计事件。
3. 增加列表和详情按钮。
4. 运行专项与全量测试。
5. 在本地开发库用人工任务完成停止、启动闭环验收。

## 9. 验收标准

- 管理员可在任务列表直接停止活动人工任务。
- 已停止且仍在有效期内的任务可恢复调度，并显示正确下一运行时间。
- 停止不强杀 RPA，启动不越过原时间网格。
- 系统任务、停止中、已完成、已删除和过期任务不会被非法启动。
- 每次有效启动/停止都有管理员审计事件。
- 任务目标、运行历史、采集结果和权限规则无回归。
- 专项、全量自动化测试和真实浏览器验收通过。
