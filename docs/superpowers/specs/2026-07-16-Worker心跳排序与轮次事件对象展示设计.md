# Worker 心跳排序与轮次事件对象展示设计

## 背景与现状

Worker 心跳页面当前按 `worker_type、hostname、worker_id` 正序展示。Worker ID 内含进程号和随机后缀，历史进程可能长期排在当前在线进程之前，导致用户误以为心跳没有更新。

结构化日志的关联对象仅通过 `target_run_id` 关联单个任务目标。`misfire`、`collection_run_finished` 等轮次级事件本身没有 `target_run_id`，即使运行实例包含多个公众号，页面也只能显示“本轮全部目标”。此外，`misfire` 等事件摘要映射已存在于待合并分支，尚未进入 `main`，因此线上仍显示“未分类事件”。

## 目标与非目标

### 目标

- Worker 页面优先展示在线进程，同一状态下按最近心跳时间倒序排列。
- 已过期的历史 Worker 继续保留用于排查，但统一排在在线 Worker 之后。
- 轮次级事件展示所属链路、本轮目标数量，并允许查看本轮具体公众号或微信群名称。
- 目标级事件继续直接展示唯一的具体采集对象。
- 将现有事件摘要映射和日报遗留任务修复一并合并到 `main`。

### 非目标

- 本次不删除历史 Worker 心跳记录，也不新增自动清理策略。
- 本次不修改采集调度、心跳上报周期或 Worker 存活 TTL。
- 本次不改变事件写入模型，不为轮次级事件补写伪造的 `target_run_id`。
- 本次不修改公众号或微信群的业务名称。

## 影响范围

- 后端查询：Worker 排序、事件列表的轮次目标聚合。
- 服务展示模型：运行事件增加目标数量和目标名称集合。
- 前端页面：关联对象根据目标级或轮次级事件展示不同文案，并为多目标轮次提供展开明细。
- 测试：存储查询、展示模型、事件页面和 Worker 页面回归。
- 发布：合并现有 `codex/fix-report-backlog` 分支并重启 Web、Collector、Pipeline，使全部修复加载生效。

## 数据模型与权限模型

不新增表和字段。轮次目标信息在事件查询时，从既有关系中读取：

`event.run_id -> wechat_collection_job_target_run.run_id -> wechat_collection_job_target.id`

按事件聚合：

- `target_count`：该运行实例的目标数量。
- `target_names`：按任务优先级、名称和目标运行 ID 稳定排序后的名称集合。

事件页面继续沿用现有管理员鉴权与脱敏规则，不扩大普通页面权限，也不展示正文、账号凭据或配置快照。

## 后端实现方案

### Worker 排序

`wechat_worker_heartbeat` 查询使用以下顺序：

1. `within_ttl` 倒序，在线 Worker 优先。
2. `last_heartbeat_at` 倒序，最近心跳优先。
3. `worker_type、hostname、worker_id` 作为稳定排序兜底。

存活判断仍使用查询时传入的 TTL 截止时间，页面显示状态与排序依据保持一致。

### 轮次事件目标聚合

事件列表查询增加按 `run_id` 聚合的目标信息。为避免主查询直接连接一对多目标造成事件重复，使用按运行实例聚合的派生表，每个 `run_id` 只返回一行。

目标级事件：

- `subject_name` 来自现有 `target_run_id` 关联目标。
- 页面直接展示具体名称，不重复展示轮次目标列表。

轮次级事件：

- `subject_name` 为空且 `run_id` 存在时，使用聚合出的 `target_count` 与 `target_names`。
- 摘要显示“公众号/微信群 · 本轮 N 个目标”。
- 多目标名称通过原生 `details/summary` 展开，避免主表被长文本撑开。

系统级事件：

- 同时没有 `target_run_id` 和 `run_id` 时维持“系统事件”，不推断不存在的对象。

### 事件摘要

保留并合并待发布分支中的事件映射，包括 `misfire -> 错过计划已合并执行`，以及 `missed_count -> 错过计划` 指标标签。

## 前端影响

- Worker 页面首屏首先看到当前 Collector 和 Pipeline，历史过期进程位于其后。
- 日志页面目标级事件仍显示“公众号 · 具体名称”。
- 轮次级事件显示“公众号 · 本轮 9 个目标”等摘要，并可展开查看名称。
- 单目标轮次直接显示名称，不增加无意义的展开操作。
- 不新增 JavaScript，使用服务端渲染和原生 HTML 交互，保持无脚本环境可用。

## 测试与回归方案

- 存储测试验证 `_WORKERS` 在线优先和最近心跳倒序。
- 事件查询测试验证聚合连接不会重复事件，并能返回目标数量与有序名称。
- 服务测试验证：目标级、单目标轮次、多目标轮次、系统级事件的展示模型。
- Web 测试验证 Worker 行顺序、轮次目标摘要、展开明细和事件中文摘要。
- 执行相关测试后运行完整 `pytest`。
- 重启本地服务后验证 `/workers` 与 `/events`：页面身份、非空、无错误覆盖层、控制台健康、交互展开及截图。

## 风险与分阶段落地建议

- 事件查询增加运行目标聚合，事件量较大时可能增加数据库负担；聚合严格按 `run_id` 输出一行，并复用现有索引 `wechat_collection_job_target_run(run_id, job_target_id)`，避免事件行膨胀。
- `GROUP_CONCAT` 可能受数据库长度限制；当前单任务目标上限较小，仍在安全范围。展示模型同时保留目标数量，即使名称被截断也不会误报数量。
- 历史 Worker 记录仍会增长，本次仅解决排序误导。后续如需清理，应另行设计保留周期与审计要求。
- 发布时先完成测试和页面验证，再合并 PR；合并后重启 Web、Collector、Pipeline，确认当前进程心跳出现在首行并观察至少一个采集周期。
