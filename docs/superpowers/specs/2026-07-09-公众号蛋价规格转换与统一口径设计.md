# 公众号蛋价规格转换与统一口径设计

## 背景与现状

公众号禽蛋报价抽取首版已经保存原始报价明细，并区分了 `publish_time`、`collect_time` 与 `quote_date`。当前明细仍保留各公众号原始规格，例如 `30斤`、`27.5斤`、`360枚/箱`、`45斤`、`4.90元/筐装` 等。不同规格无法直接比较，日报和后续行情分析需要统一到稳定业务口径。

真实样本显示，部分规则抽取行仍可能把电话、序号、非报价数字或备注数字误识别为价格，因此规格转换阶段必须保留换算方法、置信度和可用性标记，不能把所有换算结果无条件纳入主口径。

## 目标与非目标

目标：

1. 在 `wechat_article_egg_price_item` 上新增统一口径字段。
2. 首版将可换算报价统一为 `元/斤`。
3. 支持三类首批高频规格：
   - 原价已经是 `元/斤`，例如福建闽融 `4.90元`。
   - 整体报价带 `quote_basis=30斤/27.5斤/40斤/45斤`，例如一箱蛋 `133 / 30斤`。
   - 箱装或表格行带重量，例如家美鲜、河北馆陶、信立 `220 / 45斤`。
4. 对鸡蛋主口径输出 `include_in_standard_price`，只纳入 `product_family=chicken_egg` 且换算结果处于合理区间的行。
5. 保留 `standard_price_*`、`conversion_method`、`conversion_confidence`、`conversion_notes_json`，便于审计和后续规则迭代。

非目标：

1. 不新增独立前端页面。
2. 不做 OCR。
3. 不合并重复市场报价，不做地区均价。
4. 不改变原始提取字段，不覆盖 `price_text`、`weight_text`、`quote_basis`。
5. 不对鸡蛋以外产品建立主口径；非鸡蛋可有标准价字段，但 `include_in_standard_price=0`。

## 数据模型或权限模型

新增字段位于 `wechat_article_egg_price_item`：

| 字段 | 含义 |
| --- | --- |
| `standard_price_low` | 统一口径价格下限 |
| `standard_price_high` | 统一口径价格上限 |
| `standard_price_unit` | 统一口径单位，首版固定为 `yuan_per_jin` |
| `conversion_basis_weight_low` | 换算分母重量下限，单位斤 |
| `conversion_basis_weight_high` | 换算分母重量上限，单位斤 |
| `conversion_basis_weight_unit` | 换算分母重量单位，首版为 `jin` |
| `conversion_method` | 换算方法 |
| `conversion_confidence` | 换算置信度 |
| `conversion_notes_json` | 换算备注 |
| `include_in_standard_price` | 是否纳入统一鸡蛋价格主口径 |

`conversion_method` 枚举：

1. `already_yuan_per_jin`：价格本身在合理元/斤区间。
2. `quote_basis_weight`：用 `quote_basis` 中的重量换算。
3. `row_weight`：用行级 `weight_low/weight_high` 换算。
4. `unconverted`：无法可靠换算。

主口径合理区间：

1. 首版鸡蛋 `元/斤` 合理区间为 `[2.0, 10.0]`。
2. 换算结果不在合理区间时，保留 `standard_price_*` 为空，`include_in_standard_price=0`，备注 `out_of_reasonable_range`。

## 后端实现方案

1. 在 `article_egg_price.py` 增加 `PriceStandardization` dataclass。
2. `_build_item` 在完成原始价格和重量解析后调用 `_standardize_price(...)`。
3. 标准化规则：
   - 如果 `price_low/price_high` 为空，`conversion_method=unconverted`。
   - 如果价格本身在 `[2, 10]`，视为 `already_yuan_per_jin`。
   - 否则优先用行级重量作为分母；重量范围按低价除高重量、高价除低重量形成价格区间。
   - 行级重量缺失时，从 `quote_basis` 提取 `xx斤` 作为分母。
   - 换算后再次做 `[2, 10]` 合理区间校验。
4. `include_in_standard_price=True` 条件：
   - `include_in_egg_price=True`
   - `product_family=chicken_egg`
   - `standard_price_low/high` 均不为空
   - 换算方法不是 `unconverted`

## 前端影响

当前无前端。后续展示报价明细时，应同时展示原始价格和标准价，并把低置信或未换算行与主口径分开。

## 测试与回归方案

1. 单元测试先红后绿覆盖三类换算和异常过滤。
2. 全量 `pytest -q`。
3. 本地 MySQL 执行迁移，验证新增字段和索引存在。
4. 重跑 `analyze_article` 刷新标准价字段。
5. 查询真实库：统计 `include_in_standard_price=1` 的明细数量、账号分布和标准价范围。

## 风险与分阶段落地建议

风险：

1. 首版规则仍依赖上游抽取质量，误抽数字可能进入候选。
2. 不同产区口径可能使用不同包装或成交场景，统一为元/斤只解决价格单位，不解决交易条件差异。
3. 合理区间 `[2, 10]` 是首版防错阈值，后续应根据历史分布调整。

分阶段：

1. 本阶段先把 `元/斤` 统一口径落库，并带置信度和可用性标记。
2. 下一阶段增加按地区、交易场景、包装政策的主口径过滤。
3. 再下一阶段建设日度标准价聚合表和趋势分析。
