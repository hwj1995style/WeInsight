# 公众号统一下游处理与历史补任务设计

## 背景与现状

WeInsight 的公众号链路分为 WeRSS 采集、正文清洗、文章分析和报价提取。当前 `wechat_public_account_config.enabled` 控制采集，`downstream_clean_enabled` 控制新文章是否创建 `clean_article` 任务；清洗成功后系统再创建 `analyze_article`，分析阶段继续执行报价提取。

截至 2026-07-14，九个目标公众号均已采集文章，但只有湖南三尖农牧公司开启 `downstream_clean_enabled`。因此 2026-07-10 至 2026-07-14 的大量文章只进入 raw 表，没有清洗或分析任务，报价矩阵最新有效日期仍停留在 2026-07-09。现有字段在业务上实际控制整条下游链路，但前端没有统一开关，也没有安全、可审计的历史补处理入口。

## 目标与非目标

### 目标

1. 将现有下游控制统一命名为“下游处理”。
2. 一个账号级开关控制新文章是否自动进入“清洗 → 分析 → 报价提取”。
3. 在公众号状态页支持单账号开启或关闭下游处理。
4. 支持按单个公众号或全部已开启公众号补处理历史文章。
5. 默认只补缺失环节，支持显式强制重新分析。
6. 保证任务创建幂等、并发安全，并提供操作汇总和结构化审计。
7. 在功能发布后开启九个目标公众号并补处理 2026-07-10 至当天的数据。

### 非目标

1. 不修改 WeRSS 的公众号采集开关、采集频率或来源目录。
2. 不把清洗和分析合并成同一个数据库任务类型；统一的是业务开关和管理入口，运行时仍保留两个阶段。
3. 不在 HTTP 请求内直接执行正文读取、清洗、分析或报价提取。
4. 不删除历史清洗、分析或报价结果。
5. 不允许“一箱蛋”进入下游自动处理或历史补处理范围。
6. 不提供超过 31 个自然日的单次补处理。

## 影响范围

1. 公众号状态领域模型：增加统一下游处理状态和可操作标识。
2. 配置仓储：复用 `downstream_clean_enabled`，增加按来源 ID 更新的安全接口。
3. 历史补处理仓储：按范围识别缺失阶段并批量创建或恢复任务。
4. 服务层：负责权限无关的业务校验、日期限制、模式和汇总。
5. Web 路由：新增开关和历史补处理 POST 接口，沿用 Session、CSRF 与管理员鉴权。
6. 公众号状态模板：增加开关、补处理表单、风险提示和结果提示。
7. 结构化事件：记录管理员操作范围和计数，不记录文章敏感内容。
8. 发布操作：开启目标账号并补录历史任务。

## 数据模型与状态语义

### 复用字段

继续使用：

```text
wechat_public_account_config.downstream_clean_enabled
```

前端和服务层统一称为“下游处理”，不新增第二个分析开关，避免产生“清洗开启、分析关闭”这类不完整状态。

### 开启状态

1. 不修改 `enabled` 或 WeRSS 上游状态。
2. 新采集文章创建 `clean_article`。
3. 清洗成功后由现有逻辑创建 `analyze_article`。
4. 分析任务继续执行现有报价提取和分析结果更新。
5. 开启动作本身不隐式补历史文章。

### 关闭状态

1. 新文章只写 raw，不创建新的清洗任务。
2. 已经处于 `running` 的任务继续执行。
3. 已存在的 `pending` 任务不自动取消。
4. 不删除历史清洗、分析或报价数据。

## 页面与交互

### 公众号状态表

新增“下游处理”列：

- 开启：显示“自动清洗与分析”。
- 关闭：显示“仅采集”。
- 每行提供一个开关表单。
- 操作前显示确认文案，明确不会改变 WeRSS 采集状态。
- “一箱蛋”不显示开关，也不能通过接口绕过限制。
- 已停用但仍在 WeRSS 列表中的账号继续显示当前开关状态；开启下游处理不等于重新启用采集。

### 历史补处理表单

公众号状态页顶部增加“补处理历史文章”入口，表单字段：

1. 范围类型：单个公众号 / 全部已开启公众号。
2. 单账号选择：仅允许当前 WeRSS 目录内且非一箱蛋的账号。
3. 开始日期、结束日期。
4. 默认最近 7 个自然日，包含当天。
5. 最大跨度 31 个自然日。
6. 模式：只补缺失环节 / 强制重新分析。
7. 强制重新分析时必须勾选风险确认。

单账号模式允许选择当前下游开关关闭的账号，用于只补历史、不改变未来自动处理；界面必须明确提示这一点。

## 历史补处理规则

### 只补缺失环节

以 raw 文章为事实范围，逐篇判断：

1. 没有 clean 结果：创建或恢复 `clean_article`。
2. 已有 clean 但没有 analysis：创建或恢复 `analyze_article`。
3. 已有 analysis：跳过，即使没有报价明细也不重跑。
4. 对应任务为 `pending` 或 `running`：跳过，不重复排队。
5. 对应任务为 `failed`：重置为 `pending`，重试次数归零，清除安全错误摘要，更新 `next_run_time`。
6. 对应任务为 `success` 但结果缺失：恢复为 `pending`，以结果表事实为准。

### 强制重新分析

1. 仅处理已有 clean 结果的文章。
2. 不重新采集、不重新清洗。
3. `analyze_article` 为 `running` 时跳过。
4. 其他状态创建或重置为 `pending`。
5. 不在提交补任务时删除现有 analysis 或报价明细。
6. worker 成功执行后，复用现有事务逻辑更新 analysis 并替换该文章的报价明细。

## 范围与日期校验

1. 开始日期不得晚于结束日期。
2. 结束日期不得晚于 Asia/Shanghai 当前业务日期。
3. 日期范围包含首尾两天。
4. 最大跨度为 31 个自然日。
5. 默认范围为当天及之前 6 天。
6. 全部范围只选择 `downstream_clean_enabled=1` 且当前仍存在于 WeRSS 目录的目标账号。
7. 单账号必须存在于当前 WeRSS 目录，且不得是一箱蛋。
8. 全部模式即使数据库中存在一箱蛋或历史废弃账号，也必须显式排除。

## 后端接口

### 更新统一开关

```text
POST /sources/articles/{source_id}/downstream-processing
```

参数：`enabled=1|0` 和 CSRF token。按来源 ID 查询并校验账号，禁止一箱蛋，更新成功后使用 Post/Redirect/Get 返回公众号状态页。

### 历史补处理

```text
POST /sources/articles/downstream-processing/backfill
```

参数：

- `scope=single|enabled`
- `source_id`（single 时必填）
- `start_date`
- `end_date`
- `mode=missing_only|force_analyze`
- `confirm_force=1`（force_analyze 时必填）
- CSRF token

请求只负责事务内排队和审计，不同步消费任务。

## 仓储与事务方案

新增专用历史补处理仓储，使用参数绑定和集合 SQL：

1. 先解析并锁定账号范围。
2. 查询日期范围内 raw、clean、analysis 和 process task 的必要状态。
3. 根据模式批量计算新建、恢复和跳过集合。
4. 使用任务唯一键和 `INSERT ... ON DUPLICATE KEY UPDATE` 保证重复提交幂等。
5. 对 `pending`、`running` 不覆盖。
6. 强制分析不删除旧结果。
7. 在同一事务中写结构化安全事件或审计记录。

不新增物理汇总表。若现有任务表唯一约束不足以阻止 `(task_type, ref_type, ref_id)` 重复，应先补充迁移和对应并发测试；不得仅依赖应用层先查后插。

## 操作结果与审计

操作完成后展示：

- 命中文章数
- 新建清洗任务数
- 恢复清洗任务数
- 新建分析任务数
- 恢复或重置分析任务数
- 已有结果跳过数
- 运行中跳过数
- 不符合范围跳过数

审计事件记录管理员 ID/用户名、范围、日期、模式和汇总计数，不记录文章正文、URL、完整文章哈希列表、数据库密码或原始 JSON。

## 权限与安全

1. 沿用管理员 Session 鉴权。
2. 所有变更接口必须通过 CSRF。
3. 只接受单值表单字段，拒绝重复参数。
4. 来源 ID、枚举、布尔值和日期严格校验。
5. 错误响应不回显数据库错误、文章链接或内部 SQL。
6. 一箱蛋在服务层和仓储层双重排除。
7. 模板继续自动转义账号名和提示内容。

## 测试与回归方案

### 领域与服务测试

1. 单账号开启、关闭及不改变采集状态。
2. 一箱蛋不能开启或补处理。
3. 单账号与全部已开启范围。
4. 默认 7 天、最大 31 天、未来日期和反向日期。
5. 缺 clean 创建 clean 任务。
6. 有 clean 缺 analysis 创建 analyze 任务。
7. 有 analysis 无报价默认跳过。
8. failed/success 但结果缺失时恢复 pending。
9. pending/running 幂等跳过。
10. 强制分析只处理已有 clean，且 running 跳过。
11. 强制分析不提前删除旧 analysis 和报价。
12. 重复提交与并发提交不产生重复任务。

### Web 测试

1. 状态页展示统一开关和补处理入口。
2. POST 成功使用 PRG 并显示安全汇总。
3. 未登录、CSRF 缺失、重复字段和非法枚举被拒绝。
4. 强制分析缺少确认被拒绝。
5. 单账号关闭状态下补历史显示明确提示。
6. 模板自动转义，页面不输出正文、URL 或任务错误详情。

### 全量回归

1. WeRSS 采集和十分钟调度不受影响。
2. 新 raw 只在开关开启时创建 clean。
3. clean 成功继续创建 analyze。
4. analyze 继续更新报价明细与报价矩阵。
5. 日志、运行实例和其他后台页面不受影响。

## 风险与分阶段落地

### 风险

1. 历史文章批量读取正文可能增加 WeRSS 和网页回退压力。
2. 强制分析会重建报价，短时间内增加分析 worker 负载。
3. 旧任务状态可能与结果表不一致，必须以结果事实和运行状态联合判断。
4. 批量开启九账号后会形成历史积压，应限制日期并观察队列。

### 落地顺序

1. 实现仓储、服务和幂等测试。
2. 实现开关与补处理 Web 接口。
3. 实现页面交互、结果提示和审计。
4. 发布后开启九个目标公众号的下游处理。
5. 补处理 2026-07-10 至当天，默认 missing_only。
6. 观察 clean/analyze 队列、失败重试和报价矩阵日期推进。
7. 如个别已分析文章需要重新抽价，再按账号和短日期范围使用 force_analyze。

## 验收标准

1. 九个目标公众号可分别控制统一下游处理，一箱蛋始终排除。
2. 开关开启后新文章自动完成清洗、分析和报价提取任务排队。
3. 管理员可按单账号或全部已开启账号补最近 31 天内历史任务。
4. 默认只补缺失环节，重复提交不重复创建任务。
5. 强制重新分析有风险确认且不提前删除旧报价。
6. 页面展示清晰的操作汇总，日志只包含安全审计信息。
7. 2026-07-10 至当天的九账号历史文章完成补处理后，报价矩阵最新日期开始推进。
8. 全量测试通过，正式开发服务完成真实操作验证。
