# 结构化日志降噪、对象展示与分级保留设计

## 1. 背景与现状

结构化日志保存在 MySQL 表 `wechat_collection_job_event`。截至 2026-07-15，开发库共有 12,388 条事件，最近 24 小时新增 5,740 条；其中 INFO 12,290 条，`collection_target_started` 与 `collection_target_finished` 合计 11,026 条，占约 89%。当前前端只限制查看最近 3 个日历月，数据库没有自动清理策略，历史记录会持续增长。

事件已经通过 `target_run_id` 关联目标运行，再通过 `job_target_id` 关联 `target_name_snapshot`，能够识别具体微信群或公众号；但日志列表查询没有返回目标名称，页面只展示任务、运行和目标 ID，因此无法直接判断影响对象。

目标运行表已经保存目标的开始时间、结束时间、状态、读取数、新增数、重复数、跳过数和错误摘要。逐目标“开始/完成”INFO 事件与目标运行记录高度重复，不适合继续作为长期日志保存和默认展示内容。

## 2. 目标与非目标

### 2.1 目标

- 日志列表直接展示微信群或公众号名称，并支持按对象名称筛选。
- 区分“运行状态数据”“运维事件”和“审计事件”，避免把每次状态变化都长期保存为日志。
- 停止新增价值较低的高频逐目标 INFO 事件，以运行实例和目标运行记录承载执行过程。
- 对存量日志实施分级保留和小批量自动清理。
- 默认页面优先展示告警、错误、运行结果和配置变更，技术细节仍可按需展开。
- 保持现有脱敏要求，不把聊天正文、文章正文、完整链接、凭据或联系方式写入日志。

### 2.2 非目标

- 不删除运行实例、目标运行实例、采集结果、清洗结果、分析结果或报价结果。
- 不改变微信群和公众号的采集频率、下游处理频率及开关状态。
- 不建设外部日志平台，不把日志发送到第三方系统。
- 不通过事件表还原完整业务数据；日志仅用于运维、审计和故障定位。

## 3. 影响范围

- 数据库：`wechat_collection_job_event` 的查询和分批清理，不新增业务数据表。
- 采集 Worker：取消冗余的逐目标开始/完成事件写入，保留目标运行表状态更新。
- 运行监控仓储：关联目标名称、支持对象筛选和分级清理。
- 后台服务：日志展示模型增加对象名称与对象类型，新增日志保留服务。
- Web：日志筛选、日志摘要表和技术详情。
- 配置：日志保留天数、清理批次及清理周期。
- 测试：事件写入边界、对象关联、筛选、保留策略、分批清理、浏览器展示和全量回归。

## 4. 数据模型与权限模型

### 4.1 对象名称来源

不在事件表重复增加对象名称字段。查询目标事件时按以下链路读取稳定快照：

`event.target_run_id -> wechat_collection_job_target_run.job_target_id -> wechat_collection_job_target.target_name_snapshot`

展示模型增加：

- `subject_name`：微信群或公众号名称快照；无单一目标时为空。
- `subject_type`：根据任务 `pipeline_type` 映射为“微信群”或“公众号”。
- `subject_label`：有目标时显示对象名称；运行级事件显示“本轮全部目标”；系统事件显示“系统”。

对象筛选使用精确匹配的目标名称快照，避免模糊搜索造成大范围扫描。页面下拉选项来自最近 3 个月可见事件关联的去重目标名称，不读取正文表。

### 4.2 权限

沿用现有管理员会话权限。日志页和清理任务不新增外部接口，不允许前端直接触发删除。技术详情继续经过安全截断和 HTML 转义。

## 5. 事件保存策略

### 5.1 不再写入的高频事件

以下 INFO 事件停止新增：

- `collection_target_started`
- `collection_target_finished`
- `collection_run_claimed`

这些信息已经存在于运行实例、目标运行实例及 Worker 心跳中。取消事件写入不影响任务执行、状态统计或故障恢复。

### 5.2 继续保存的事件

- 运行结果：`collection_run_finished`，作为本轮汇总事件保存。
- 故障：`pipeline_stage_failed`、租约过期、misfire、健康检查异常及其他 WARNING/ERROR。
- 审计：任务创建、停用、删除、配置变更和 WeRSS 目录变化。
- 未知事件：按其实际级别进入默认保留规则，不能因未识别而跳过错误日志。

运行汇总事件必须包含目标总数、成功数、失败数、跳过数等白名单指标；具体目标结果由运行详情页展示，不在日志页重复生成多行。

## 6. 分级保留与自动清理

### 6.1 默认保留期

| 类型 | 默认保留期 | 说明 |
|---|---:|---|
| 历史高频目标开始/完成及运行领取 INFO | 7 天 | 仅用于短期排障，后续不再新增 |
| 普通 INFO 运行结果 | 14 天 | 支持近期运行趋势和问题回看 |
| WARNING / ERROR | 3 个日历月 | 与后台可查看范围一致 |
| 审计事件 | 6 个日历月 | 任务与配置变化需更长追踪周期 |

审计事件白名单包括 `job_created`、`job_stop_requested`、`job_deleted`、`werss_catalog_sync_changed` 及后续显式登记的配置变更事件。事件类型不在白名单时，不得自动获得 6 个月保留期。

### 6.2 配置项

在运行配置中增加以下默认值，并允许通过环境变量覆盖：

- `event_info_retention_days: 14`
- `event_verbose_retention_days: 7`
- `event_warning_error_retention_months: 3`
- `event_audit_retention_months: 6`
- `event_cleanup_interval_hours: 24`
- `event_cleanup_batch_size: 1000`

配置校验限制批次为 100–5000，清理周期不低于 1 小时，各保留期不得小于 1 天或 1 个月。

### 6.3 清理执行方式

- 由常驻 Pipeline Worker 执行保留任务，不依赖管理员打开网页。
- Worker 启动后检查一次，随后最多每 24 小时执行一次。
- 每类规则按主键小批量删除，每批最多 1000 条；单批独立事务提交，避免长事务和大范围锁。
- 单轮最多删除 20 批，仍有积压时留待下一轮，防止清理影响正常处理。
- 清理成功不逐批写事件表，避免“为清理日志再制造日志”；只写进程普通文件日志中的汇总计数。
- 清理失败不得中断清洗和分析主循环，只记录脱敏错误并等待下个周期重试。
- 使用数据库或进程级互斥保证同一时刻只有一个清理者；当前单 Pipeline Worker 模式下以数据库命名锁或等效租约防止未来多实例重复执行。

## 7. 后端查询与服务方案

### 7.1 日志查询

日志数据查询增加对目标运行和任务目标的 LEFT JOIN，返回对象名称快照。无目标关联的运行级或系统级事件仍可正常显示。

筛选条件增加 `subject_name` 精确筛选，并继续叠加现有任务、运行、目标、链路、级别、开始时间、结束时间和最近 3 个月边界。分页总数与数据查询必须使用相同 JOIN 和条件。

### 7.2 展示优先级

日志页面默认不预选单一级别，但按以下规则组织内容：

- ERROR、WARNING 使用醒目标识。
- 运行结果和审计事件使用正常摘要行。
- 技术事件类型、Worker ID、阶段和原始安全指标继续放在折叠详情。
- 目标名称显示在摘要主区域；任务 ID、运行 ID、目标 ID 作为次要链接。

若目标名称关联失败，降级显示“目标 #ID”，不得导致整页失败。

## 8. 前端方案

- 筛选区新增“对象”下拉框，选项显示“微信群 · 名称”或“公众号 · 名称”。
- “关联对象”列优先显示对象名称和对象类型，下一行再显示任务、运行、目标链接。
- 页面说明明确：“运行过程请查看运行实例；日志仅展示运维、故障和审计事件。”
- 对已停止写入的逐目标开始/完成事件，历史 7 天内仍按对象名称正常展示。
- 不在日志页面展示消息正文、文章标题、文章链接或报价原文。

## 9. 测试与回归方案

### 9.1 写入降噪

- 目标正常开始和完成时不写高频事件，但目标运行状态及计数正常更新。
- 运行完成写一条汇总事件。
- 目标或 Pipeline 失败仍写 ERROR，且能关联对象。
- 取消事件写入不会影响任务最终状态和幂等行为。

### 9.2 对象展示与筛选

- 微信群、公众号目标均能从快照关联出正确名称。
- 对象筛选与链路、级别、日期条件共同生效。
- 无目标事件显示“本轮全部目标”或“系统”。
- 目标关联缺失时安全降级为 ID。
- 名称经过转义，不产生 HTML 注入。

### 9.3 保留策略

- 四类事件分别验证边界前、边界时刻和边界后的删除行为。
- 3 个月和 6 个月使用上海时区日历月计算，覆盖月末与闰年。
- 清理每批不超过配置上限，积压超过 20 批时安全停止。
- 清理失败不终止 Pipeline Worker。
- 并发清理只有一个实例获得执行权。

### 9.4 页面与回归

- 日志页桌面端和窄屏浏览器截图复核。
- 控制台无错误和警告，筛选交互正常。
- 完整 pytest 回归，确认采集、运行实例、Worker、清洗、分析、报价和日报不受影响。

## 10. 风险与分阶段落地

- 风险：取消逐目标事件后排障信息不足。运行详情已保存目标状态、时间、计数和错误；先补强对象展示与运行详情测试，再停止事件写入。
- 风险：清理条件错误造成审计或错误日志过早删除。使用事件类型白名单、级别优先规则、边界测试和小批量删除控制。
- 风险：历史积压首次清理造成数据库压力。首次同样限制每轮 20 × 1000 条，不做一次性全表删除。
- 风险：未来多 Pipeline Worker 重复清理。使用数据库互斥锁，并保证删除条件天然幂等。
- 风险：对象名称快照重复或更名。筛选和展示使用事件发生时关联的目标快照，不追溯改写历史名称。

建议分三阶段发布：

1. 先上线对象名称关联、筛选和页面说明，不删除数据。
2. 再停止冗余高频事件写入，观察一个运行周期，确认运行详情信息完整。
3. 最后启用小批量分级清理，先记录 dry-run 计数，再切换实际删除。

回滚时可恢复事件写入并关闭清理配置；已按策略删除的冗余 INFO 日志不恢复，但运行实例和业务结果不受影响。
