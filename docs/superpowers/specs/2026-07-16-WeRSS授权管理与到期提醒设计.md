# WeRSS 授权管理与到期提醒设计

## 1. 背景与现状

WeInsight 的公众号采集已切换为本机 WeRSS + RSS 链路。当前集成只通过固定的只读接口 `GET /api/v1/wx/mps` 同步公众号目录，使用 AK/SK，且运维边界明确禁止目录同步客户端向 WeRSS 发起写请求。

现在需要在 WeInsight 的“公众号”页面增加 WeRSS 授权管理能力：展示授权是否有效及到期时间、在页面内扫码授权；距离到期不足 24 小时时在所有管理页面顶栏显示待办入口，并向可配置收件人发送邮件提醒。

本机固定镜像 `ghcr.io/rachelos/we-mp-rss@sha256:53912f...` 的 OpenAPI 与容器内实现已核对，提供以下接口：

- `GET /api/v1/wx/sys/info`：系统与微信授权信息，支持 JWT 或 AK/SK。
- `GET /api/v1/wx/auth/qr/code`：启动并获取授权二维码，仅支持管理员 JWT。
- `GET /api/v1/wx/auth/qr/status`：查询扫码状态，仅支持管理员 JWT。
- `GET /api/v1/wx/auth/qr/over`：结束扫码，仅支持管理员 JWT。
- `POST /api/v1/wx/auth/token`：用 WeRSS 用户名和密码换取短期 JWT。

上游响应包含公众号 Token、Cookie 等敏感信息；WeInsight 只允许读取并保留白名单字段，绝不向浏览器、日志、数据库或邮件输出 Token、Cookie、二维码原始地址及 WeRSS JWT。

## 2. 可行性结论

整体技术可行，但需要新增一条与现有只读目录同步完全隔离的“授权管理链路”。

| 能力 | 结论 | 约束 |
| --- | --- | --- |
| 授权状态与到期时间 | 可实现 | 从 `sys/info` 白名单解析；不能直接透传上游响应 |
| 页面内扫码授权 | 可实现 | 需要独立 WeRSS 管理账号换取 JWT；仅 WeInsight 已登录管理员可操作 |
| 顶栏不足 24 小时提醒 | 可实现 | 使用数据库中的最近状态快照，避免每次页面访问同步调用 WeRSS |
| 点击提醒跳转公众号页 | 可实现 | 固定跳转 `/sources/articles#authorization-management` |
| 邮件提醒 | 可实现 | 新增 SMTP 配置、定时检查、幂等去重和有限重试 |
| 沿用当前目录 AK/SK 完成扫码 | 不可行 | 二维码接口只接受管理员 JWT，且不能扩大只读目录客户端职责 |

## 3. 目标与非目标

### 3.1 目标

1. 公众号页展示 `有效`、`即将到期`、`已过期`、`未授权`、`状态不可用` 五类状态。
2. 展示公众号名称、到期时间、剩余时长、最近检查时间；不展示 Token。
3. 管理员可在 WeInsight 页面内发起扫码授权并看到成功、二维码过期、失败或取消结果。
4. 授权剩余时间小于等于 24 小时时，在管理后台所有页面顶栏显示待办提醒。
5. 提醒点击后直接定位到公众号页授权卡片。
6. 向配置的一个或多个邮箱发送一次到期提醒；失败时有限重试，避免邮件风暴。
7. 授权成功后刷新状态，并为新的到期时间建立新的提醒周期。

### 3.2 非目标

1. 不在 WeInsight 展示、复制、修改或持久化公众号 Token/Cookie。
2. 不把 WeRSS 的全部管理后台代理进 WeInsight。
3. 不通过 WeInsight 自动续期或绕过微信扫码授权。
4. 不向浏览器回显已保存的 WeRSS 或 SMTP 密码；页面只显示“已配置/未配置”，密码字段仅用于重新设置。
5. 不修改现有 `/api/v1/wx/mps` 目录同步客户端及其只读凭据。

## 4. 影响范围

### 4.1 后端

- 新增 WeRSS 授权状态客户端与二维码管理客户端。
- 新增授权状态服务、扫码会话服务、提醒服务与 SMTP 邮件发送器。
- 新增状态快照与邮件通知记录的存储仓库。
- 在 Pipeline Worker 增加授权状态检查与通知投递调度。
- 在 Web 应用注入授权服务，并增加公众号授权相关路由。

### 4.2 前端

- `公众号` 页面顶部新增“授权管理”卡片。
- `base.html` 顶栏新增到期待办提醒。
- 新增小型原生 JavaScript 模块处理二维码启动、轮询、取消和成功刷新。
- 补充授权状态、二维码弹窗、待办提醒的响应式样式。

### 4.3 配置与部署

- 新增 WeRSS 管理账号、授权检查周期、提醒阈值和 SMTP 配置。
- WeRSS 管理密码、SMTP 密码只能由环境变量提供。
- Pipeline Worker 需要重启以加载调度与邮件配置，Web 服务需要重启以加载页面和授权代理配置。

## 5. 权限与安全模型

### 5.1 双客户端隔离

保留现有 `WeRSSCatalogClient`，它仍只允许调用 `/api/v1/wx/mps`。新增 `WeRSSAuthorizationClient`：

- 状态读取仅允许 `/api/v1/wx/sys/info`。
- 管理操作仅允许 `/api/v1/wx/auth/token`、`/auth/qr/code`、`/auth/qr/status`、`/auth/qr/over`。
- URL 由配置的本机 `base_url` 与固定路径拼接，不接受用户输入 URL，不跟随重定向。
- 上游响应大小、内容类型和字段类型均设上限并严格校验。
- JWT 只保存在进程内存，按过期时间提前刷新，绝不落库或写日志。

### 5.2 WeRSS 管理凭据

新增配置：

```yaml
integrations:
  werss_authorization:
    enabled: true
    base_url: http://127.0.0.1:8001
    username: weinsight_auth_controller
    password_env: WERSS_AUTH_ADMIN_PASSWORD
    check_interval_seconds: 300
    warning_threshold_hours: 24
```

优先在 WeRSS 创建专用强密码用户。当前上游二维码接口只要求“任意有效 JWT”，没有可验证的细粒度扫码权限，因此该账号在 WeRSS 侧仍具有超出单一接口的潜在能力。补偿控制为：仅绑定本机回环地址、WeInsight 客户端固定接口白名单、凭据只在服务端环境变量中、操作只开放给已登录 WeInsight 管理员、完整记录非敏感审计事件。

### 5.3 浏览器侧控制

- 启动和取消扫码使用 POST + CSRF。
- 状态轮询只返回白名单枚举，不返回上游正文。
- 二维码由 WeInsight 后端获取并以 `Cache-Control: no-store` 代理；浏览器不接触 WeRSS JWT、管理密码和上游内部 URL。
- 扫码会话绑定当前 WeInsight 管理员会话，5 分钟超时，同一时刻全系统最多一个活动扫码会话。
- 二维码响应限制为图片类型和合理大小；禁止任意 URL 抓取，防止 SSRF。

## 6. 状态模型与判断规则

新增单例表 `wechat_werss_authorization_state`：

| 字段 | 含义 |
| --- | --- |
| `singleton_id` | 固定为 1 |
| `status` | `valid`、`expiring`、`expired`、`unauthorized`、`unavailable` |
| `account_name` | 授权公众号名称，可为空 |
| `expires_at` | 按 Asia/Shanghai 解释并存储的到期时间 |
| `last_checked_at` | 最近检查时间 |
| `last_successful_check_at` | 最近成功读取时间 |
| `last_error_code` | 脱敏错误码，不保存异常原文 |
| `authorization_version` | 由公众号标识与到期时间的摘要生成，用于通知去重 |
| `updated_at` | 数据更新时间 |

状态判断不能只相信 WeRSS 的 `login` 布尔值。当前上游实现可能使用扫码时保存的 `remaining_seconds`，该值不保证随时间递减。WeInsight 按以下优先级独立判断：

1. 响应不可达、结构不合法或到期时间无法解析：`unavailable`，保留上一次成功值用于页面说明，但不得把不可用误报为已过期。
2. 没有授权信息或明确未登录，且没有有效到期时间：`unauthorized`。
3. `expires_at <= now`：`expired`。
4. `0 < expires_at - now <= 24h`：`expiring`。
5. 其余：`valid`。

所有时间计算使用 `Asia/Shanghai` 的有时区时间；数据库写入前转换为项目统一时间语义。

## 7. 邮件通知模型

新增 `wechat_werss_authorization_notice` 表：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `authorization_version` | 授权周期摘要 |
| `notice_type` | 首期为 `expiring_24h`、`expired` |
| `status` | `pending`、`sending`、`sent`、`retry_wait`、`failed` |
| `attempt_count` | 已尝试次数 |
| `next_attempt_at` | 下次重试时间 |
| `sent_at` | 成功发送时间 |
| `last_error_code` | 脱敏错误码 |
| `recipient_count` | 收件人数，不保存地址正文 |
| `create_time` / `update_time` | 审计时间 |

唯一键为 `(authorization_version, notice_type)`。同一授权到期周期在 24 小时阈值只创建一次通知；扫码成功得到新的到期时间后自然生成新的 `authorization_version`。

SMTP 配置示例：

```yaml
notifications:
  werss_authorization_email:
    enabled: true
    host: smtp.example.com
    port: 587
    username: notifier@example.com
    password_env: WERSS_AUTH_SMTP_PASSWORD
    security: starttls
    from_address: notifier@example.com
    recipients:
      - owner@example.com
```

约束：

- `security` 只能为 `starttls`、`ssl` 或 `plain`，生产配置禁止 `plain`。
- 收件人启动时校验格式并去重，至少一个有效地址才允许启用。
- 主题包含“公众号授权即将到期”或“公众号授权已过期”；正文只包含公众号名称、到期时间、剩余时长和 WeInsight 公众号页链接。
- 失败后按 5、15、60 分钟重试，最多 3 次；超过后记录失败，不在每个调度周期重复新建通知。
- SMTP 发送设置连接和读写超时；日志只记录结果码、通知记录 ID 和收件人数。

## 8. 后端实现方案

### 8.1 组件

建议新增：

- `app/integrations/werss_authorization.py`：严格白名单的上游客户端与响应 DTO。
- `app/integrations/smtp_mailer.py`：SMTP 发送适配器。
- `app/domain/werss_authorization.py`：状态枚举、快照、通知命令。
- `app/services/werss_authorization_service.py`：状态刷新、扫码会话、提醒判断。
- `app/storage/werss_authorization_repo.py`：状态单例与通知幂等仓库。
- `app/web/static/werss-authorization.js`：二维码交互。

### 8.2 Web 路由

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| GET | `/sources/articles` | 同时渲染最近授权状态快照 |
| POST | `/sources/articles/authorization/refresh` | 人工刷新授权状态 |
| POST | `/sources/articles/authorization/qr/start` | 创建受控扫码会话 |
| GET | `/sources/articles/authorization/qr/{session_id}/image` | 返回 no-store 二维码图片 |
| GET | `/sources/articles/authorization/qr/{session_id}/status` | 返回扫码状态枚举 |
| POST | `/sources/articles/authorization/qr/{session_id}/cancel` | 取消并清理扫码会话 |

上游不可用时统一返回可展示的业务错误码，不将连接异常、凭据、响应正文或堆栈返回浏览器。

### 8.3 调度

在 Pipeline Worker 中增加独立的 5 分钟间隔作业：

1. 获取状态并更新单例快照。
2. 根据阈值事务性创建幂等通知记录。
3. 领取一条到期通知并发送邮件。
4. 更新发送结果和下次重试时间。

`max_instances=1`、`coalesce=true`。人工刷新只更新状态，不直接发送邮件，避免浏览器重复点击触发多封邮件；下一次 Pipeline 周期负责投递。

## 9. 前端交互

### 9.1 公众号页授权卡片

卡片锚点为 `authorization-management`，展示：

- 状态徽标。
- 公众号名称。
- 到期时间和剩余时长。
- 最近成功检查时间；状态不可用时显示“暂时无法连接 WeRSS”，同时注明上次成功状态。
- “刷新状态”和“扫码授权”按钮。
- 邮件提醒是否启用及脱敏收件人数。

点击“扫码授权”打开弹窗。弹窗显示二维码、剩余有效时间和状态文案；每 3 秒轮询，成功后关闭弹窗并刷新卡片。二维码过期、网络失败或用户取消时停止轮询并清理会话。

### 9.2 顶栏待办

当状态为 `expiring` 或 `expired` 时，`base.html` 在账号信息左侧显示醒目的待办按钮：

- 即将到期：`公众号授权将在 23小时15分 后到期`。
- 已过期或未授权：`公众号授权已失效，待处理`。
- 点击跳转 `/sources/articles#authorization-management`。

`unavailable` 不显示“到期”待办，避免误报；若最近成功快照已是 `expiring` 或 `expired`，可继续显示该快照提醒，并额外标注“状态待刷新”。

顶栏读取数据库快照，不同步请求 WeRSS。Web 进程内增加最多 30 秒的只读缓存，平衡跨页面及时性和数据库访问。

## 10. 测试与回归方案

### 10.1 单元测试

1. 状态响应白名单解析，确认 Token、Cookie 不进入 DTO。
2. 到期时间边界：24 小时整、24 小时内、已过期、缺失、格式错误和时区转换。
3. 上游 HTTP 超时、401、重定向、超大响应、非 JSON 和字段类型异常。
4. JWT 内存缓存与提前刷新；错误信息脱敏。
5. 二维码 URL/path 白名单、图片类型与大小限制、扫码会话绑定和单会话互斥。
6. 通知唯一键去重、授权版本变化、失败重试和最大次数。
7. SMTP TLS 配置、收件人校验、正文不含敏感字段。

### 10.2 Web 测试

1. 未登录用户无法读取状态、启动扫码或读取二维码。
2. POST 路由校验 CSRF。
3. 五类状态正确渲染；只有应提醒状态显示顶栏待办。
4. 待办链接定位公众号授权卡片。
5. 二维码响应带 `Cache-Control: no-store`，不包含上游地址和 Token。
6. 页面静态资源和响应式布局回归。

### 10.3 集成与部署验证

1. 使用 Fake WeRSS 完成状态、扫码成功、二维码过期、上游不可用全流程。
2. 使用本机固定 WeRSS 镜像做受控契约测试，禁止输出真实 Token。
3. 使用本地 SMTP 捕获服务验证邮件主题、正文、收件人和去重，不向真实邮箱发送测试邮件。
4. 运行现有公众号目录同步、任务、日志、结果和管理后台测试，确认只读同步边界未改变。
5. 部署后人工扫码一次，验证授权卡片更新、顶栏提醒消失、后续 RSS 采集正常。

## 11. 风险与控制

| 风险 | 等级 | 控制措施 |
| --- | --- | --- |
| WeRSS `sys/info` 返回原始 Token | 高 | 严格 DTO 白名单；禁止透传、落库和日志输出 |
| WeRSS JWT 无扫码专用细粒度权限 | 高 | 专用用户、回环网络、固定路径白名单、WeInsight 管理员权限和审计 |
| 上游接口未声明稳定兼容契约 | 中 | 镜像 digest 固定、启动契约检查、升级前回归 |
| 二维码接口为全局状态，多人并发冲突 | 中 | 全系统单活动扫码会话，超时和取消清理 |
| 上游 `remaining_seconds` 可能不实时递减 | 中 | 以解析后的绝对到期时间独立计算 |
| Pipeline 停止导致邮件未发送 | 中 | 顶栏显示最近检查时间；现有 Worker 状态监控继续承担进程告警 |
| SMTP 故障造成邮件风暴 | 中 | 唯一键去重、有限重试、最大次数 |
| 页面请求直接依赖 WeRSS 造成变慢 | 低 | 页面只读数据库快照，人工刷新才同步请求 |

## 12. 分阶段落地建议

### 阶段一：状态与页面提醒

- 建表、状态客户端、定时检查、公众号授权卡片、顶栏提醒。
- 暂不启用邮件和扫码写操作，先验证状态准确性及敏感信息隔离。

### 阶段二：页面内扫码

- 配置专用 WeRSS 管理用户。
- 上线二维码代理、会话互斥、轮询和审计。
- 受控完成一次真实扫码验收。

### 阶段三：邮件提醒

- 配置 SMTP 与收件人。
- 先用邮件捕获服务验证，再启用真实投递。
- 观察一个授权周期的去重和重试行为。

## 13. 验收标准

1. 授权状态、到期时间和剩余时长与 WeRSS 实际授权一致，误差不超过一个检查周期。
2. 剩余时间小于等于 24 小时后，所有后台页面顶栏均出现可点击待办。
3. 点击待办定位到公众号页授权卡片。
4. 同一授权周期的 24 小时提醒最多成功发送一封；失败最多尝试三次。
5. 管理员可在 WeInsight 内扫码完成授权，成功后状态刷新且待办消失。
6. 浏览器响应、数据库、应用日志和邮件均不存在 WeRSS Token、Cookie、JWT 或管理密码。
7. 现有公众号目录同步仍只调用 `/api/v1/wx/mps`，原 AK/SK 和既有采集结果不受影响。

## 14. 前端安全配置增量设计

经确认，WeRSS 管理凭据、SMTP 参数和收件人改为可在公众号页面直接配置，保存后 Web 与 Pipeline Worker 动态读取，不要求重启服务。

### 14.1 数据与加密

新增单例表 `wechat_werss_authorization_settings`，保存 WeRSS 管理用户名、SMTP 非敏感参数、收件人 JSON 和两个加密密码。密码使用 Windows DPAPI `CurrentUser` 作用域加密后以二进制写入数据库；数据库中不保存明文，页面和接口不返回密文。Web 与 Pipeline Worker 必须由同一 Windows 运行账号启动，否则 Pipeline 无法解密 Web 保存的密码并会以脱敏错误码失败。

环境变量继续作为回退：数据库已保存密码时优先使用数据库密文；数据库未配置时读取 `WERSS_AUTH_ADMIN_PASSWORD` 或 `WERSS_AUTH_SMTP_PASSWORD`。页面只展示最终是否已配置，不说明密码来源。

### 14.2 页面字段

- WeRSS 管理用户名、管理密码（留空表示保留原值）。
- SMTP 启用开关、服务器、端口、加密方式、用户名、密码、发件人。
- 多个收件人，按换行或逗号分隔，保存前校验、去重并规范化。
- “保存配置”“测试 WeRSS 连接”“发送测试邮件”三个操作。

密码输入框始终为空，不设置 `value`，浏览器自动填充关闭。页面仅显示“管理密码已配置”和“SMTP 密码已配置”。

### 14.3 接口与审计

- `POST /sources/articles/authorization/settings`：保存配置，JSON 请求 + CSRF，严格拒绝未知字段。
- `POST /sources/articles/authorization/settings/test-werss`：验证当前保存的管理凭据，只返回成功或脱敏错误码。
- `POST /sources/articles/authorization/settings/test-email`：向当前保存的全部收件人发送测试邮件。

每次保存和测试写入结构化审计事件，只记录操作者、动作、结果、收件人数和是否启用，不记录用户名、邮箱正文、密码、JWT、Token 或上游响应。事件类型加入已分类映射，避免日志页面显示“未分类事件”。

### 14.4 风险补充

- DPAPI 密文与 Windows 账号绑定；更换运行账号或迁移主机前必须在新环境重新输入密码。
- 数据库备份只包含不可跨账号直接解密的密文，但仍按敏感备份管理。
- 测试邮件必须显式点击，不由保存动作自动触发。
- 密码留空表示保留，不能通过页面读取旧密码；需要轮换时直接输入新密码覆盖。
