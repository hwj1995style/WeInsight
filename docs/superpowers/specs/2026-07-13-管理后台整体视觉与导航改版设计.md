# WeInsight 管理后台整体视觉与导航改版设计

## 1. 背景与现状

当前管理后台已经覆盖总览、微信群、公众号、任务、运行、日志、Worker、结果、日报和账户设置，但整体仍保留早期 POC 风格：顶部横向导航拥挤、页面内容区域偏窄、视觉层级弱、表格与监控页面在宽屏下利用率不足。

登录后首先进入的根路径页面仅展示账户、运行环境和数据库配置，缺少业务价值；同时默认密码黄色提醒占据全局空间。用户已明确不需要该提示页，也不需要默认密码提醒，并选择“方向 A：专业运维控制台”作为整个后台的统一视觉方向。

## 2. 目标与非目标

### 2.1 目标

- 登录成功后直接进入实时总览。
- 根路径 `/` 对已登录用户重定向到 `/dashboard`。
- 移除全局默认密码提醒，不再主动提示修改默认密码。
- 将整个管理后台统一为深色侧栏、浅色工作区的专业运维控制台。
- 桌面侧栏默认展开并支持折叠，折叠状态保存在浏览器本地。
- 平板和手机使用抽屉式导航。
- 统一页面标题、面板、KPI、表格、筛选器、表单、状态标签、空状态和响应式布局。
- 保持现有业务路径、权限、数据读取、表单提交和 ECharts 图表能力不变。

### 2.2 非目标

- 不引入 React、Vue 或新的前端构建链。
- 不修改后端业务接口、数据库模型、权限模型或 Worker 行为。
- 不增加新的业务模块、指标或虚构数据。
- 不重新设计账户密码规则；账户设置页面继续保留。
- 不使用渐变、玻璃拟态、装饰插画或大面积高饱和效果。

## 3. 影响范围

### 3.1 路由与登录行为

- `POST /login` 成功后重定向到 `/dashboard`。
- `GET /` 在通过鉴权后重定向到 `/dashboard`。
- 删除根路径控制台提示页的现行入口；`home.html` 仅保留账户密码页面所需内容，或拆分为独立密码模板。
- 匿名访问现有受保护页面仍重定向到 `/login`。

### 3.2 页面范围

- 登录页。
- 实时总览。
- 微信群和公众号来源列表、创建与编辑表单。
- 任务列表、创建、详情与停止操作。
- 运行列表与详情。
- 日志列表与实时事件。
- Worker 和微信健康状态。
- 群与公众号结果页。
- 日报列表、详情、请求与导出页面。
- 账户设置页面。

## 4. 信息架构与应用壳

### 4.1 桌面端

- 左侧固定深海军蓝导航栏。
- 展开宽度约 224px，显示品牌、图标和文字；折叠宽度约 72px，仅显示图标并通过 `title` 或可访问标签保留含义。
- 默认展开；折叠状态写入 `localStorage`，键名固定且版本化。
- 主内容区域由轻量顶部工具栏和页面内容组成。
- 顶部工具栏展示当前页面名称、更新时间或页面级辅助信息、当前账号和退出入口。
- 账户设置放入账号菜单或顶部工具区，不再作为全局突出按钮。

### 4.2 移动端和平板

- 侧栏转换为左侧抽屉，默认关闭。
- 顶部显示菜单按钮、页面标题和账号操作。
- 打开抽屉后提供遮罩；点击导航、遮罩或按 Escape 关闭。
- 表格保留横向滚动，不强行转换成信息丢失的卡片列表。

### 4.3 导航顺序

保持现有业务顺序：

1. 总览
2. 微信群
3. 公众号
4. 任务
5. 运行
6. 日志
7. Worker
8. 结果
9. 日报

导航图标采用一致的线性 SVG 风格，图标使用 `currentColor`，不依赖外部 CDN。

## 5. 视觉设计系统

### 5.1 色彩

- 页面背景：冷灰白。
- 内容面板：纯白。
- 侧栏：深海军蓝。
- 主文字：近黑蓝。
- 次级文字：中性灰蓝。
- 主强调色：清晰蓝色。
- 成功：绿色。
- 警告：橙色。
- 失败：红色。
- 停用和未知：中性灰。

颜色必须通过 CSS 变量集中管理。状态色同时配合文字或图标，不仅依赖颜色表达含义。

### 5.2 字体与密度

- 保留系统字体栈：`Inter`、`Segoe UI`、`PingFang SC`、`Microsoft YaHei`。
- 页面标题约 28–32px，面板标题约 16–18px，正文 14px，辅助信息 12–13px。
- 表格与控制项采用明确字体大小和行高，不使用浏览器默认值。
- 监控页保持信息密度，但通过对齐、间距和分组提高可读性。

### 5.3 面板与容器

- 面板使用 1px 细边框、轻阴影和约 10px 圆角。
- 避免面板内再套多层卡片。
- 页面内容宽度从当前 1120px 放宽到适合运维表格的宽屏容器，保留合理左右留白。
- KPI 卡片紧凑排列；图表区域可并排；积压和明细表格使用完整宽度。

### 5.4 控件

- 主按钮、次按钮、危险按钮和文字按钮使用统一高度、圆角和交互状态。
- 输入框、选择器、文本域统一边框、背景、焦点环和禁用状态。
- 状态标签保持紧凑，避免过度胶囊化。
- 所有交互控件提供 hover、focus-visible、active 和 disabled 状态。

## 6. 页面设计

### 6.1 登录页

- 使用方向 A 的海军蓝与冷灰体系。
- 保持品牌说明与登录表单的清晰分区。
- 不展示默认账号密码、不展示修改默认密码提醒。
- 登录错误和修改密码成功提示继续保留。

### 6.2 实时总览

- 顶部显示“实时总览”和最近更新时间。
- 第一行展示 Worker、微信状态、UI 锁、采集任务和今日运行。
- 第二层展示群采集、公众号采集、来源启用数、日报和积压。
- 运行趋势与采集结果图在宽屏并排，窄屏按顺序堆叠。
- 任务积压表使用全宽布局。
- ECharts 配色与新设计令牌保持一致，并保留文本回退表。

### 6.3 数据列表与详情

- 列表页统一标题、操作区、筛选面板、表格、分页和空状态。
- 详情页统一摘要区、定义列表、关联对象和危险操作区。
- 长路径、错误摘要和 JSON 指标继续安全换行。

### 6.4 表单

- 保持现有字段与后端校验。
- 使用统一的字段分组、说明文本、必填状态和错误提示。
- 桌面支持两列的表单仅用于适合并排的短字段；长字段和关键字段保持整行。

## 7. 数据模型、权限与后端影响

- 不修改数据库表、Repository、Service 或业务数据模型。
- 不修改 Session、CSRF、登录限流、密码哈希和权限校验。
- `AuthenticatedAdmin.using_default_password` 可继续存在于后端兼容模型中，但前端不再读取或渲染提醒。
- 路由变更仅涉及登录成功和根路径的重定向目标。

## 8. 前端实现方案

- 继续使用 FastAPI、Jinja2、原生 CSS 和少量原生 JavaScript。
- `base.html` 负责应用壳、侧栏、顶部工具栏和抽屉结构。
- 新增本地内联或独立 SVG 图标片段，禁止外部图标 CDN。
- `app.css` 重构为令牌、基础元素、应用壳、组件、页面和响应式分区。
- 侧栏控制脚本只处理折叠、抽屉、Escape 和本地状态，不接触业务数据。
- 模板只增加必要的语义 class 和结构，不改变现有表单字段名及 URL。

## 9. 测试与回归方案

### 9.1 自动化测试

- 登录成功跳转 `/dashboard`。
- 已登录访问 `/` 跳转 `/dashboard`。
- 匿名访问 `/` 仍跳转 `/login`。
- 页面不包含默认密码提醒文案。
- 应用壳包含侧栏折叠按钮、移动抽屉语义和完整导航。
- 当前导航项继续正确设置 `aria-current="page"`。
- 所有既有 Web、鉴权、来源、任务、运行、日志、Worker、结果和日报测试保持通过。
- 全量 `pytest -q` 通过。

### 9.2 浏览器视觉与交互验证

- 使用真实登录流程检查核心页面。
- 桌面宽屏验证 1440px 与 1920px。
- 平板验证约 768px。
- 手机验证 390px。
- 检查侧栏展开、折叠、刷新后持久化、移动抽屉、Escape、遮罩关闭。
- 检查表格横向滚动、筛选器、表单、分页、状态标签和错误提示。
- 检查 ECharts 尺寸、配色、无脚本文本回退和窗口 resize。
- 对比已确认的方向 A 概念图与最终浏览器截图，记录视觉差异并修正。

## 10. 风险与分阶段落地

### 10.1 风险

- 应用壳结构变更可能影响所有继承 `base.html` 的页面。
- 现有模板 class 较多，CSS 重构可能造成局部回归。
- 宽表格、图表和移动抽屉可能出现溢出或层级问题。
- 当前 Web 进程需要重启才能加载模板和样式变更。

### 10.2 分阶段落地

1. 先实现路由跳转、移除提醒和新应用壳。
2. 建立视觉令牌与共享组件样式。
3. 改造实时总览并完成第一轮视觉校准。
4. 批量统一列表、详情、表单、结果和日报页面。
5. 完成桌面、平板和手机浏览器回归。
6. 运行全量自动化测试并重启开发 Web 服务验收。

## 11. 验收标准

- 登录后不再出现管理控制台提示页。
- 任何页面都不显示默认密码提醒。
- 整个后台使用一致的方向 A 应用壳和视觉系统。
- 桌面侧栏默认展开、可折叠且刷新后保持状态。
- 移动端导航可正常打开和关闭。
- 现有功能、权限、表单和图表无回归。
- 自动化测试和浏览器视觉验收全部通过。
