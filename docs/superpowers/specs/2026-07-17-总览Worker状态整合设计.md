# 总览 Worker 状态整合设计

## 1. 背景与现状

当前侧栏提供独立的“Worker”页面，页面读取全部心跳记录，因此进程每次重启都会新增一行，历史过期实例会持续堆积。日常运维真正需要的是 `pipeline` 和 `collector` 两类 Worker 各自最新实例的状态，而不是在主导航中浏览完整心跳历史。

总览页已经展示 Worker 数量、微信客户端状态和 UI 锁状态，但缺少最新 Worker 的类型、最近心跳、启动时间等直接信息。独立 Worker 页面同时展示微信健康和 UI 锁详情，其中总览已有相应摘要，信息入口重复。

对“安全错误摘要全部为空”的实际数据和写入链路排查结果如下：

- 当前心跳表共 36 条记录：21 条 `collector`、15 条 `pipeline`，`last_error_summary` 均为空，且没有 `degraded` 状态记录。
- `pipeline` Worker 的心跳写入代码始终将 `last_error_summary` 设置为 `None`；阶段失败写入结构化日志，不写入心跳摘要。
- `collector` 仅在领取失败、租约丢失、心跳写入失败等 Worker 级降级时设置安全摘要；恢复为 `running` 后会在下一次心跳清空摘要。
- 当前最新 `pipeline` 和 `collector` 均为 `running`，所以页面显示“—”符合现有“当前故障摘要”语义，不是读取或渲染故障。

## 2. 目标与非目标

### 2.1 目标

1. 在总览首排系统运行卡中展示最新的 `pipeline` 和 `collector` 各一条，不再单独占用一个 Worker 区域。
2. 每条展示 Worker 类型、在线/过期及运行状态、最近心跳、启动时间。
3. 安全错误摘要不直接铺开；仅在存在摘要时提供“查看错误”按钮，通过弹窗展示脱敏后的完整摘要。
4. 从侧栏移除独立 Worker 导航；旧 `/workers` 地址重定向到 `/dashboard#worker-status`，兼容旧书签。
5. 总览继续展示最新微信状态和 UI 锁摘要，不再保留重复的独立历史页面入口。
6. 总览首部总共只保留两排通栏卡片：第一排展示微信、UI 锁、Pipeline 和 Collector；第二排展示任务、采集、配置与日报。Worker 不再形成第三排宽松区域。
7. 微信健康值 `ok` 在页面上显示为“正常”；UI 锁仅展示自身状态，不再带 Worker 跳转链接。
8. 保留两张 24 小时图表，移除图表下方重复的“逐小时运行终态守恒表”和“采集结果文本明细”。

### 2.2 非目标

1. 不删除心跳历史数据，历史仍用于在线判断、审计和故障排查。
2. 不把业务目标失败或 Pipeline 阶段失败复制到心跳摘要；这些历史故障继续由结构化日志承载。
3. 不改变 Worker 心跳间隔、TTL、进程启动或故障恢复逻辑。
4. 不在总览展示完整 Worker ID、历史进程列表或原始异常堆栈。

## 3. 影响范围

- 后端查询：总览运行快照增加两类最新 Worker 记录。
- 服务模型：`RuntimeDashboardSnapshot` 增加 `workers` 字段，并继续执行安全输出处理。
- 前端页面：总览增加两张 Worker 状态卡片和错误摘要弹窗；侧栏移除 Worker 导航。
- 路由兼容：`/workers` 改为 303 重定向至总览锚点。
- 测试：覆盖最新实例选择、缺失类型、过期状态、摘要弹窗、安全输出及旧地址重定向。

## 4. 数据与权限模型

不新增数据表，不改变管理员权限。最新实例按 `worker_type` 分组，以 `last_heartbeat_at DESC, worker_id DESC` 选取第一条；在线状态仍按当前心跳 TTL 判断。

总览最多返回两条 Worker 数据。`last_error_summary` 继续经过现有安全输出函数处理，弹窗只渲染服务层已脱敏内容，不读取结构化日志技术指标或异常堆栈。

当某一类型没有任何心跳记录时，页面固定保留该类型并显示“未注册”，避免用户误以为系统只需要一种 Worker。

## 5. 后端实现方案

在运行监控仓库增加“每种类型最新一条”的查询，使用窗口函数：

1. 对 `worker_type` 分区。
2. 按最近心跳和 Worker ID 倒序编号。
3. 只返回 `row_rank = 1` 的 `pipeline`、`collector`。
4. 使用 TTL 截止时间计算 `is_live`。

`RuntimeDashboardSnapshot` 携带安全化后的最新 Worker 元组。总览服务失败时，空快照返回空元组，页面按两种固定类型渲染“未注册”。独立 Worker 查询可以暂时保留在服务内部，避免一次改动扩大到无关调用；Web 路由不再渲染历史页面。

## 6. 前端实现方案

总览第一排通栏卡保留 `worker-status` 锚点，用四个紧凑单元并排展示微信、UI 锁、Pipeline 和 Collector。Pipeline 与 Collector 不再使用内嵌大卡片，而是在同一行内展示在线标识、运行状态、最近心跳、启动时间和错误入口。390px 移动端再按单元纵向排列。

每个 Worker 紧凑单元固定展示：

- 顶部：Worker 类型名称，以及在线/过期状态标识。
- 主体：运行状态、最近心跳、启动时间三项信息。
- 底部：无摘要时显示“无当前错误”；有摘要时显示“查看错误”按钮。

卡片使用现有白色表面、浅色边框、圆角和语义状态色，不新增渐变、装饰图或与当前总览不一致的视觉语言。过期卡片通过状态色和边框强调，但不改变整体布局。

错误摘要存在时使用原生 `<dialog>` 弹窗展示，支持关闭按钮和 Escape 关闭。摘要为空时明确显示“无当前错误”，避免“—”造成字段未加载的误解。

总览顶部改为两张通栏复合卡片，卡片内部用紧凑指标单元承载原有数值：

- 第一排系统运行：微信状态、UI 锁、Pipeline、Collector。原 Worker 数量摘要由两个具体 Worker 状态取代，避免重复。
- 第二排业务概览：采集任务、今日运行、任务积压、微信群与公众号采集、微信群与公众号配置、生成日报。

微信客户端状态和 UI 锁继续保留在系统状态卡片中。`ok` 是内部枚举值，页面统一转换为“正常”；UI 锁的说明只解释锁状态，不再显示“查看 Worker 状态”。

图表作为总览的主要历史视图保留，不再同时渲染相同数据的可见文本表格。图表本身继续提供 `role="img"` 和动态 `aria-label`；脚本加载失败时提示用户稍后刷新，不再引导至已移除的文本表。

## 7. 测试与回归方案

1. 仓库查询对每种 Worker 只返回最近一条，并正确计算在线/过期。
2. 服务层对 Worker ID、主机、版本和摘要执行现有安全输出处理。
3. 总览固定展示 Pipeline、Collector 两张卡片；某类型缺失时显示“未注册”。
4. 摘要为空时显示“无当前错误”，存在时只通过弹窗展示安全文本。
5. `/workers` 返回 303 并跳转到 `/dashboard#worker-status`。
6. 侧栏不存在 Worker 导航，总览中的微信状态和 UI 锁仍正常展示。
7. 运行全量测试，并用 Playwright 验证桌面端、390px 移动端、弹窗打开关闭和锚点跳转。
8. 总览顶部总共只有两排通栏复合卡片，Worker 状态已紧凑合入第一排，不存在第三排独立 Worker 容器。
9. UI 锁不含 Worker 跳转，页面不再包含“逐小时运行终态守恒表”和“采集结果文本明细”。

## 8. 风险与分阶段落地建议

| 风险 | 控制措施 |
| --- | --- |
| 最新记录已过期导致误判系统正常 | 最新记录仍展示，但以 TTL 明确标记“过期” |
| 摘要为空被误解为数据缺失 | 使用“无当前错误”替代破折号，并在文案中说明是当前故障摘要 |
| 将历史业务失败误当 Worker 健康状态 | 心跳摘要继续只代表 Worker 级当前降级，历史失败保留在日志页 |
| 删除导航导致旧链接失效 | `/workers` 保留重定向并指向总览锚点 |
| 总览信息继续膨胀 | 固定两张紧凑卡片，不展示历史实例、完整 ID、PID 或技术堆栈 |

建议一次完成查询、总览展示、重定向和导航收口；不需要数据库迁移。若未来需要 Worker 历史审计，应从日志或专门审计入口按需查询，不恢复主导航中的常驻历史列表。

## 9. 验收标准

1. 总览第一排紧凑展示最新 Pipeline 和 Collector 状态，且最近心跳、启动时间正确；页面不存在独立 Worker 大卡区域。
2. 当前健康实例显示“无当前错误”；降级摘要可通过弹窗查看且已脱敏。
3. 独立 Worker 导航消失，旧地址能正确跳回总览。
4. 微信状态和 UI 锁摘要不丢失。
5. 全量自动化测试及真实浏览器验收通过。
