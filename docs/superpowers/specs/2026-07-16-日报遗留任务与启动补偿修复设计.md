# 日报遗留任务与启动补偿修复设计

## 背景与现状

公众号文章分析仍向 `wechat_article_process_task` 写入 `article_daily_report` 日期任务。现行 Pipeline 已切换为消费 `wechat_report_generation_request`，不再领取前述旧任务。历史文章重分析时，`ON DUPLICATE KEY UPDATE` 又会把已完成日期重置为 `pending`，导致首页展示不可消费的待处理任务。

2026-07-16 排查时共有 34 条旧任务：28 条日期已经存在日报，5 条历史日期尚无日报，1 条为当天。电脑在 2026-07-16 00:10 处于关机状态，Pipeline 重启后只注册下一次定时补偿，没有立即补建 2026-07-15 的补偿请求。

## 目标与非目标

### 目标

1. 停止创建或重新打开已退役的 `article_daily_report` 任务。
2. Pipeline 每次启动时幂等补建前一日的 compensation 请求，避免关机错过 00:10 后永久缺报。
3. 首页和运行指标不再把旧任务显示为真实积压。
4. 清理现有旧任务状态，并通过现行请求机制补生成缺失的历史日报。
5. 保留未知事件回退展示，同时为数据库已出现的事件补齐中文摘要。

### 非目标

1. 不改变群日报的现有任务机制。
2. 不删除历史任务、日报或文章分析数据。
3. 不改变日报内容模型、最终版口径或手工生成入口。
4. 不自动修正 `2022-11-01` 等来源日期；该日期按已有结构化数据生成历史日报并保留审计痕迹。

## 影响范围

- 文章分析：不再调用旧日报任务创建接口。
- Pipeline 启动：启动心跳后立即幂等创建昨日补偿请求；随后正常 tick 消费。
- 首页与运行指标：过滤已退役的 `article_daily_report` 任务，并展示真实报告请求积压。
- 数据修复：旧任务统一标记为 `success`；缺失历史日报通过 `wechat_report_generation_request` 创建 compensation 请求。
- 日志展示：补齐当前已知事件摘要与 misfire 指标。

## 数据模型与状态规则

`wechat_report_generation_request` 是日报生成的唯一待处理队列。允许进入积压统计的状态为 `pending`、`running`、`failed`，任务类型统一展示为 `report_generation`，链路展示为 `report`。

`wechat_article_process_task.task_type='article_daily_report'` 被视为退役兼容数据：

- 代码不再新增或重置；
- 数据修复只将非终态改为 `success`，不删除记录；
- 首页和文章运行指标显式排除，避免未执行迁移时继续误报。

## 后端实现方案

1. 从 `ArticleAnalysisService` 协议和成功路径移除 `create_daily_report_task` 调用。
2. `pipeline_main` 在长期运行和 `--once` 模式中，先执行 `ensure_daily_compensation_now()`，再执行或调度 tick。创建操作使用既有幂等键 `compensation:all:<date>`。
3. Dashboard backlog SQL：
   - 文章处理任务排除 `article_daily_report`；
   - 合并 `wechat_report_generation_request` 的真实非终态统计。
4. Article runtime metrics SQL 显式排除旧任务，防止 CLI 继续误报。
5. 新增幂等迁移脚本，将旧日报任务非终态更新为 `success`。
6. 本机修复时，为所有“存在旧 pending 任务且历史日报缺失、日期早于今天”的日期创建 compensation 请求；当天不提前生成最终日报。

## 前端影响

不改变页面结构。首页“待处理任务”将只展示可被当前 Worker 消费的队列；真实日报请求显示为“日报 / report_generation”。旧的“公众号 / article_daily_report / pending 34”将消失。

## 测试与回归方案

1. 文章分析成功后不再创建旧日报任务。
2. Pipeline 长期运行与 `--once` 启动均先补偿昨日，再执行 tick 或注册调度。
3. Dashboard SQL 排除旧任务并纳入真实报告请求。
4. Article runtime metrics 排除旧任务。
5. 迁移脚本幂等且不删除历史记录。
6. 真实开发库验证：旧 pending 数为 0；缺失历史日期均有请求且执行成功；首页真实积压与请求表一致。
7. 运行完整测试套件。

## 风险与分阶段落地建议

- 启动补偿可能与 00:10 定时任务重复创建；既有唯一幂等键保证只生成一个请求。
- 历史缺失日报可能包含异常来源日期；仍按现有分析数据生成，不静默改写日期。
- 先发布代码与迁移，再执行本机数据修复；修复前后的计数均记录，失败时保留原任务行便于审计。
