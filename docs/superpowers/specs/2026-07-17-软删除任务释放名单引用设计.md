# WeInsight 软删除任务释放名单引用设计

## 1. 背景与现状

采集任务删除采用软删除：`wechat_collection_job.status` 更新为 `deleted`，任务、目标快照、运行实例和采集结果继续保留。

当前 `wechat_collection_job_target.group_config_id` 与 `article_config_id` 对名单配置使用 `ON DELETE RESTRICT`，且引用查询在删除名单时包含 `deleted` 任务。因此，即使管理员已经删除任务，目标行仍永久占用名单外键，名单删除仍返回“该名单正被采集任务引用”。这使“先删除任务，再删除名单”的操作无法闭环。

## 2. 目标与非目标

### 2.1 目标

- 任务软删除后，不再阻止其引用名单的安全删除。
- 删除名单时，在同一事务中将已删除任务目标上的配置外键主动置空。
- 保留任务名称、目标名称、优先级、配置快照、运行历史、事件和采集结果。
- `scheduled`、`active`、`stop_requested`、`stopped`、`completed` 任务继续阻止名单删除；管理员必须先停止并删除任务。
- 错误页面明确提示需要先停止并删除引用任务。

### 2.2 非目标

- 不物理删除任务、目标快照、运行实例、事件或采集结果。
- 不允许直接删除正在运行、停止中、已停止或已完成但尚未删除的任务引用名单。
- 不改变任务状态机、停止安全检查点或 Worker 调度逻辑。
- 不自动级联删除任何业务历史数据。

## 3. 影响范围

- 初始化表结构：`sql/init.sql`。
- 现有数据库迁移：新增 `sql/migrations/20260717_001_release_deleted_job_source_references.sql`。
- 名单引用查询：`app/storage/source_reference_repo.py`。
- 名单删除事务：`app/storage/source_mutation_repo.py`。
- 删除冲突提示：`app/web/routes/sources.py`。
- 数据库、Repository、Service 和 Web 回归测试。

## 4. 数据模型与权限模型

### 4.1 任务目标约束

`wechat_collection_job_target` 继续保留两个可空外键：

- `group_config_id`
- `article_config_id`

约束由“必须且只能有一个外键非空”调整为“最多一个外键非空”：

- 新建任务目标时仍由应用保证恰好一个外键非空。
- 名单删除后，历史目标允许两个外键都为空。
- 两个外键同时非空仍由 CHECK 约束拒绝。

两个配置外键继续使用 `ON DELETE RESTRICT` 作为最终保护。删除事务只会主动置空 `deleted` 任务目标上的对应外键；目标名称和完整配置已经保存在 `target_name_snapshot` 与 `config_snapshot_json`，因此解绑不会破坏历史展示和审计。保留 `RESTRICT` 还能阻止遗漏引用或并发写入造成的误删。

### 4.2 权限

- 仍沿用现有管理员 Session、CSRF 和操作权限。
- 不增加新的删除接口或绕过既有鉴权。

## 5. 后端实现方案

### 5.1 引用查询

`active_only=True` 继续只返回 `scheduled`、`active`、`stop_requested`，用于停用保护。

`active_only=False` 改为返回所有 `status <> 'deleted'` 的任务，用于删除和改名保护。这样 `stopped` 与 `completed` 仍会阻止删除，只有完成任务软删除后才释放业务引用。

### 5.2 数据库迁移

迁移按以下顺序执行：

1. 保留两个现有 `ON DELETE RESTRICT` 外键。
2. 删除 `ck_job_target_exactly_one`。
3. 新增“最多一个配置外键非空”的 CHECK 约束。

迁移不更新现有数据、不删除目标行，也不放宽数据库外键保护；执行前仍应按部署流程备份数据库。

### 5.3 删除流程

1. Service 检查名单存在且已停用。
2. Repository 在事务中锁定名单并查询所有非 deleted 任务引用。
3. 存在非 deleted 引用时返回安全冲突及任务名称。
4. 仅剩 deleted 引用时，在同一事务中将这些历史目标的对应配置外键置空。
5. 删除名单配置行；`RESTRICT` 外键继续兜底阻止任何未解绑引用，目标快照与所有历史关系继续保留。

### 5.4 前端提示

删除被阻止时提示：“该名单仍被采集任务引用，请先停止并删除以下任务后重试。”并继续展示安全转义后的任务名称。

## 6. 前端影响

- 页面结构和操作入口不变。
- 只调整删除冲突文案，使操作步骤明确。
- 任务删除成功后，管理员返回名单页即可继续删除已停用名单。

## 8. 风险与分阶段落地建议

### 8.1 风险

- 未执行迁移时，解绑操作会被旧的“必须存在一个外键”CHECK 拒绝，整个删除事务回滚。
- CHECK 约束调整错误可能允许两个配置外键同时非空，造成链路类型歧义。
- 如果历史展示误用实时配置表而非快照，解绑后可能显示缺失；现有详情与运行链路应继续基于目标快照验证。

### 8.2 落地顺序

1. 先发布并执行数据库迁移。
2. 再发布引用查询与错误提示代码。
3. 验证一个已删除任务引用的停用名单可以删除。
4. 验证活动、停止中、已停止和已完成任务仍会阻止删除。
5. 完成全量回归后再进入生产环境。

## 9. 验收标准

- 管理员停止并删除任务后，可以删除对应的已停用名单。
- 任务快照、运行历史、事件和采集结果仍可查询。
- 非 deleted 任务引用不会被绕过。
- 删除名单不会级联删除任务目标或业务历史。
- 全量自动化测试和 SQL 迁移检查通过。
