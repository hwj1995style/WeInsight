# 正文动态推码规则设计

## 背景与现状

报价矩阵会把公众号正文中的已知码数转换为 30–50 码，并对缺失码数进行补全。当前家美鲜定位逻辑能够识别“32 以下顺减 -6”，但实现绑定在公众号名称上；江西表格中的“精品大码以上每斤 +2”“小码以下每斤 -4 元”尚未参与补全，导致矩阵错误套用通用最小临近差值。

两类步长都属于当篇文章的报价规则，数值可能随文章变化，不能固化为公众号常量。

## 目标与非目标

### 目标

1. 从当篇文章已定位的 DOM 表格中动态识别边界推码文字。
2. 支持“某码以下顺减 N”“大码以上每斤 +N”“小码以下每斤 -N”等当前样式。
3. 使用表格实际最高或最低已知码数及其今日价作为锚点，生成矩阵范围内的缺失码数。
4. 正文规则优先于通用最小临近差值；正文没有有效规则时继续使用通用规则。
5. 家美鲜、江西褐壳蛋和粉壳蛋均按各自表格内出现的规则独立计算。

### 非目标

1. 不按公众号写死 +2、-4 或 -6。
2. 不保存完整正文、HTML 或 OCR 原文。
3. 不改变已观测报价，也不对正文未声明的内部缺口强行应用边界规则。
4. 不改变贵阳高低价、成都重叠区间取高价等既有转换规则。

## 影响范围

- 报价内容定位：把表格内规则文字转换成补充报价行。
- 报价抽取：继续消费标准表格行，无需识别公众号身份。
- 报价矩阵：优先使用抽取阶段产生的当篇规则结果，剩余缺价再走通用算法。
- 历史重算：重新分析历史文章后可按当时正文规则生成结果；只读矩阵不会凭空取得未落库的旧正文规则。

## 数据模型与权限模型

本次不新增数据库字段。规则只在正文处理期间存在于内存中，并被展开成与当前表格结构一致的码数和今日价行；最终仍只保存结构化报价明细。数据库权限、后台查询权限和正文不落库边界保持不变。

## 后端实现方案

1. 在报价内容定位层增加通用表内边界规则解析器，不读取公众号名称决定步长。
2. 解析表格的净重/毛重与今日价列，先收集实际报价行，确定最低、最高锚点。
3. 对明确阈值的“X 以下顺减 N”，以 X 码实际今日价为锚点，向 30 码逐码减 N。
4. 对类别边界的“大码以上每斤 +N”，以最高实际码数为锚点，向 50 码逐码加 N；对“小码以下每斤 -N”，以最低实际码数为锚点，向 30 码逐码减 N。
5. 只补不存在的码数，不覆盖表格原始报价；解析失败、缺锚点、步长为零或方向不明确时不生成数据。
6. 每张表独立解析，因此江西褐壳蛋、粉壳蛋可以使用各自正文中的规则。
7. 既有矩阵通用推算继续作为最后兜底，用于正文规则没有覆盖的缺失码数。

## 前端影响

页面结构和列展示不变，只更新计算结果。由正文规则展开的价格沿用当前单元格展示方式，不增加“推算”文字。

### 报价矩阵行列定位交互

为降低宽表横向、纵向对照时看错行列的风险，报价单元格增加十字高亮：

1. 鼠标停在报价单元格时，突出该码数整行、该报价整列以及当前交叉单元格。
2. 纵向高亮同时覆盖品类表头和所属公众号分组表头，便于确认当前报价来源。
3. 鼠标移出矩阵后恢复；单元格获得键盘焦点时保持相同行为。
4. 报价单元格采用单一 Tab 入口和方向键移动，避免宽表产生大量 Tab 停靠点。
5. 高亮使用半透明叠色和边框，不覆盖原始报价、推算值底色及悬停说明。
6. 空值单元格同样参与行列定位，保证整张矩阵的导航一致。

## 测试与回归方案

1. 家美鲜样例：45 码 237 至 32 码 193，识别“32 以下顺减 -6”，得到 31 码 187、30 码 181。
2. 江西样例：50 码 222 至 38 码 186，识别高端 +2 和低端 -4，得到 37 码 182、30 码 154；50 码以内不覆盖原始值。
3. 动态步长测试：把规则数值替换为其他值，结果随正文变化，证明未按公众号写死。
4. 无规则、无锚点和格式不合法时不扩展，矩阵仍按通用规则工作。
5. 江西褐壳蛋、粉壳蛋分别处理，不串用另一张表的规则。
6. 运行报价定位、抽取、矩阵、仓储和结果页相关测试，再运行全量测试。
7. 结果页验证鼠标悬停、移出、点击聚焦以及方向键移动后的行列高亮状态。
8. 桌面和窄屏横向滚动场景均检查高亮连续性、固定码数列和粘性表头。

## 风险与分阶段落地建议

- 规则文案可能出现新的同义表达。首版仅接受方向、边界和数值都明确的保守模式，未识别时回退通用规则，避免误推。
- DOM 合并单元格可能把规则放入标题或单行。解析器同时检查表格标题和单列规则行，但不扫描全文其他区域。
- 历史库中已经生成的明细不含规则语义。代码上线后需重新分析目标历史文章，才能让历史结果使用对应正文规则。
