# 公众号状态页视觉层级与补处理交互设计

## 1. 背景与现状

公众号状态页同时承载公众号状态列表、WeRSS 授权管理和历史文章补处理。当前授权管理与补处理表单都以完整内容面板展示在列表之前，首屏大部分空间被辅助操作占用，公众号列表需要向下滚动后才能查看。历史补处理还把“处理范围”和“单个公众号”拆成两个并列选择框，操作关系不够直接。

## 2. 目标与非目标

### 2.1 目标

1. 将公众号列表提升为页面的主要内容，进入页面后首屏即可看到列表表头和多行状态。
2. 授权管理压缩为轻量状态栏，保留状态、到期时间、最近检查及操作入口，不再展示公众号名称。
3. 历史补处理改为按需打开的单一弹窗，避免常驻表单挤占页面高度。
4. 用一个“处理对象”多选框同时表达“全部已开启公众号”和“一个或多个指定公众号”，消除范围与公众号之间的多控件依赖。
5. 保持现有后端请求字段和权限边界不变，降低回归风险。

### 2.2 非目标

1. 不修改 WeRSS 授权状态判断、扫码授权、邮件提醒或敏感信息处理逻辑。
2. 不修改历史补处理的日期上限、处理模式、排队逻辑和后端校验。
3. 不重做后台全局导航或引入新的前端框架。

## 3. 影响范围

- 模板：调整公众号页内容顺序、授权状态栏、补处理弹窗和表单控件。
- 样式：新增列表主视图、紧凑授权栏、操作工具栏和响应式弹窗样式。
- 交互：新增补处理弹窗开关及单一处理对象与隐藏后端字段的同步。
- 测试：更新页面结构、可访问性和交互契约断言；保留全部后端补处理测试。

## 4. 数据模型与权限模型

不新增或修改数据表。授权状态仍来自既有只读快照；历史补处理仍提交范围、来源、日期、模式和 CSRF 字段。页面中的单一“处理对象”控件负责映射：

| 可见选项 | `scope` | `source_id` |
| --- | --- | --- |
| 全部已开启公众号 | `enabled` | 空 |
| 指定一个公众号 | `single` | 对应来源 ID |
| 指定多个公众号 | `selected` | 去重后的来源 ID 集合 |

所有配置、扫码和补处理操作继续要求已登录管理员权限及既有 CSRF 校验。

## 5. 后端实现方案

后端路径不变，表单继续提交到 `/sources/articles/downstream-processing/backfill`。模板保留隐藏字段 `scope`、`source_id`，并新增 `source_ids`：选择一个公众号时兼容既有 `single + source_id` 契约；选择多个公众号时使用 `selected + source_ids`。后端对 ID 集合执行去重、数量上限和正整数校验，仓库使用绑定参数一次查询所选来源，避免字符串拼接 SQL 或前端循环提交造成部分成功。

服务端仍是最终校验边界，前端映射不能替代后端对范围、来源、日期和模式的验证。

## 6. 前端实现方案

页面顺序调整为：

1. 紧凑页面标题与刷新按钮。
2. 公众号列表主面板；面板标题区同时提供“补处理历史文章”主操作。
3. 紧凑授权管理状态栏；不展示授权公众号名称。
4. 授权配置弹窗、扫码弹窗和历史补处理弹窗。

公众号列表继续采用表格，不改成卡片，保持高密度横向比较。授权管理由四块摘要卡压缩为三项行内状态。历史补处理弹窗在桌面端使用中等宽度，在移动端占满可用宽度；处理对象、起止日期、处理模式放在同一个窗口内。处理对象使用带复选框的下拉面板，按钮摘要展示当前选择。“全部已开启公众号”与具体公众号互斥，具体公众号之间可以多选。

补处理任务提交后的结果摘要仍在主页面可见，放在列表上方，便于确认刚提交任务的结果。

## 7. 测试与回归方案

### 7.1 模板与路由测试

1. 断言公众号列表在授权管理之前渲染。
2. 断言授权摘要不再包含公众号名称字段。
3. 断言补处理表单位于可访问的 `dialog` 中，并有明确的打开、关闭按钮。
4. 断言仅有一个可见“处理对象”多选入口，单选映射 `single + source_id`，多选映射 `selected + source_ids`，“全部已开启”映射 `enabled`。
5. 断言重复 ID、非法 ID、空选择和超过数量上限会在调用仓库前拒绝。
6. 断言强制分析确认仍按模式动态启用并设为必填。
7. 运行现有授权、补处理、公众号状态和 CSRF 测试。

### 7.2 浏览器回归

验证路径：`/sources/articles` → 首屏看到公众号列表 → 打开“补处理历史文章”弹窗 → 选择单个公众号并确认隐藏字段映射 → 关闭弹窗。

桌面端与移动端检查页面身份、非空内容、无错误覆盖层、控制台无相关错误、弹窗焦点与关闭行为、表格横向滚动、文本裁切和布局溢出。

## 8. 风险与分阶段落地建议

| 风险 | 控制措施 |
| --- | --- |
| JavaScript 未执行导致隐藏字段未同步 | 可见选择框设为必填；初始化时立即同步；服务端继续严格校验 |
| 弹窗在小屏幕溢出 | 限制最大高度并让内容区独立滚动，移动端使用全宽布局 |
| 列表前置后授权提醒不易发现 | 顶栏到期待办保持不变，列表下方保留紧凑状态栏和全部操作入口 |
| 任务提交结果被列表淹没 | 成功摘要放在列表标题区之后、表格之前并使用状态提示语义 |

本次一次性完成结构、样式和交互调整；不需要数据迁移，可随模板静态资源发布，异常时可直接回退模板和 CSS。

## 9. 验收标准

1. 常见桌面视口首屏中，公众号列表占据页面主要区域并能看到表头及多行数据。
2. 授权管理不展示公众号字段，状态、到期时间和最近检查仍清晰可见。
3. 历史补处理默认不占用主页面高度，点击按钮后在一个弹窗内完成全部配置。
4. “处理对象”只有一个可见多选入口，能选择全部已开启公众号或同时选择多个可补处理公众号。
5. 原有授权、扫码、邮件配置、下游开关和补处理请求均无功能回归。
