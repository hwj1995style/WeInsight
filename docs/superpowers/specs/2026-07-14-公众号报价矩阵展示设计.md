# 公众号报价矩阵展示设计

## 背景与现状

WeInsight 当前“报价结果”页面以逐条明细形式展示公众号、日期、地区、规格、原始报价、标准价格和换算信息，适合排障，但不便于横向比较同一报价日、同一码数下各公众号的价格。

业务期望参考现有报价表，将结果组织为“码数 × 公众号”的矩阵：行是 50 至 30 码，列是公众号报价，底部展示各公众号的取数规则。由于不同公众号的单位、区间表达、低高价和缺码处理方式不同，本次改动不仅是前端样式调整，还需要增加稳定、可测试的后端聚合规则。

## 目标与非目标

### 目标

1. 将 `/results/prices` 默认视图改为报价矩阵。
2. 默认展示系统中最新报价日期，并支持切换历史报价日期。
3. 同一公众号同一报价日存在多篇文章时，只使用发布时间最新的一篇。
4. 固定输出 50 至 30 码，按降序排列。
5. 同时存在低价和高价的公众号拆成低价、高价两列。
6. 河南金咕咕按元/斤展示，其余公众号按元/箱展示，不用统一口径价格覆盖原始报价。
7. 按公众号预先配置固定取数规则，并在矩阵底部展示规则说明。
8. 对缺失码数使用距离最近的一组已知价差进行外推，并明确标识推算值。
9. 只纳入当前 WeRSS 采集列表中的九个目标公众号。

### 非目标

1. 不新增公众号规则的前端人工配置入口。
2. 不修改 WeRSS 的公众号配置、采集任务或十分钟调度。
3. 不修改现有文章清洗、增量分析和报价抽取流程。
4. 不新增报价汇总结果表或独立跑批。
5. 不删除底层原始报价明细查询能力。
6. 不跨公众号、跨报价日期或跨低高价列补值。

## 影响范围

1. 领域层：新增报价矩阵、公众号列、码数行、价格单元格和规则说明模型。
2. 查询层：补充矩阵聚合所需的文章标识、发布时间、重量上下限和原始价格上下限等安全字段。
3. 服务层：新增按日期选文、按公众号规则转换和矩阵生成逻辑。
4. Web 路由：`/results/prices` 使用矩阵查询；保留日期筛选。
5. 前端模板与样式：将逐条明细表替换为宽矩阵，并支持固定表头、固定首列和横向滚动。
6. 测试：增加规则单元测试、查询测试、路由模板测试和真实页面视觉回归。

## 数据模型与规则模型

本次不修改数据库表。新增只读领域模型，建议结构如下：

- `PriceMatrix`：报价日期、数据更新时间、来源数量、列定义、码数行和规则说明。
- `PriceMatrixColumn`：公众号标识、展示名、单位、价格类型、排序和列键。
- `PriceMatrixRow`：码数以及按列键索引的价格单元格。
- `PriceMatrixCell`：价格、数据来源类型、是否推算和用于解释的原始码数或价差。
- `AccountMatrixRule`：公众号匹配名、展示名、单位、列拆分方式、市场过滤、码数匹配、边界处理、缺码处理和说明文字。

价格单元格来源至少区分：

1. `observed`：公众号原始报价直接匹配或由原始重量区间展开。
2. `extrapolated`：通过最近价差外推得到。
3. `empty`：无法可靠生成价格。

前端不得把 `extrapolated` 显示成与原始值完全相同的视觉状态。

## 数据选择口径

1. 用户未指定 `quote_date` 时，查询纳入范围内数据的最大报价日期。
2. 指定日期时，只查询该报价日期。
3. 同一公众号同一报价日按 `publish_time` 降序选择最新文章；发布时间相同时使用稳定的文章标识或记录 ID 决胜。
4. 只纳入鸡蛋主报价口径和九个目标公众号。
5. 同一文章内先依据公众号规则筛选市场、产品和有效价格，再映射到码数。
6. 原始值优先级始终高于推算值。
7. 无法唯一判定的冲突不得静默覆盖，应按公众号规则决胜；仍无法决胜则留空并保留可诊断原因。

## 公众号固定规则

规则集中保存在后端只读注册表中，不恢复人工配置入口。注册表包含列顺序、单位、单列或双列、市场过滤、区间处理、冲突处理、外推开关及页面说明。

首版规则如下：

1. 家美鲜鸡蛋 佳美鲜：元/箱；精确码数或重量区间展开；价格存在上下限时拆低价、高价列。
2. 河北馆陶鸡蛋报价：元/箱；净重区间内各码使用同一报价；价格存在上下限时拆低价、高价列。
3. 河南金咕咕蛋品：元/斤；规格区间展开；区间边界重复时取较高价格；价格存在上下限时拆低价、高价列。
4. 贵阳鸡蛋价格：元/箱；明确拆分低价、高价列；重量区间展开。
5. 蓝天禽蛋联盟：元/箱；只纳入配置指定的目标市场；重量区间内各码使用同一报价。目标市场名称需用真实数据核对，不能依赖截图文字识别结果直接硬编码。
6. 湖南三尖农牧公司：元/箱；精确码数或重量区间展开；价格存在上下限时拆低价、高价列。
7. 成都鸡蛋价格：元/箱；净重区间内各码使用同一报价；价格区间拆低价、高价列。
8. 河北辛集城方蛋品：元/箱；精确码数或重量区间展开；价格存在上下限时拆低价、高价列。
9. 江西九江褐壳蛋：元/箱；精确码数或重量区间展开；价格存在上下限时拆低价、高价列。

最终列数由规则注册表和当日适用规则决定，但顺序固定。同一公众号的低价列在前、高价列在后。

## 最近价差外推算法

缺失码数不是简单复制邻近价格，而是使用距离缺失码数最近的一组已知价差进行线性外推。

### 基本算法

1. 每个公众号、每个价格列、每个报价日期独立计算。
2. 先完成所有原始精确匹配和原始区间展开，得到已知码数价格集合。
3. 对缺失码数寻找距离最近的两个已知码数，并计算单位码差价：`(价格2 - 价格1) / (码数2 - 码数1)`。
4. 向高码外推时优先采用缺失区间下方最近的一组已知值；向低码外推时优先采用缺失区间上方最近的一组已知值。
5. 内部缺口上下距离相同时，优先使用较小码数一侧的价差。
6. 多个连续缺失码沿用同一最近价差逐级推算，直到遇到新的原始已知值或到达 30/50 码边界。
7. 只有一个已知价格、不能形成价差时保持空白。
8. 不跨公众号、不跨日期、不跨低高价列计算。
9. 推算过程中使用 Decimal，按该公众号原始价格精度展示，避免浮点误差。

### 示例

已知 35 码 200 元、36 码 204 元、37 码 208 元、38 码 212 元、39 码 214 元、40 码 216 元：

- 34 码和 33 码使用距离最近的 35→36 价差 4 元，得到 196 元和 192 元。
- 41 码和 42 码使用距离最近的 39→40 价差 2 元，得到 218 元和 220 元。

### 可解释性

每个推算单元格保留：

- 使用的两个已知码数及价格；
- 单位码差价；
- 推算方向；
- `extrapolated` 标记。

页面以浅色标识或“推算”角标区分推算值，并通过悬浮提示展示依据。

## 后端实现方案

1. 在安全结果查询仓储中新增矩阵原始行查询或扩展专用 DTO，只选择矩阵所需的白名单字段，禁止返回正文、链接和原始 JSON。
2. 查询先定位目标报价日，再获取九个公众号当日数据。
3. 服务层按公众号和文章分组，选出当天最新文章。
4. 规则引擎先执行市场过滤、码数解析、区间展开、低高价拆分和冲突决胜。
5. 在原始映射完成后执行最近价差外推，生成 50 至 30 码完整行。
6. 路由将矩阵模型传入模板；无数据时展示明确空状态。
7. 原有 `list_price_details` 继续保留，供内部测试和后续原始明细核验使用。

## 前端影响

1. 页面标题改为“公众号报价矩阵”，说明当前为同日报价横向对比。
2. 顶部保留报价日期选择器，默认值显示后端确定的最新日期。
3. 展示报价日期、最近更新时间和来源数量。
4. 表头按公众号分组；双列公众号使用二级表头“低价”“高价”；单位放在公众号表头中。
5. 码数列固定在左侧，表头固定，宽表支持横向滚动。
6. 原始报价使用正常文本；推算值使用低强调背景或角标，并提供推算依据提示。
7. 无法生成的单元格保持空白。
8. 表格下方展示与公众号对应的取数规则说明，移动端改为按公众号排列的规则卡片，避免超宽底部单元格难以阅读。
9. 延续管理后台方向 A 的视觉体系，不直接复制 Excel 的边框、字体和密度。
10. 可保留默认收起的原始明细核验入口，但首版不把它作为矩阵实现的阻塞项。

## 权限与安全

1. 沿用现有 Web 登录和结果页面权限，不新增角色。
2. 查询严格使用参数绑定和字段白名单。
3. 页面不输出文章链接、正文、原始行 JSON 或其他敏感采集内容。
4. 推算值必须可识别，不得伪装成公众号原始报价。

## 测试与回归方案

### 单元测试

1. 默认日期选择最大报价日期。
2. 指定历史日期只返回该日矩阵。
3. 同公众号同日多篇文章只采用发布时间最新的一篇。
4. 九个公众号列顺序、名称和单位正确。
5. 30 至 50 码完整生成并按降序输出。
6. 精确码数、重量区间展开和区间边界冲突规则正确。
7. 价格上下限正确拆成低价、高价列。
8. 最近价差向下、向上及内部缺口外推正确。
9. 价差变化时始终采用距离缺失码数最近的一组。
10. 上下距离相同时使用较小码数一侧价差。
11. 只有一个已知值时不推算。
12. 低价、高价互不串值，公众号和日期之间互不串值。
13. Decimal 计算和显示精度正确。
14. 原始值不会被推算值覆盖。

### 查询与 Web 测试

1. 查询只选择允许展示的字段，不返回正文、链接或原始 JSON。
2. 日期参数非法时返回现有 422 错误页。
3. 页面包含分组表头、固定码数行、单位、规则说明和推算标记。
4. 无数据日期显示空状态。
5. 非 WeRSS 九公众号数据不进入矩阵。
6. 现有群消息结果和文章结果页面不受影响。

### 视觉回归

1. 启动连接当前 MySQL 和真实数据的正式开发服务。
2. 使用真实登录态打开报价结果页面。
3. 分别检查桌面宽屏、窄屏横向滚动、固定表头和固定首列。
4. 截图核对列分组、行对齐、单位、空值、原始值和推算值辨识度。
5. 由实现者先自行判断视觉效果并修正明显问题，再提交用户验收。

## 风险与分阶段落地建议

### 风险

1. 不同公众号的规格文本并不完全一致，可能出现无法解析的码数。
2. 截图中的市场名称可能存在识别误差，必须用真实数据核对后配置。
3. 线性外推是业务规则，不等于公众号原始报价，若缺少视觉标识容易造成误解。
4. 同一公众号文章中可能包含多个市场或产品，过滤不足会导致冲突。
5. 宽矩阵在小屏幕上天然需要横向滚动。

### 分阶段落地

1. 第一阶段：实现规则注册表、最新文章选择、原始区间展开和矩阵领域模型。
2. 第二阶段：实现最近价差外推和可解释性元数据。
3. 第三阶段：完成矩阵模板、规则说明、响应式样式和视觉回归。
4. 第四阶段：用真实九公众号数据核验市场过滤与各公众号专属规则，修正规则配置后发布。

## 验收标准

1. `/results/prices` 默认展示最新报价日的 50 至 30 码矩阵。
2. 用户可以切换历史报价日期。
3. 同公众号同日仅使用最新文章。
4. 河南金咕咕显示元/斤，其余显示元/箱。
5. 价格区间按低价、高价拆列。
6. 缺失码数严格按最近一组已知价差外推，并明显标识为推算值。
7. 表格底部或规则区完整展示各公众号固定取数规则。
8. 页面不展示非 WeRSS 九公众号数据。
9. 页面在宽屏和窄屏下均可读，完成真实数据截图检查。
10. 全量自动化测试通过，现有采集、清洗、分析及其他结果页面无回归。
