# WeRSS 正文按需读取设计

## 背景与现状

WeInsight 已使用 WeRSS 的标准 RSS Feed 替换公众号 RPA。当前 RSS 适配层只将标题、原文链接、作者、摘要、发布时间和采集时间写入 `wechat_article_raw`，后续 `clean_article` 任务仍通过 Playwright 打开微信公众号原文页取得正文元数据。

WeRSS 已在同一台 Windows 采集机的 Docker Desktop 中保存公众号正文。继续只依赖公众号原文页会保留页面失效、登录限制和微信风控风险；但把完整正文或 HTML 再复制到 WeInsight，会改变现有“不长期保存文章正文”的数据保留边界。

## 目标与非目标

### 目标

1. `clean_article` 任务优先从本机受控 WeRSS 读取文章正文。
2. WeRSS 无正文或读取失败时，按明确规则回退到微信公众号原文页解析。
3. 正文仅在单次任务内存中存在；WeInsight 只保存正文长度、内容哈希、正文来源、解析状态和既有结构化结果。
4. 复用现有清洗、报价提取、分析和日报任务，不让这些下游组件直接依赖 WeRSS。
5. WeRSS 故障只造成文章正文任务重试，不影响 RSS 链接采集、微信群 RPA 或已完成结果查询。

### 非目标

1. 不在 WeInsight 持久化原始 HTML、纯文本正文、图片二进制或 Cookie。
2. 不直接读取或耦合 WeRSS 私有数据库表。
3. 不开放 WeRSS 到公网，不支持任意正文服务地址。
4. 首期不做图片 OCR、复杂 HTML 表格还原或正文版本归档。

## 方案选择

采用“正文只存 WeRSS，WeInsight 按需读取”的方案。WeRSS 是正文唯一持久化来源；WeInsight 在任务执行时通过本机 HTTP 读取正文，在内存中净化和提取文本，随后只写结构化结果和安全元数据。

相比在 WeInsight 保存纯文本，该方案避免正文双份存储和新的数据保留责任；相比同时保存 HTML 与纯文本，它显著降低存储、权限、备份和危险 HTML 管理成本。代价是待处理任务依赖 WeRSS 在线，因此必须提供受控回退、重试和可观测性。

## 影响范围

- RSS Feed 模型与映射：保存可用于定位 WeRSS 正文的稳定条目标识或受控正文 URL 元数据。
- `wechat_article_raw`：新增正文定位及来源元数据，不新增正文列。
- 文章解析服务：从单一浏览器解析器改为正文提供器编排。
- `wechat_article_clean`：新增 `content_source`、`content_hash` 和正文获取状态元数据，不新增正文列。
- 配置：增加 WeRSS 正文端点、超时和响应大小限制；继续复用私有地址精确允许名单。
- 监控与运维：增加 WeRSS 命中、网页回退、失败重试和积压指标。
- 群采集链路、微信 UI 锁、文章分析结果结构和日报输出不改变。

## 数据模型

### `wechat_article_raw`

新增以下可空字段：

- `content_locator`：WeRSS 内部文章标识或相对正文路径，不保存认证信息。
- `content_locator_type`：首期固定为 `werss_article_id` 或 `werss_relative_path`。

RSS 条目没有可靠定位信息时保持为空，后续直接走原文页回退。

### `wechat_article_clean`

新增以下字段：

- `content_source VARCHAR(20)`：`werss` 或 `web`。
- `content_hash CHAR(64)`：净化后正文文本的 SHA-256，用于一致性核对，不可用于恢复正文。
- `content_fetch_status VARCHAR(30)`：成功时为 `success`；任务失败仍由现有任务表记录，不写伪成功 clean 记录。

继续保留 `content_length`。禁止新增 `content_html`、`content_text` 或其他可恢复正文的字段。

迁移使用 MySQL 8.x 可重复执行的 `information_schema` + 存储过程模式；`sql/init.sql` 与迁移保持一致。

## 后端实现方案

### 正文提供器边界

定义统一接口：

```python
class ArticleContentProvider(Protocol):
    def parse(self, source: ArticleParseSource) -> ParsedArticleContent: ...
```

实现三个组件：

1. `WeRssArticleContentProvider`：只接受配置的 `127.0.0.1:8001` 端点和已校验 locator；读取正文，执行大小限制和 HTML 净化，返回内存中的 `ParsedArticleContent`。
2. `PlaywrightArticleContentProvider`：包装现有公众号原文页解析逻辑。
3. `FallbackArticleContentProvider`：先调用 WeRSS；只有“locator 缺失、正文不存在、正文为空、可恢复 HTTP/解析错误”时调用网页提供器。安全校验失败、响应超限等策略错误不得静默回退，避免绕过安全控制。

`ArticleParseService` 只依赖统一提供器。提供器返回 `content_source` 和用于本次分析的净化文本；服务计算 `content_hash`，写入 clean 元数据并创建既有分析任务。任务结束后不把正文写入数据库、日志或异常摘要。

### WeRSS 正文定位

首选从 Feed 条目中提取 WeRSS 提供的稳定文章标识；若 Feed 只提供公众号原文链接，则适配器可以从同源、固定模板的 WeRSS 字段或相对链接取得 locator。禁止根据不受信任 URL 拼接任意目标地址。

WeInsight 只调用公开、稳定的本机 HTTP 路由，不读取 WeRSS 私有表。若当前 WeRSS 版本没有合适的稳定正文接口，先使用其同源文章查看路由作为受控适配器，并用契约测试固定响应；镜像升级前必须重新验证该契约。

### HTML 处理

- 响应上限默认 5 MiB，连接与读取超时默认 30 秒。
- 只接受 `text/html` 或明确支持的 JSON 内容类型。
- 移除 `script`、`style`、`iframe`、事件属性和不可见内容。
- 提取可见纯文本用于现有清洗与分析，正文不得落调试文件。
- 哈希基于规范化纯文本 UTF-8 字节计算。

## 前端影响

来源管理页不新增正文编辑或查看能力。运行结果和监控页可显示：

- 正文来源：WeRSS / 网页回退。
- 正文长度和内容哈希短前缀。
- WeRSS 正文获取成功数、网页回退数和失败数。

页面不得展示正文、完整内容哈希以外的可恢复内容、WeRSS 管理凭据或内部认证参数。

## 错误处理与重试

- WeRSS 暂时不可用、超时或正文尚未同步：尝试网页回退；网页也失败则沿用现有 `clean_article` 最多 3 次、间隔 60 秒的重试。
- locator 不存在：直接网页回退，并记录结构化原因 `werss_locator_missing`。
- WeRSS 返回空正文：网页回退，记录 `werss_content_empty`。
- 非允许端点、重定向越界、响应超限或内容类型不允许：任务失败并记录结构化安全错误，不记录响应体。
- 两种来源都失败：任务错误摘要只包含错误类型和结构化代码，不包含正文、Cookie 或完整敏感 URL。
- WeRSS 恢复后，现有失败任务可通过受控重试入口重新执行。

## 权限与安全

1. 正文端点固定为配置中的本机 WeRSS 地址，必须通过现有私有地址精确允许名单。
2. HTTP 客户端沿用 DNS/地址固定、逐跳重定向验证和响应大小限制。
3. WeInsight 不持有 WeRSS 管理员密码；若正文路由需要令牌，令牌只通过本机环境变量注入，不进数据库、日志或任务快照。
4. collector/pipeline 数据库角色仅获得新增元数据列所需权限，不增加正文表权限。
5. 日志、监控、异常和测试夹具不得包含真实文章正文。

## 测试与回归方案

### 单元测试

- Feed 能提取合法 locator，缺失或越界 locator 被拒绝。
- WeRSS 提供器验证端点、重定向、内容类型、大小、超时、空正文和 HTML 净化。
- fallback 的允许与禁止回退矩阵。
- `content_hash` 基于规范化文本且结果稳定。
- repo 只写来源、哈希、长度等元数据，不出现正文列或正文参数。

### 集成测试

- 使用本机假 WeRSS 服务完成“Feed 入 raw → WeRSS 正文 → clean → analyze task”。
- WeRSS 关闭时网页回退成功。
- 两者都失败时任务重试且日志无正文。
- MySQL 8.4 首次和二次执行迁移均成功。

### 真实 POC

在现有“一箱蛋”单公众号 24 小时 POC 中增加：

- WeRSS 正文获取成功率、网页回退次数、失败次数。
- clean/analyze 任务积压和最终成功率。
- 同一文章经旧网页解析与 WeRSS 路径得到的正文长度、哈希和结构化报价差异。
- 重启 WeRSS 后失败任务可恢复。
- RSS 正文处理期间 `wechat_ui_lock` 的 article 持有数仍为 0；只有网页回退使用浏览器，不使用微信 UI。

## 分阶段落地

### 阶段一：影子验证

实现 WeRSS 提供器和元数据记录，但对受控样本同时运行 WeRSS 与网页解析，只比较长度、哈希和结构化结果；不改变生产任务的最终来源。

### 阶段二：正文优先

单公众号验证通过后启用 WeRSS 优先、网页回退。持续观察至少 24 小时，要求正文任务无永久失败、结构化报价无不可解释退化、日志无正文泄漏。

### 阶段三：扩大范围

按 1 → 3 → 最多 20 个公众号逐步扩大。每次 WeRSS 镜像升级必须重跑正文接口契约测试和一个真实公众号样本。

## 验收标准

1. WeInsight 数据库、日志和运行目录中不存在文章正文或原始 HTML。
2. 合法 WeRSS 正文可以直接驱动现有 clean、analyze、报价和日报链路。
3. WeRSS 正文不可用时按策略回退或重试，RSS 链接采集与微信群链路不受阻塞。
4. 所有安全边界、迁移幂等、回退矩阵和敏感输出测试通过。
5. “一箱蛋”连续 24 小时观察无永久正文任务失败，且结构化结果与网页基线无不可解释差异。

## 风险与缓解

- WeRSS 非公开正文路由随版本变化：固定镜像摘要、增加契约测试、升级前验证。
- 正文同步晚于 Feed：空正文采用网页回退或任务重试，不把空结果标为成功。
- 下游当前只使用内容长度而未真正消费正文：实施时需把净化正文作为内存参数传给分析/提取组件，同时用测试证明任务结束后不持久化正文。
- WeRSS 单点故障：保留网页回退、独立备份和失败任务重试；不让该故障扩散到微信群链路。
