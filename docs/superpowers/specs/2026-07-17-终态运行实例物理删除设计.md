# 终态运行实例物理删除设计

## 背景与现状

运行实例页面当前仅支持列表和详情查看，没有删除入口。`wechat_collection_job_target_run` 和 `wechat_collection_job_event` 通过 `ON DELETE RESTRICT` 引用 `wechat_collection_job_run`，直接删除运行实例会被数据库拒绝。页面只展示最近 3 个自然月，超出范围的运行实例仍保留在数据库中。

用户确认允许从数据库物理删除 `success`、`failed`、`partial_success`、`cancelled`、`aborted` 等终态运行实例。

## 目标与非目标

### 目标

- 管理员可从运行详情页物理删除单个终态运行实例。
- `queued`、`running` 等非终态实例必须拒绝删除。
- 在一个数据库事务中删除关联事件、目标运行明细和运行实例。
- 删除成功后保留独立的管理员审计事件。
- 前端提供明确的不可逆提示和二次确认。
- 完整覆盖状态规则、事务顺序、权限和 Web 交互测试。

### 非目标

- 不批量删除运行实例。
- 不删除采集原始数据、清洗分析结果、日报或业务结果。
- 不删除 `wechat_group_collect_log`、`wechat_article_collect_log` 等采集日志。
- 不删除目标运行记录中引用的本机故障截图文件。
- 不改变运行实例最近 3 个月的页面可见范围。
- 不自动清理历史运行实例；定期保留策略后续单独设计。

## 影响范围

- 服务层：运行状态校验、删除结果模型和安全错误。
- 存储层：终态校验、行锁、关联数据删除和审计写入。
- Web 层：删除 POST 路由、CSRF、严格表单字段和错误响应。
- 前端：运行详情页终态删除按钮及确认提示。
- 权限：生产 Web 角色增加三张运行相关表的 DELETE 权限。
- 事件展示与保留：新增运行删除审计事件摘要，并纳入 6 个月审计保留。

## 数据模型与权限模型

不修改表结构和外键。删除顺序遵循现有 `RESTRICT` 约束：

1. 锁定 `wechat_collection_job_run` 目标行并读取状态、任务 ID。
2. 删除 `run_id` 指向运行实例或 `target_run_id` 指向其目标明细的 `wechat_collection_job_event`。
3. 删除 `wechat_collection_job_target_run` 明细。
4. 删除 `wechat_collection_job_run` 实例。
5. 插入 `collection_run_deleted` 管理员审计事件；该事件只关联仍存在的任务 ID，`run_id` 和 `target_run_id` 为空，已删除运行 ID 写入指标 JSON。

生产 `weinsight_web_role` 需要：

- `wechat_collection_job_run`：`SELECT, DELETE`；
- `wechat_collection_job_target_run`：`SELECT, DELETE`；
- `wechat_collection_job_event`：`SELECT, INSERT, DELETE`。

管理员身份继续由现有会话中间件和 CSRF 校验保护。审计事件记录管理员用户名，但不记录敏感请求信息。

## 后端实现方案

新增 `RuntimeMonitorService.delete_terminal_run(run_id, actor, now)`：

- 校验正整数运行 ID、非空管理员名称和上海时区时间；
- 调用仓库事务删除；
- 未找到返回 `RunNotFoundError`；
- 非终态返回 `RunDeletionNotAllowedError`；
- 成功返回包含运行 ID、任务 ID、原状态、删除事件数和目标明细数的结果。

仓库通过 `SELECT ... FOR UPDATE` 在同一事务内重新确认状态，避免运行状态检查与删除之间发生竞争。任一步失败时整个事务回滚，不留下半删除数据或孤立审计。

新增接口：

- `POST /runs/{run_id}/delete`；
- 仅接受 `csrf_token` 和固定确认字段；
- 成功返回 303 到 `/runs`；
- 参数非法返回 422；
- 实例不存在返回 404；
- 状态不允许返回刷新后的详情和 409。

## 前端影响

- 删除入口仅显示在终态运行详情页，不在列表页提供快捷删除。
- 按钮使用危险操作样式，文案为“删除运行实例”。
- 浏览器确认文案明确说明：运行明细与结构化事件将永久删除，采集结果和采集日志仍保留。
- `queued`、`running` 页面不渲染删除表单。
- 删除成功后返回运行列表，被删除实例不再出现。

## 测试与回归方案

1. 服务层参数化验证五种终态均允许删除。
2. 验证 `queued`、`running` 被拒绝，且仓库不发生删除。
3. 验证不存在实例返回安全 404，不泄露内部数据。
4. 仓库测试验证 `FOR UPDATE`、事件删除、目标明细删除、运行删除和审计插入的固定顺序及绑定参数。
5. 仓库集成测试验证真实 MySQL 外键下删除成功且审计保留。
6. Web 测试验证终态按钮、非终态隐藏、严格字段、CSRF、303、404、409 和 GET 不可用。
7. SQL 权限测试验证 Web 角色只有所需三张表获得 DELETE 权限。
8. 事件摘要和保留测试验证 `collection_run_deleted` 属于审计事件。
9. 执行相关定向测试与全量回归。
10. 使用本地页面完成“终态详情 → 确认删除 → 返回列表 → 实例消失”的真实浏览器验证，并检查控制台错误。

## 风险与分阶段落地建议

- 物理删除不可逆，必须依靠详情页入口、固定确认字段和浏览器二次确认降低误操作风险。
- 关联事件删除会让原运行的技术排障上下文消失；独立审计仅保留删除行为、原状态和删除计数。
- 采集结果、采集日志和截图不随运行删除，避免扩大用户授权的删除范围，但可能留下无法从运行页反查的历史业务数据。
- Web 角色权限必须与代码同步发布，否则生产删除会因权限不足回滚。
- 第一阶段先落单条终态删除；批量删除和自动保留策略不在本次范围。
