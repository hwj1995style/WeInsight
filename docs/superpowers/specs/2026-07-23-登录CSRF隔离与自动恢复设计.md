# 登录 CSRF 隔离与自动恢复设计

## 背景与现状

管理后台登录页采用 Double Submit Cookie：`GET /login` 生成随机 token，同时写入名为 `login_csrf` 的 Cookie 和登录表单隐藏字段；`POST /login` 要求两者完全一致。

当前实现存在两个可重复触发的问题：

1. Cookie 只按域名和路径匹配，不区分端口。开发环境 `127.0.0.1:8848`、端到端测试环境 `127.0.0.1:8849` 以及其他本机端口共用固定 Cookie 名 `login_csrf`。访问任一环境的登录页都会覆盖其他环境的 token。
2. 每次 `GET /login` 都无条件生成新 token。同一环境打开多个登录标签页、刷新其中一个标签页，或从浏览器历史恢复旧表单时，旧页面隐藏字段会立即失效。

此外，登录页停留超过 20 分钟后 Cookie 会过期。上述任一情况发生后，`POST /login` 当前直接返回纯文本 `CSRF validation failed`，用户只能手工刷新或清理 Cookie，缺少可恢复路径。

## 目标与非目标

### 目标

- 不同本机环境和端口使用不同的登录 CSRF Cookie 名，互不覆盖。
- 同一环境的多个登录标签页复用同一有效 token，刷新页面不使其他标签页失效。
- token 缺失、过期或不匹配时继续拒绝本次登录，不调用密码校验，但返回新的登录页面和 token，让用户可以直接重试。
- 保持 Double Submit Cookie、常量时间比较、SameSite、Secure 和有效期等安全边界。
- 清理旧版固定名 `login_csrf` Cookie，避免升级后继续干扰。

### 非目标

- 不取消登录 CSRF 校验。
- 不延长管理员 Session 有效期，不修改数据库 Session 模型。
- 不改变用户名密码校验、登录限流或锁定策略。
- 不为登录表单引入前端框架或 JavaScript token 同步。

## 影响范围

- 配置模型：认证配置增加独立的 `login_csrf_cookie_name`。
- 开发、E2E、POC 和生产配置模板：使用不同的登录 Cookie 名。
- 登录路由：复用有效 token、刷新 Cookie 有效期、错误后重新签发 token。
- 登录页面：显示可操作的过期提示，不再显示纯文本错误页。
- 测试：覆盖跨环境 Cookie、多个标签页、过期恢复、常量时间比较和 Cookie 安全属性。

## 数据模型与权限模型

不修改数据库表、角色或权限。

`AuthConfig` 新增 `login_csrf_cookie_name`，并校验 Session Cookie、已登录 CSRF Cookie、登录 CSRF Cookie 三个名称均为合法且互不相同的 Cookie 名。

建议配置：

- 开发：`weinsight_dev_login_csrf`
- E2E：`weinsight_e2e_login_csrf`
- POC：`weinsight_poc_login_csrf`
- 生产模板：`weinsight_prod_login_csrf`

## 后端实现方案

### 稳定 token

`GET /login` 优先读取当前环境配置的登录 CSRF Cookie：

- Cookie 是符合格式的现有 token 时，继续用于表单，并用相同值刷新 20 分钟有效期；
- Cookie 缺失或格式异常时，生成新的 32 字节 URL-safe token；
- 响应删除旧版固定名 `login_csrf` Cookie。

这样同一环境多个标签页共享稳定 token，不会因为刷新或打开新标签页互相失效。

### 安全失败与自动恢复

`POST /login` 仍使用 `secrets.compare_digest` 比较 Cookie 和表单 token。缺失或不匹配时：

1. 不进入登录限流和密码哈希，不调用认证服务；
2. 生成全新 token；
3. 返回 HTTP 403 的完整登录页面；
4. 显示“登录页面已过期或在其他窗口被刷新，请重新提交”；
5. 写入新环境 Cookie，使当前页面下一次提交可直接成功；
6. 删除旧版固定名 Cookie。

403 状态保留安全语义，完整 HTML 页面提供恢复能力。

### 登录成功

登录成功后继续设置 Session Cookie 和已登录 CSRF Cookie，同时删除当前环境的登录 CSRF Cookie 和旧版固定名 Cookie。

## 前端影响

- 正常登录页视觉和交互不变。
- token 失效后不再进入纯文本白页，而是在原登录卡片中显示错误提示。
- 用户重新输入密码或重新提交即可继续，无需手工清理浏览器 Cookie。

## 测试与回归方案

1. 配置测试验证四套配置使用不同登录 CSRF Cookie 名。
2. 验证连续两次 `GET /login` token 不旋转，并刷新相同 Cookie 的有效期。
3. 验证另一个环境的 Cookie 不影响当前环境登录。
4. 验证缺失、错误和过期 token 均返回 HTML 403、重新签发 token，且不调用认证服务。
5. 验证使用新 token 重试可以正常登录。
6. 验证恶意或异常格式 Cookie 会被替换而不反射到表单。
7. 保留常量时间比较、Secure、SameSite、Path、Max-Age 和登录限流测试。
8. 运行认证定向测试和项目全量测试。
9. 使用真实浏览器验证多标签/跨端口覆盖模拟、错误恢复和成功登录，检查控制台与页面状态。

## 风险与分阶段落地建议

- 配置字段为必填项，代码与所有部署配置必须同步发布；生产自定义配置需要增加该字段。
- 旧版 Cookie 会在首次打开登录页或登录成功时删除，不需要用户手工清理。
- 稳定 token 只在 20 分钟窗口内复用，不降低随机性或扩大 Session 权限。
- 仍可能发生真正过期或浏览器清理 Cookie，但会自动回到可恢复登录页，不再形成死路。
