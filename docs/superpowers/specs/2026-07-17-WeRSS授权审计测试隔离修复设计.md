# WeRSS 授权审计测试隔离修复设计

## 背景与现状

授权设置路由会把配置保存、参数校验失败、WeRSS 管理凭据测试和测试邮件发送结果写入采集事件表。`tests/test_web_sources.py` 虽然注入了假的授权服务，但没有注入事件仓库，`create_app` 因而回退到真实的 `MysqlCollectionEventRepo`，并使用 `config.dev.yaml` 指向的 `weinsight_dev`。

每次执行相关测试会向开发库写入固定的四条审计事件：配置更新、预期的参数校验失败、WeRSS 测试成功和邮件测试成功。2026-07-17 已产生 108 条此类测试事件。事件列表又按事件类型生成摘要，导致两类成功测试都显示为“授权提醒连接测试成功”。

## 目标与非目标

### 目标

- Web 路由单元测试中的授权审计事件只写入内存仓库。
- 授权路由测试不再向开发库新增事件。
- 保留并验证成功、失败事件的级别、类型、原始消息和指标。
- 在事件列表中区分 WeRSS 管理凭据测试与测试邮件发送。
- 形成历史测试事件的精确清理清单。

### 非目标

- 本次不建设独立测试数据库；数据库 Repository 集成测试后续再单独规划。
- 不改变生产环境审计事件的写入规则和保留策略。
- 不调整授权配置、凭据存储或邮件发送逻辑。
- 未经确认不删除历史事件。

## 影响范围

- 测试依赖：`tests/test_web_sources.py`。
- 事件摘要：`app/services/runtime_monitor_service.py` 及对应测试。
- 历史数据：仅生成满足固定测试签名的事件 ID 清单，不在本次代码执行中自动删除。
- 后端接口地址、请求结构和响应结构保持不变。

## 数据模型与权限模型

不修改表结构、字段、索引或管理员权限。生产请求仍使用当前管理员身份写入真实事件仓库。

测试新增内存事件仓库，只实现 `append_event` 并保存 `NewCollectionEvent` 对象。测试仓库不连接 MySQL，也不生成跨进程持久数据。

## 后端实现方案

1. 在 `tests/test_web_sources.py` 增加 `RecordingEventRepo`，记录传入的事件并返回递增 ID。
2. 测试应用工厂显式传入该仓库，阻断 `create_app` 对真实事件仓库的默认构建结果被授权路由使用。
3. 为四类授权操作增加审计断言，覆盖事件类型、级别、消息和 `metrics_json`。
4. 保持 `_append_authorization_audit` 的生产逻辑不变。
5. 扩展事件视图摘要生成：根据授权测试事件的原始消息或指标目标，分别生成 WeRSS 与邮件测试摘要；未知目标保留通用摘要。

## 前端影响

事件页面结构和交互不变，仅结果摘要文案更精确：

- `WeRSS管理凭据测试成功` 显示为“WeRSS 管理凭据测试成功”。
- `授权提醒测试邮件发送成功` 显示为“授权提醒测试邮件发送成功”。
- 对应失败事件同样区分 WeRSS 和邮件目标，并保留 `WARN` 前缀。

## 测试与回归方案

1. 运行授权设置保存成功测试，断言内存仓库收到 `werss_authorization_settings_changed`。
2. 运行未知字段测试，断言返回 422，内存仓库收到 warning 级别且原因为 `validation`。
3. 运行两个连接测试，断言分别产生 WeRSS 和 email 目标事件。
4. 为事件视图增加摘要区分测试，并保留未知目标回退用例。
5. 执行 `tests/test_web_sources.py` 和 `tests/test_runtime_monitor_service.py`。
6. 执行全量测试。
7. 在定向及全量测试前后查询开发库授权事件总数，必须保持不变。
8. 使用本地事件页面验证两类摘要可区分，页面无相关控制台错误。

## 历史数据清理方案

先查询并导出候选事件 ID、时间、类型、消息和指标。候选记录必须满足同一批次内依次出现以下固定签名：

1. `werss_authorization_settings_changed`；
2. `werss_authorization_settings_failed` 且 `reason=validation`；
3. `werss_authorization_test_succeeded` 且 `target=werss`；
4. `werss_authorization_test_succeeded` 且 `target=email`。

清理前保留 JSON 或 SQL 备份，并由用户确认候选范围。删除不作为应用启动、迁移或测试的一部分。

截至 2026-07-17 的只读核对结果为 46 组、184 条候选事件，ID 范围为 12591–13105，时间范围为 2026-07-16 18:10:04 至 2026-07-17 16:58:48。其中 2026-07-17 共 27 组、108 条。该结果仅作为待确认清单，未执行删除。

## 风险与分阶段落地建议

- 第一阶段完成测试仓库注入和审计断言，立即停止新增污染。
- 第二阶段完成摘要区分和页面验证。
- 第三阶段在用户确认后备份并清理历史测试事件。
- 当前测试应用仍可能因为其他未注入服务创建 MySQL Engine，但授权审计不再使用真实事件仓库。本次通过数据库计数零变化验证边界；完整测试数据库隔离后续独立实施。
- 摘要区分依赖事件消息和指标兼容旧数据，未知格式必须回退通用文案，避免事件页渲染失败。
