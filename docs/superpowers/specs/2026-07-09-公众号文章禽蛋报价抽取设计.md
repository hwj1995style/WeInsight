# 公众号文章禽蛋报价抽取设计

## 背景与现状

当前公众号/订阅号链路已经完成文章链接采集、Playwright 解析、`wechat_article_clean` 入库、`wechat_article_analysis` 分析和文章日报生成。现有 `wechat_article_analysis.price_items_json` 只保存少量规则抽取结果，真实 `analyze_article` 阶段默认使用 `NoopArticleTransientExtractor`，不会重新打开文章链接读取正文，也不会从不同公众号格式中抽取完整报价明细。

已采集和补充分析的 7 个真实公众号样本显示，报价格式差异明显：

| 公众号 | 样本格式 | 首版解析方式 |
| --- | --- | --- |
| 一箱蛋 | 全国长文本列表，含“报价单位：30斤”“徐州-丰县鸡蛋价格 133 稳”等大量行 | 文本行解析，继承报价单位和地区上下文 |
| 福建闽融鸡蛋报价平台 | 编号短句，如“红蛋价格：4.90元/筐装(稳)” | 文本行解析 |
| 信立鸡蛋当日价格 | 分组标题加行报价，如“停车场红壳蛋带包装价”“毛重49-51斤223-223元稳定” | 文本块解析，继承分组标题 |
| 贵阳鸡蛋价格 | DOM 表格和正文行，含“大码 52斤以上 228-233” | DOM 表格优先，文本行补充 |
| 上海禽蛋价格综合报价 | 纯文本短价目清单，含洋鸡蛋、绿壳蛋、鸭蛋、咸蛋、鹌鹑蛋等 | 文本行解析，按产品族分类 |
| 家美鲜鸡蛋 佳美鲜 | DOM 表格，含主表、绿壳蛋、草鸡蛋、贵州/广西区域报价等 | DOM 表格优先，保留表格级上下文 |
| 河北馆陶鸡蛋报价 | DOM 表格，含“360枚/箱”“45斤 192元 192元 0”等 | DOM 表格解析，继承全局规格和备注 |

由此可见，`price_items_json TEXT` 不适合承载完整长文报价明细，尤其是一箱蛋这类一篇可能包含数百到数千条报价行的文章。首版需要新增明细表保存逐行报价，`price_items_json` 仅作为兼容预览和汇总。

## 目标与非目标

### 目标

1. 在 `analyze_article` 阶段重新打开文章链接，运行时读取正文和 DOM 表格。
2. 从 DOM 表格、文本行、文本块中抽取全量禽蛋报价明细。
3. 首版抽取全量禽蛋报价，再通过 `product_family=chicken_egg` 和 `include_in_egg_price=1` 支持鸡蛋主口径过滤。
4. 新增 `wechat_article_egg_price_item` 保存完整逐行明细，避免长文 JSON 溢出。
5. 保留公众号、文章、发布时间、来源位置、表格级上下文、产品、规格、重量、价格、涨跌、趋势、置信度和解析备注。
6. 不长期保存文章正文、HTML、图片、截图或 OCR 原文。
7. 保持 group 链路隔离，所有新增任务和表只属于 article 链路。

### 非目标

1. 首版不接入 PaddleOCR，不新增 OCR 依赖。
2. 首版不处理纯图片报价；仅记录 `image_quote_not_supported_v1` 质量备注。
3. 首版不做规格统一换算，不把 30 斤、27.5 斤、360 枚/箱等转换为统一单位。
4. 首版不建设前端页面。
5. 首版不改变文章采集规则，不为了报价抽取扩大采集范围。
6. 首版不调用外部 AI，不把正文发给模型。

## 影响范围

### 后端模块

1. `app/domain/article_analysis.py`
   - 增加禽蛋报价 item 数据结构和解析入口。
   - 保留原有文章摘要、标签、表格摘要能力。
2. `app/pipelines/article_analysis_service.py`
   - 将默认 Noop transient extractor 替换为可配置的 Playwright transient extractor。
   - 分析结果需要同时写 analysis 和 egg price item。
3. `app/pipelines/article_parse_service.py`
   - 可复用 Playwright browser executable 解析逻辑。
   - 不修改 clean 表长期存储正文的边界。
4. `app/storage/article_analysis_repo.py`
   - 增加明细表写入接口。
   - 在同一 article 事务内 upsert analysis，并按 `article_hash` 删除重插明细。
5. `app/main.py`
   - `build_real_article_analysis_service` 需要装配真实 extractor。
   - CLI 输出仍只显示计数，不输出正文、链接或报价全文。
6. `config/config.dev.yaml` 和 `config/config.prod.example.yaml`
   - 增加报价抽取开关和预览条数配置。
7. `sql/init.sql` 和新迁移脚本
   - 新增 `wechat_article_egg_price_item`。

### 数据库

新增 article 明细表，不修改 group 表。`wechat_article_analysis.price_items_json` 仍保留，但只保存预览、统计和截断标记。

### 运行环境

首版不新增第三方依赖。继续使用当前 Python、Playwright、MySQL 运行环境。

### analysis 表兼容字段

`wechat_article_analysis.price_items_json` 首版保存：

```json
{
  "version": "egg_price_v1",
  "total_item_count": 128,
  "preview_limit": 20,
  "truncated": true,
  "items": []
}
```

`extracted_tables_json` 保存表格摘要和质量信息，不保存完整正文或 HTML：

```json
{
  "version": "egg_table_summary_v1",
  "tables": [
    {
      "source_media_type": "dom_table",
      "source_table_index": 0,
      "title": "通货装车价（含包装）",
      "headers": ["净重", "价差", "昨日价", "今日价", "涨跌"],
      "row_count": 14,
      "parsed_item_count": 14
    }
  ],
  "notes": []
}
```

### 权限与安全边界

1. 只读取 `wechat_article_clean.article_url` 并运行时打开页面。
2. 不写入正文、HTML、图片、截图或 OCR 原文。
3. CLI、日志、日报和汇总日报不得输出文章链接、正文或完整报价原文列表。
4. 新增表只保存结构化字段和必要的行级原文片段，如 `raw_row_json`，不保存整篇原文。

## 后端实现方案

### 分析流程

```text
wechat_article_process_task(analyze_article)
-> MysqlArticleAnalysisRepo.list_pending_analyze_articles
-> PlaywrightArticleTransientExtractor.extract(article)
-> extract DOM body text, DOM tables, line blocks
-> EggPriceExtractor.extract(transient_data)
-> analyze_clean_article builds summary/preview
-> repo.upsert_article_analysis_with_price_items
-> create article_daily_report task
-> mark analyze task success
```

### extractor 设计

新增 `PlaywrightArticleTransientExtractor`：

1. 复用 `resolve_playwright_browser_executable_path`。
2. 打开 `article_url` 后只读取：
   - `#js_content` 或 `.rich_media_content` 的可见文本。
   - DOM table 的表头、行和附近标题。
   - 大图数量和是否疑似图片报价的摘要备注。
3. 关闭浏览器后只返回内存结构，不落正文。

新增 `EggPriceExtractor`：

1. `extract_from_dom_tables`
   - 适配家美鲜、贵阳、河北馆陶。
   - 表格标题、前置行和备注作为上下文继承到每条 item。
2. `extract_from_text_lines`
   - 适配一箱蛋、福建闽融、信立、上海。
   - 支持报价单位、地区块、产品标题、包装说明继承。
3. `classify_product_family`
   - 洋鸡蛋、红蛋、粉蛋、绿壳蛋、草鸡蛋、普通鸡蛋归入 `chicken_egg`。
   - 鸭蛋归入 `duck_egg`。
   - 鹌鹑蛋归入 `quail_egg`。
   - 皮蛋、变蛋归入 `preserved_egg`。
   - 咸蛋、咸鸭蛋、海鸭蛋按语义归入 `salted_egg` 或 `duck_egg`，首版优先 `salted_egg`。
4. `infer_trend`
   - `涨`、正数涨跌为 `up`。
   - `跌`、`落`、负数涨跌为 `down`。
   - `稳`、`稳定`、`0` 为 `flat`。
5. `parse_price_range` 和 `parse_weight_range`
   - 只解析明显数字、区间和单位。
   - 不做跨规格换算。

### 7 个公众号适配要点

1. 一箱蛋
   - 上下文行 `报价单位：30斤` 写入 `quote_basis`。
   - `徐州-丰县鸡蛋价格 133 稳` 拆为 `region=徐州`、`market_name=丰县`、`price_text=133`、`trend=flat`。
2. 福建闽融鸡蛋报价平台
   - `红蛋价格：4.90元/筐装(稳)` 拆为 `product_name=红蛋`、`price_text=4.90元`、`package_policy=筐装`、`trend=flat`。
3. 信立鸡蛋当日价格
   - 分组标题 `停车场红壳蛋带包装价` 继承到后续行。
   - `毛重49-51斤223-223元稳定` 拆出重量区间、价格区间和趋势。
4. 贵阳鸡蛋价格
   - DOM 表格优先；广告表格需要通过标题、字段和行密度过滤。
   - `大码 | 52斤以上 | 228-233 | -` 保留码数、重量、价格区间。
5. 上海禽蛋价格综合报价
   - 纯文本短清单，一行一条。
   - 抽全量禽蛋报价，使用 `product_family` 区分鸡蛋、鸭蛋、咸蛋、鹌鹑蛋等。
6. 家美鲜鸡蛋 佳美鲜
   - DOM 表格优先。
   - 主表继承 `360枚/箱`、`通货装车价`、`含包装`。
   - 绿壳蛋、草鸡蛋、贵州、广西等表继承表格标题作为产品和地区上下文。
   - `以质论价`、`双色`、`32以下顺减-4` 写入 `row_note` 或 `parse_notes_json`。
7. 河北馆陶鸡蛋报价
   - DOM 表格优先。
   - 全局 `360枚/箱` 和“不含运费和包装费”写入 `quote_basis`、`package_policy` 或 `parse_notes_json`。

## 风险与分阶段落地建议

### 风险

1. 公众号格式继续变化。
   - 应对：解析规则集中在 `EggPriceExtractor`，新增账号只补规则和样本测试。
2. 一箱蛋等长文报价量很大。
   - 应对：完整明细落表，`price_items_json` 只保存预览和统计。
3. 广告、联系电话、鸡苗或兽药内容被误抽。
   - 应对：增加非报价词过滤、报价字段密度判断和 `no_price_data` 测试。
4. DOM 表格中存在合并单元格或标题行。
   - 应对：保留表格级上下文，允许行解析失败时写备注，不强行转换。
5. 纯图片报价后续出现。
   - 应对：首版记录 `image_quote_not_supported_v1`，二期再评估 OCR。
6. 结构化字段过多但标准化未完成。
   - 应对：首版保留原始语义，不做规格换算；标准化另起设计。

### 分阶段落地

1. 阶段 A：表结构和领域模型
   - 新增明细表、dataclass、JSON 预览结构和 schema 测试。
2. 阶段 B：DOM 表格和文本解析
   - 覆盖 7 个公众号样本，不接 OCR。
3. 阶段 C：接入真实 analyze_article
   - Playwright transient extractor 装配到真实 service。
4. 阶段 D：运行验收
   - 对 7 个公众号各跑 1 篇样本，检查明细和安全输出。
5. 阶段 E：二期评估
   - 如出现纯图片报价，再设计 OCR 依赖、模型缓存、图片临时处理和质量评估。

## 设计边界确认

1. 首版只做 DOM 表格、文本行和文本块抽取。
2. 首版不做 OCR，不新增 PaddleOCR 依赖。
3. 首版抽全量禽蛋报价，鸡蛋主口径通过 `product_family=chicken_egg` 过滤。
4. 完整报价明细落 `wechat_article_egg_price_item`。
5. `price_items_json` 只保留预览和统计。
6. 不保存正文、HTML、图片、截图或 OCR 原文。
7. 不改 group 链路。
8. 后续规格转换、统一标准、OCR 支持另行设计。
