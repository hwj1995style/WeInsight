# 九公众号报价定位、标准转换与缺价推算设计

## 背景与现状

当前系统已具备公众号文章采集、文章分析、蛋价明细落库和 30–50 码报价矩阵展示能力，但现行分析链路使用 `ProviderBackedArticleTransientExtractor`，只保留已归一化的正文文本，主动清空 HTML 表格。这会丢失家美鲜、贵阳、馆陶、成都、辛集和九江等文章中已存在的结构化报价。

对 2026-07-15 实际文章 DOM 的只读核对结果如下：

| 公众号 | 目标报价形态 | 首选方式 |
| --- | --- | --- |
| 家美鲜鸡蛋 佳美鲜 | 多个 HTML 表格中的通货装车主表 | DOM 表格定位 |
| 河北馆陶鸡蛋报价 | HTML 表格 | DOM 表格定位 |
| 河南金咕咕蛋品 | 正文无文字和表格，报价仅在图片中 | OCR 兜底 |
| 贵阳鸡蛋价格 | HTML 表格 | DOM 表格定位 |
| 蓝天禽蛋联盟 | 正文文本中的阜阳精品蛋大码段 | 文本块定位 |
| 湖南三尖农牧公司 | 报价仅在单张大图中 | OCR 兜底 |
| 成都鸡蛋价格 | HTML 表格 | DOM 表格定位 |
| 河北辛集城方蛋品 | HTML 表格 | DOM 表格定位 |
| 江西九江褐壳蛋 | 褐壳和粉壳多个 HTML 表格 | 同时定位褐壳、粉壳两张主表并分类 |

现有矩阵根据价格是否存在 `price_high` 动态决定是否拆列，与已确认的“只有贵阳拆低价、高价，其他账号都输出单列”不一致。现有缺价推算已具备相邻价差思路，但需增加“同距离候选取绝对每码价差最小者”的确定性规则。

## 目标与非目标

### 目标

1. 在抽取价格前，先按账号定位当日文章中唯一的目标报价块。
2. 对七个可文本化账号优先使用 DOM 表格或正文文本，不调用 OCR。
3. 只对河南金咕咕蛋品和湖南三尖农牧公司使用本机 Windows OCR 兜底。
4. 将原始报价转换为 30–50 码汇总矩阵，只有贵阳保留低价、高价两列。
5. 已观测价格永不被推算值覆盖；缺失码数按最小临近区间差值推算补全。
6. 推算值保留 `extrapolated` 标识、依据码数、依据价格和每码差值，可在前端查看。
7. 不保存文章全文、HTML、原图、图片 URL 或 OCR 原文。

### 非目标

1. 不建设通用任意公众号 OCR 平台。
2. 不调用外部 AI 或外部 OCR 服务。
3. 不修改微信群采集、分析和日报链路。
4. 不将江西九江粉壳表增加为第十个账号列；本期账号主口径为褐壳蛋。
5. 不使用一个账号的价格推算另一个账号的缺价。

## 影响范围

### 后端

1. `app/pipelines/article_transient_extractor.py`
   - 目标九账号改用 Playwright 读取 DOM 表格和图片候选。
   - 两个图片账号才调用 OCR。
2. `app/domain/egg_price_quote_locator.py`
   - 新增账号专属报价块定位和标准化。
3. `app/domain/article_egg_price.py`
   - 抽取前调用报价块定位。
   - 支持 `text_block` 和 `image_ocr` 表格来源。
4. `app/services/windows_ocr_service.py` 和 `scripts/windows_ocr.ps1`
   - 使用 Windows Runtime 简体中文 OCR，图片只写入系统临时目录，识别完立即删除。
5. `app/domain/price_matrix.py`
   - 固定贵阳为低价、高价两列，其他账号单列。
   - 增加成都价格区间取高价。
   - 完善最小临近区间差值推算。
6. `app/main.py` 和 `app/workers/pipeline_runtime_factory.py`
   - 装配文本优先、OCR 兜底的 transient extractor。

### 数据库

不新增表和字段。原始观测报价继续落入 `wechat_article_egg_price_item`；缺价推算仅在读取矩阵时生成，不回写为原始报价。

### 前端

不新增页面。现有报价矩阵继续展示 30–50 码：只有贵阳出现低价、高价子列；江西九江出现褐壳蛋、粉壳蛋两个品类列，两列都按单一报价映射。推算单元格继续显示“推算”标识和悬停说明。

## 数据模型或权限模型

### 临时 OCR 结果

OCR 返回临时结构：

```json
{
  "source_media_type": "image_ocr",
  "source_image_index": 0,
  "title": "湖南三尖精品蛋360枚/箱收购价",
  "headers": ["净重", "价差", "昨日价", "今日价", "涨跌"],
  "rows": [["48", "-1", "262", "262", "0"]]
}
```

该结构只在分析进程内存中存在。落库时仅写入既有明细结构和必要的行级审计字段，不写入整张图片的 OCR 文本。

### 权限和安全边界

1. 只读当前已授权公众号文章内的图片。
2. 图片 URL 必须为 HTTPS，且属于允许的微信图片域名。
3. 设置图片数量、尺寸、响应字节数和 OCR 超时上限。
4. 临时图片在 `finally` 路径删除。
5. CLI、日志、日报和数据库均不输出图片 URL、原图或整图 OCR 原文。

## 后端实现方案

### 两级定位

1. 文章级定位
   - 使用已采集的账号名、报价日期和文章标题。
   - 广告、论坛、次日趋势分析等无目标块文章返回 `no_price_data`。
2. 文章内定位
   - 表格使用标题、表头、报价基准、行数和数字密度组合判定，不固定依赖 DOM 序号；DOM 主报价表至少包含 2 行，排除趋势文章中的单行参考表。
   - 蓝天文本块以“阜阳地区鸡蛋报价 / 精品蛋报价 / 大码蛋系列”为起点，以“小码蛋系列”或下一地区为终点。
   - 九江同时保留褐壳、粉壳主表，并在定位阶段写入明确的 `product_name`，禁止混入同一列。

### 九账号标准转换

| 公众号 | 转换规则 |
| --- | --- |
| 家美鲜鸡蛋 佳美鲜 | 精确码数直接映射；“32 以下顺减-6”等规则向低码展开 |
| 河北馆陶鸡蛋报价 | 净重区间内每个整数码使用同一今日价 |
| 河南金咕咕蛋品 | 重量区间展开；边界同时命中两个区间时取较高价 |
| 贵阳鸡蛋价格 | “xx 斤以上”按下一个更高阈值前的整数码展开；低价、高价分开映射 |
| 蓝天禽蛋联盟 | 只取阜阳地区精品蛋大码系列；重量区间内同价 |
| 湖南三尖农牧公司 | 精确净重直接映射今日价 |
| 成都鸡蛋价格 | 净重区间内同价；参考价格为区间时取高价；相邻规格区间边界重复时取较高价，且保留为原始观测值 |
| 河北辛集城方蛋品 | 精确重量直接映射今日价 |
| 江西九江褐壳蛋 | 褐壳、粉壳分别成列；“褐壳蛋/红壳蛋/红蛋”统一归入褐壳蛋列；精确重量直接映射今日价；两列独立推算 |

### 最小临近区间差值推算

1. 先完成所有原始价格映射和重量区间展开。
2. 只对仍为空的码数进行推算，不覆盖 `observed` 单元格。
3. 将同一列已知码数按从低到高排序，相邻两点形成候选区间：

```text
每码差值 = (高码价格 - 低码价格) / (高码 - 低码)
```

4. 优先选择与缺失码数距离最近的候选区间。
5. 若多个候选区间距离相同，选择绝对每码差值最小的区间；仍相同时选低码起点更小的区间。
6. 区间内缺失做线性插值，已知范围之外使用最近一侧区间向外推算。
7. 不足两个有效观测点时保留为空。
8. 贵阳低价、高价完全独立推算。

## 测试与回归方案

### 单元测试

1. 报价块定位
   - 家美鲜多表时只选通货装车主表。
   - 贵阳只选“规格/毛重/含包装价”表。
   - 蓝天只选阜阳精品蛋大码文本块。
   - 九江排除粉壳表。
   - 广告文章返回空。
2. OCR
   - 非目标账号不调用 OCR。
   - 金咕咕和湖南三尖 OCR 行可转为标准表格。
   - OCR 失败只返回质量备注，不让文章分析任务整体失败。
3. 标准转换
   - 覆盖九账号的精确码、重量区间、阈值区间和边界冲突规则。
   - 只贵阳产生低价、高价两列。
   - 成都价格区间取高价。
4. 缺价推算
   - 内部缺口插值。
   - 两端向外推算。
   - 同距离时取绝对每码差值最小区间。
   - 原始价不被覆盖。
   - 单个观测点不推算。

### 集成与回归

1. 使用九账号脱敏 DOM/OCR 样本执行文章分析，核对明细数和矩阵列。
2. 运行蛋价抽取、transient extractor、报价矩阵、安全查询、结果服务和 Web 页面定向测试。
3. 运行全量 `pytest -q`。
4. 真实文章只读验收时，确认七个文本账号 OCR 调用次数为 0，两个图片账号才有 OCR 调用。

## 风险与分阶段落地建议

### 风险

1. 公众号 DOM 标题或表格顺序变化。
   - 应对：使用多锺点评分，不依赖固定表序号。
2. OCR 将小数点、负号或数字识别错误。
   - 应对：账号专属行格式、重量范围、价格合理区间和最小有效行数联合校验；校验不通过则不落价格。
3. 文本表格中同时存在昨日价、今日价和涨跌。
   - 应对：专属定位后只映射今日价列。
4. 推算值被误解为原始报价。
   - 应对：保留 `source` 和计算说明，页面显式标注“推算”。
5. Windows OCR 在非 Windows 环境不可用。
   - 应对：软失败并保留 `image_ocr_unavailable`备注，七个文本账号不受影响。

### 分阶段落地

1. 阶段 A：报价块定位、七账号文本解析和定向测试。
2. 阶段 B：只对金咕咕、湖南三尖接入 Windows OCR 兜底。
3. 阶段 C：九账号标准转换和最小临近区间差值推算。
4. 阶段 D：真实文章只读验收和全量回归。

## 设计边界确认

1. 文章内先定位目标报价块，再解析价格。
2. 七个账号文本优先，金咕咕和湖南三尖才 OCR。
3. 输出固定为 30–50 码，只贵阳拆低价、高价。
4. 缺价按最小临近区间差值推算，原始价永不被覆盖。
5. 江西九江只保留褐壳主口径。
6. OCR 图片、图片 URL 和整图 OCR 原文不落库、不输出。
7. 无数据库表结构变更，不影响 group 链路。
