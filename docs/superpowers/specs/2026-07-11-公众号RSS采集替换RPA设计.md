# WeInsight 公众号 RSS 采集替换 RPA 设计

## 1. 背景与现状

WeInsight 当前同时采集微信群消息和公众号文章。微信群与公众号入口均依赖 Windows 交互式桌面中的微信 RPA，并通过 `wechat_ui_lock` 串行操作微信 UI。公众号链路已将“微信内取文章链接”和“浏览器解析、清洗、分析、报价抽取、日报”分离，文章链接写入 `wechat_article_raw` 后，下游不再依赖微信 UI。

当前公众号 RPA 容易受微信版本、窗口结构、搜索入口、焦点、登录状态和 UI 时序影响，并与微信群采集争用 UI 锁。为降低故障率，公众号入口改为消费自建 WeRSS 输出的标准 RSS Feed；RPA 仅保留微信群采集。

WeRSS 部署在当前 Windows 采集机的 Docker Desktop 中，复用现有 MySQL 实例，但使用独立数据库和独立账号。管理员首期分别在 WeRSS 和 WeInsight 中维护公众号及 Feed 地址，不做自动名单同步。

## 2. 目标与非目标

### 2.1 目标

1. 以通用 RSS 适配层完全替换公众号 RPA 入口。
2. 公众号采集不再依赖微信进程、Windows 交互桌面或 `wechat_ui_lock`。
3. 复用现有文章解析、清洗、分析、报价抽取和日报链路。
4. WeInsight 仅依赖标准 RSS/Atom 和文章原始 URL，不绑定 WeRSS 内部数据库或私有 API。
5. WeRSS 故障与微信群 RPA 故障相互隔离。
6. 保持现有最多 20 个公众号的小规模运行边界。

### 2.2 非目标

1. 首期不从 WeInsight 创建、编辑或删除 WeRSS 中的订阅。
2. 不直接读取 WeRSS 数据库表。
3. 不保留公众号 RPA 手动命令、调度入口或应急回退链路。
4. 不迁移或重写现有文章后处理和日报业务规则。
5. 不把 WeRSS 暴露到公网，也不新增第二台运维主机。
6. 不承诺抓取公众号全部历史文章；首期关注新发布文章的及时、稳定采集。

## 3. 方案比较与决策

### 3.1 方案一：通用 RSS 适配层直接替换公众号 RPA

新增独立的 Feed 客户端、条目映射器和 RSS 文章采集服务。Feed 条目转换成现有 raw 输入后，继续使用既有下游。该方案隔离清晰、替换成本可控，未来更换 WeRSS 时无需重写业务链路。

### 3.2 方案二：在现有 ArticleCollectService 内直接改成 WeRSS 客户端

初始改动较集中，但会在现有 RPA 服务中残留 UI 锁、中断恢复和路由状态语义，并使业务服务绑定 WeRSS，不采用。

### 3.3 方案三：直接读取 WeRSS MySQL 表

读取延迟低，但依赖 WeRSS 私有数据模型，升级可能直接破坏 WeInsight，不采用。

### 3.4 决策

采用方案一。WeRSS 是可替换的上游 Feed 提供者；标准 RSS/Atom 是 WeInsight 与上游之间的唯一数据契约。

## 4. 总体架构与影响范围

数据流如下：

1. WeRSS 在 Docker Desktop 中获取授权公众号文章并生成标准 Feed。
2. WeInsight 采集 Worker 按配置拉取 Feed。
3. RSS 适配层校验并映射条目，写入 `wechat_article_raw`。
4. 成功新增 raw 后创建现有文章解析任务。
5. 既有解析、清洗、分析、报价抽取和日报链路继续运行。
6. 微信群继续由现有 RPA 采集，独占其 UI 操作边界。

主要影响范围：

- 后端：公众号采集接口、调度判定、Worker 运行状态和健康判断。
- 数据模型：公众号 Feed 配置、HTTP 缓存元数据和 RSS 运行指标。
- 前端：公众号名单、任务详情、运行详情和健康页。
- 部署：Docker Desktop 中新增 WeRSS；MySQL 新增独立数据库及账号。
- 运维：新增 WeRSS 健康、授权失效、空 Feed 和文章时效监控。
- 删除项：公众号 RPA 客户端、路由探测、状态机、进度恢复、截图、CLI、配置和专属测试。

## 5. 数据模型与权限模型

### 5.1 公众号配置

继续使用 `wechat_public_account_config` 作为 WeInsight 公众号名单的事实来源。增加或明确以下字段：

- `feed_url`：标准 RSS/Atom 地址，必填且唯一。
- `source_type`：首期固定为 `rss`，为未来替换 Feed 提供者保留边界。
- `poll_interval_minutes`：默认 10 分钟，最小 10 分钟。
- `request_timeout_seconds`：默认 30 秒。
- `last_feed_etag`：最近一次成功响应的 ETag。
- `last_feed_modified`：最近一次成功响应的 Last-Modified。
- `last_success_collect_time`：最近成功拉取时间。
- `last_error_code`：最近结构化错误码。

现有 `account_name` 继续作为业务键。Feed 地址不得包含管理员密码、登录 Cookie 或 MySQL 凭据。

### 5.2 文章结果

保留 `wechat_article_raw` 及其下游结果表，避免大范围迁移。RSS 条目映射如下：

- `account_name`：使用 WeInsight 配置值，不盲信 Feed 标题。
- `title`：RSS `title`，缺失时拒绝该条目。
- `article_url`：优先使用条目 `link`，规范化并通过安全校验后写入。
- `publish_time`：优先 `published`，其次 `updated`；两者均不可用时拒绝该条目。
- `publish_date`：按 `Asia/Shanghai` 从 `publish_time` 计算。
- `author`、`digest`：有则写入，无则为空。
- `article_hash`：继续复用现有账号、标题、发布时间和规范化 URL 算法。
- `collect_batch_id`：每次 Feed 拉取生成一个批次号。
- `collect_time`：WeInsight 成功处理条目的时间。

去重同时使用 `article_hash` 唯一约束和账号内规范化 URL 查询，防止 Feed 标题或时间轻微变化造成重复业务文章。

### 5.3 RPA 专属数据清理

RSS 链路验收通过后，通过显式数据库迁移删除仅供公众号 RPA 使用的路由缓存、采集进度及专属字段。删除前必须确认这些对象未被微信群链路、结果查询或审计使用。文章 raw、clean、analysis、报价、日报和通用采集日志不删除。

### 5.4 权限

- WeRSS MySQL 账号仅访问独立 `werss` 数据库。
- WeInsight 各账号不得访问 `werss` 数据库。
- WeRSS 不访问 WeInsight 数据库。
- WeInsight 只通过 HTTP 获取 Feed，不调用 WeRSS 私有管理 API。
- WeRSS 管理界面只绑定采集机本地或受控局域网地址，不开放公网。

## 6. 后端实现方案

### 6.1 RssFeedClient

职责为执行 HTTP 条件请求并解析 RSS/Atom，不负责业务入库。必须支持：

- `If-None-Match` 与 `If-Modified-Since`。
- 连接与读取超时。
- 响应体大小上限和有限重定向。
- RSS 2.0 与 Atom。
- XML/HTML 实体及 CDATA 的安全解析。
- 结构化返回 HTTP 状态、Feed 元数据和条目列表。

禁止自动请求非 HTTP/HTTPS 协议、环回地址、链路本地地址、私网地址和云元数据地址。每次重定向后重新执行目标地址校验，避免 SSRF 绕过。压缩响应解压后仍受大小上限约束。

### 6.2 RssArticleMapper

职责为把一个 Feed 条目映射成 `RawArticleRecord` 候选，不访问数据库。它负责字段必填校验、URL 规范化、微信文章域名白名单、发布时间解析、时区转换和摘要清理。

条目级错误不得终止整个 Feed；错误以结构化代码计数并记录，不记录正文、Cookie 或敏感 URL 查询参数。

### 6.3 RssArticleCollectService

职责为生成批次、调用客户端和映射器、幂等写入 raw、创建后处理任务及写入采集日志。

首次接入默认只导入最近 24 小时且最多 30 篇。后续处理 Feed 中全部可见条目并依赖幂等去重。Feed 合法但没有条目时，本次运行记为成功且采集数为零；HTTP 错误、超时、响应超限或 Feed 格式非法记为失败。

采集日志至少记录 Feed 条目数、新增数、重复数、无效数、HTTP 状态、耗时和结构化错误码，不输出文章正文或认证信息。

### 6.4 Worker 与调度

- 公众号 RSS 运行不检查微信健康，不获取 `wechat_ui_lock`。
- 微信健康异常只暂停微信群运行。
- WeRSS 异常只影响对应公众号运行，已有文章后处理继续执行。
- RSS 网络请求可与微信群 RPA 同时执行。
- 首期沿用现有采集 Worker，RSS 最大并发数默认为 4，且可配置降低。
- 停止任务在单个 HTTP 请求结束或单个 Feed 保存完成的安全检查点生效。
- Worker 重启后沿用漏跑合并语义，仅立即补拉一次，不逐周期追赶。

## 7. 前端影响

### 7.1 公众号名单

新增 Feed 地址、拉取周期、超时、最近成功时间和最近错误。新增或编辑时，前端提供格式提示，后端执行最终的 URL 安全校验和 Feed 连通性校验。

首期管理员先在 WeRSS 中添加公众号，再把生成的 Feed 地址录入 WeInsight。WeInsight 不自动同步 WeRSS 名单。

### 7.2 任务与运行

公众号任务继续支持多目标、有效期、每日窗口、频率、停止和软删除。最小拉取周期保持 10 分钟。

运行详情展示 HTTP 状态、Feed 条目数、新增数、重复数、无效数、耗时和最近错误。移除公众号 UI 锁等待、路由探测、窗口截图和中断阶段信息。

### 7.3 健康页

微信进程、登录和 RPA 健康明确标注为“仅影响微信群”。增加 WeRSS HTTP 健康和各 Feed 新鲜度摘要，但 WeRSS 整体异常不得把微信群 Worker 标记为不可运行。

## 8. WeRSS 部署方案

1. 在当前 Windows 采集机的 Docker Desktop 中部署 WeRSS。
2. 使用经过 POC 验证的固定镜像版本或摘要，禁止生产配置直接使用 `latest`。
3. 连接现有 MySQL 实例中的独立 `werss` 数据库，使用独立最小权限账号。
4. 凭据通过本机受限环境文件或环境变量注入，不提交仓库。
5. 配置容器自动重启、健康检查、日志轮转和必要的持久化目录。
6. 限制监听地址和防火墙范围，不暴露公网。
7. WeRSS 数据库与 WeInsight 数据库分别备份和恢复；两者没有跨库事务。

同机部署降低主机维护成本，但 Docker Desktop 重启会造成短期 Feed 不可用。恢复后依赖 Feed 可见窗口、补拉一次和幂等去重补齐，而不是补跑每个遗漏调度周期。

## 9. 错误处理与可观测性

结构化错误至少包括：

- `feed_timeout`
- `feed_http_error`
- `feed_too_large`
- `feed_invalid_format`
- `feed_url_blocked`
- `feed_redirect_blocked`
- `feed_item_missing_title`
- `feed_item_missing_publish_time`
- `feed_item_url_invalid`
- `feed_stale`

监控重点包括：最近成功拉取时间、Feed 最新文章发布时间、连续空 Feed 次数、连续失败次数、条目新增率和后处理积压。连续空 Feed 不直接等同故障，但应结合最新文章时间形成告警，帮助识别 WeRSS 授权失效或风控。

日志不得包含 WeRSS 登录信息、数据库密码、完整 Cookie、文章正文或未经脱敏的异常响应体。

## 10. 测试与回归方案

### 10.1 单元测试

- RSS、Atom、CDATA、实体和多种时间格式。
- 缺失标题、链接、发布时间及非法 XML。
- 重复条目、标题或时间变化、URL 规范化。
- 超时、响应超限、异常压缩和重定向循环。
- 私网、环回、链路本地、元数据地址及重定向后的 SSRF 拦截。

### 10.2 Repository 与服务测试

- 批量幂等入库及账号内 URL 去重。
- 新增 raw 后只创建一次解析任务。
- 单个坏条目不影响同 Feed 的其他条目。
- 单个坏 Feed 不影响其他公众号。
- 首次导入 24 小时及 30 篇上限。
- ETag、Last-Modified 和 `304 Not Modified`。

### 10.3 Worker 与隔离测试

- RSS 与微信群 RPA 可同时运行。
- RSS 运行不获取 `wechat_ui_lock`。
- 微信退出不阻塞 RSS。
- WeRSS 停止不阻塞微信群及已有文章后处理。
- 停止、重启和漏跑合并符合安全检查点语义。

### 10.4 前端与回归测试

- Feed 地址校验、任务配置、运行指标和错误展示。
- 公众号 RPA 页面元素和入口全部移除。
- 现有文章解析、清洗、报价抽取、分析、日报和安全输出测试全部通过。
- 微信群 RPA、游标、清洗、分析和日报回归全部通过。

## 11. 分阶段落地与回滚

### 11.1 阶段一：基础设施与单公众号 POC

1. 备份 WeInsight 数据库。
2. 创建并备份独立 `werss` 数据库。
3. 部署固定版本 WeRSS，添加 1 个已授权公众号。
4. 新增 RSS 配置字段和通用适配层，但不启用正式公众号任务。
5. 对照公众号实际发布记录，验证最近 24 小时文章完整率、发布时间和链接。

### 11.2 阶段二：切换与观察

1. 停止所有公众号 RPA 任务并清理未执行运行实例。
2. 启用单公众号 RSS 调度，连续观察至少 24 小时。
3. 通过后扩至 3 个公众号，再逐步扩至不超过 20 个。
4. 观察期内保留数据库备份和切换前代码版本，但不允许继续运行公众号 RPA。

### 11.3 阶段三：最终删除

1. 删除公众号 RPA 客户端、状态机、路由缓存、进度恢复、CLI、配置和专属测试。
2. 删除管理后台中的公众号 RPA 展示与控制。
3. 执行数据库迁移，删除仅供公众号 RPA 使用且已确认无引用的表和字段。
4. 更新 README、部署手册、监控说明和回滚预案。

最终删除后，回滚只能暂停 RSS 新链路并恢复切换前代码与数据库备份，不提供公众号旧 RPA 的运行时应急入口。

## 12. 验收标准

1. 公众号新文章在 WeRSS Feed 可见后 15 分钟内进入 `wechat_article_raw`。
2. 连续 24 小时无重复业务文章。
3. RSS 运行期间不获取 `wechat_ui_lock`。
4. 微信客户端退出时，公众号 RSS 仍正常采集。
5. WeRSS 停止时，微信群采集和已有文章后处理仍正常。
6. 现有文章解析、报价抽取、分析和日报结果不退化。
7. 单公众号连续 24 小时满足以上标准后，才允许执行公众号 RPA 的最终代码和表结构删除。

## 13. 风险与缓解

- **非官方采集风险**：WeRSS 仍可能遇到授权失效、风控或微信侧变化。通过固定版本、授权状态监控、连续空 Feed 告警和分阶段扩容缓解。
- **上游维护风险**：WeRSS 是可替换提供者。WeInsight 只消费标准 Feed，避免绑定私有 API 和表结构。
- **Feed 不完整风险**：以实际发布记录执行 POC 对账，并监控最新文章时间，而不是只监控 HTTP 成功。
- **同机故障风险**：Docker Desktop 重启会影响 WeRSS，但不会影响微信群 RPA；通过自动重启、健康检查和独立备份恢复。
- **删除后无 RPA 回退**：最终删除必须晚于 24 小时验收，切换前保留可恢复的代码版本和数据库备份。
- **SSRF 与资源耗尽风险**：对初始 URL 和每次重定向执行地址校验，并限制超时、响应大小、解压大小和并发数。
