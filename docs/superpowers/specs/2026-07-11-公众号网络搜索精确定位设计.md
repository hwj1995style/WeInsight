# 公众号直接搜索精确定位设计

## 背景与现状

管理员确认的真实微信路径是：顶部搜索框直接输入公众号名称，下拉框顶部出现“公众号”分类结果；不需要、也不存在点击“搜索网络结果”后再定位公众号的步骤。

此前设计把“搜索网络结果”误当成必要入口，真实 UI 验证表明该入口的 UIA 调用只会关闭下拉层，不能进入可选择的网络结果页。本设计以直接搜索的公众号分类结果为唯一 RPA 定位路径。

## 目标与非目标

目标：在现有 WxautoArticleRpaClient 中可靠聚焦顶部搜索框，输入 account_name，只点击公众号分类中名称完全等于配置值的结果，并在窗口精确校验成功后继续既有文章采集链路。

非目标：

1. 不点击或依赖“搜索网络结果”。
2. 不使用模糊匹配、首项回车、force=True 或聊天记录中的同名条目。
3. 不接入 RSS、不修改数据库、前端、调度频率或日报链路。
4. 不在 probe 通过前恢复真实采集、扩容或次日报表观察。

## 实现方案

1. 保留 ChatWith 和目标公众号窗口就绪等待作为第一层。
2. 第一层未就绪时，找到主窗口顶部搜索编辑框。
3. 使用 UIA set_focus 后确认 has_keyboard_focus 为真；未确认焦点即安全失败，绝不发送搜索键盘输入。
4. 输入 account_name 后，只扫描搜索下拉框顶部“公众号”分类区域。
5. 只接受标准化后名称与 account_name 完全相等的结果；同名聊天记录、包含名称、相近名称全部拒绝。
6. 点击精确公众号结果后，继续使用既有目标窗口精确校验；失败时不进入历史文章点击或取链。

## 影响范围

- app/rpa/wxauto_client.py：删除错误的网络搜索分支；新增安全搜索框聚焦和公众号分类精确结果选择。
- tests/test_wxauto_client.py：删除网络搜索测试；增加搜索焦点未确认拒绝、公众号精确命中、聊天记录同名拒绝和窗口未就绪测试。
- docs/operations/微信采集管理后台受控POC执行记录.md：更正为直接搜索路径，继续记录 No-Go 证据。

## 测试与验收

1. 单元测试先行覆盖焦点确认、分类筛选、精确匹配与拒绝规则。
2. 只读 article-rpa-probe 验证自动定位、历史页点击和 UIA 链接兜底；不创建任务。
3. 仅当 probe 返回 status=ok、account_found=1 且 link_count 大于等于 1 时，管理员才可确认重跑单目标 POC。

## 风险与回滚

微信搜索下拉框的控件层级可能变化。所有输入前必须确认焦点，所有选择前必须确认公众号分类和精确名称；任一条件缺失即失败退出。若 probe 未通过，保持 No-Go，不恢复自动采集；错误实现的网络搜索分支完全删除。
