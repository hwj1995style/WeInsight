# 公众号网络搜索精确定位设计

## 背景与现状

首次真实 POC 表明，文章详情页一旦由人工打开，微信内置浏览器窗口、当天文章条件和 UIA 链接兜底均可识别；当前失败点在于 RPA 不能稳定从本地会话定位“一箱蛋”。人工可行路径是微信顶部“搜索网络结果”。

现有 `WxautoArticleRpaClient` 只使用 `ChatWith()` 和主窗口本地搜索结果；两者均不能替代网络搜索入口。`force=True` 的 wxauto4 切换已验证无法定位目标，因此不纳入方案。

## 目标与非目标

目标：在现有 RPA 中增加受控的网络搜索定位路径，使已配置的公众号仅在搜索结果名称与配置值完全一致时才可打开；成功后继续复用既有历史文章、UIA URL 兜底、解析、日报链路。

非目标：

1. 不接入 RSS，不启动 RSS POC。
2. 不使用模糊匹配、首项强制回车或相近名称兜底。
3. 不输出、持久化或在 Web 展示搜索结果内容、文章链接、正文或故障截图。
4. 不自动恢复真实 POC、双链路观察或扩容；网络搜索仅先通过只读 `article-rpa-probe` 验证。

## 方案比较与决策

| 方案 | 结论 | 原因 |
| --- | --- | --- |
| wxauto4 `force=True` | 不采用 | 已在本机验证无法定位目标，且可能选择非目标会话。 |
| 网络搜索精确匹配 | 采用 | 与人工实际路径一致；可在点击前按完整名称校验。 |
| RSS 优先 | 延后 | 作为独立 POC 评估，不与当前 RPA 故障修复混合。 |

## RPA 实现方案

### 定位流程

1. 保留当前 `ChatWith()`、公众号窗口就绪等待和本地搜索作为第一层。
2. 第一层失败后，在微信主窗口顶部搜索框输入配置的 `account_name`。
3. 明确点击“搜索网络结果”入口，等待网络搜索结果面板就绪。
4. 仅寻找标准化后与 `account_name` 完全相等的公众号结果；同名以外的部分匹配、包含匹配或无名称结果全部拒绝。
5. 点击精确结果后，轮询目标公众号窗口是否可见；仅窗口精确匹配成功才设置 `current_account_name`。
6. 定位失败返回现有安全错误，不进入历史文章点击、取链接或数据写入。

### 错误与安全

- 分别记录安全错误码：网络入口缺失、网络结果超时、未找到精确结果、点击后窗口未就绪。
- 错误摘要仅保留错误码和阶段，不包含结果名称、文章 URL、正文或截图内容。
- UI 操作全程复用 `wechat_ui` 锁；不得在未完成精确窗口校验时调用采集服务。

## 影响范围

- `app/rpa/wxauto_client.py`：新增网络搜索入口、精确结果选择和条件等待辅助方法。
- `tests/test_wxauto_client.py`：新增网络入口、精确匹配、相近名拒绝和窗口未就绪测试。
- `docs/operations/微信采集管理后台受控POC执行记录.md`：记录只读 probe 结果；仍保持 No-Go，直至重新获得真实 POC 通过证据。

不修改数据库模型、前端页面、任务频率、日报与 RSS 相关实现。

## 测试与验收

1. 单元测试先行：
   - 本地路径成功时不触发网络搜索。
   - 点击网络入口后仅精确名称结果可被选择。
   - 相近名、包含名、多结果无精确名均拒绝。
   - 网络结果超时或点击后目标窗口不可见时安全失败。
2. 真实验证仅运行 `article-rpa-probe --account-name "一箱蛋"`，不创建任务、不写采集结果。
3. probe 成功且 UIA 链接兜底返回至少 1 条后，再由管理员明确确认是否重跑单目标 POC。

## 风险与回滚

微信网络搜索 UI 的类名、文本或面板层级可能变化。所有选择器均以条件等待和精确名称校验保护；任何不确定状态均失败退出。若真实 probe 失败，关闭网络搜索开关或回退到当前本地会话路径，不恢复自动采集和扩容。
