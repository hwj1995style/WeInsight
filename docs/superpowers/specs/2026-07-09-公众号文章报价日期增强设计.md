# 公众号文章报价日期增强设计

## 背景与现状

首版禽蛋报价抽取已经能在分析阶段重新打开公众号文章链接，读取正文和 DOM 表格，并落库到 `wechat_article_egg_price_item`。现有模型只有 `publish_time` 和 `publish_date`，其中 `publish_date` 表示文章发布日期。

福建闽融鸡蛋报价平台存在“文章发布时间是前一天晚上，标题发布的是当天报价”的真实场景，例如微信发布时间为 `2026-07-08 21:59`，标题为 `2026年07月09日｜福建闽融平台`。这说明文章发布时间、报价业务日期和采集时间必须分开建模。

## 目标与非目标

目标：

1. 保留微信真实文章发布时间 `publish_time` / `publish_date`，用于溯源和审计。
2. 保留采集时间 `collect_time`，用于采集链路审计，不参与报价业务归属。
3. 新增报价业务日期 `quote_date`，用于蛋价明细、日报任务和后续规格转换的业务日期。
4. 从标题优先抽取报价业务日期；标题没有明确日期时，再从正文报价上下文抽取；仍没有时回退到文章发布日期。
5. 在结构化结果中记录 `quote_date_source` 和 `quote_date_confidence`，便于识别 fallback 数据。
6. 兼容已有数据：历史行缺少 `quote_date` 时可以用 `publish_date` 回填或通过 `COALESCE(quote_date, publish_date)` 查询。

非目标：

1. 不改变首版“不做 OCR”的边界。
2. 不做规格统一换算。
3. 不改微信群链路和 `wechat_group_*` 表。
4. 不在 CLI、日志或日报中输出文章链接、正文或完整报价原文。

## 影响范围

后端：

1. `app/pipelines/article_parse_service.py`：补充微信页面可见发布时间解析，优先读取 `#publish_time`。
2. `app/domain/article_analysis.py`：在分析结果上新增 `quote_date`、`quote_date_source`、`quote_date_confidence`、`collect_time`。
3. `app/domain/article_egg_price.py`：每条报价明细新增 `quote_date` 及来源置信度字段。
4. `app/storage/article_analysis_repo.py`：查询待分析文章时关联 raw 表取 `collect_time`，写 analysis 和 egg price item 时落新字段。
5. `app/pipelines/article_analysis_service.py`：日报任务日期改为 `analysis.quote_date`，没有报价日期时回退 `analysis.publish_date`。
6. `app/storage/article_daily_report_repo.py`：日报统计日期用 `COALESCE(quote_date, publish_date)` 兼容新旧数据。

数据库：

1. `wechat_article_analysis` 新增 `collect_time`、`quote_date`、`quote_date_source`、`quote_date_confidence`。
2. `wechat_article_egg_price_item` 新增同样的 `collect_time`、`quote_date`、`quote_date_source`、`quote_date_confidence`。
3. 新增 quote_date 相关索引，不删除旧 `publish_date` 索引。

## 数据模型或权限模型

时间字段语义：

| 字段 | 含义 | 来源 | 是否作为报价业务日 |
| --- | --- | --- | --- |
| `collect_time` | 系统采集到文章链接的时间 | `wechat_article_raw.collect_time` | 否 |
| `publish_time` | 微信文章真实发布时间 | 微信页面 meta 或可见 `#publish_time` | 否 |
| `publish_date` | `publish_time` 的日期 | `publish_time.date()` | 否 |
| `quote_date` | 报价业务归属日 | 标题/正文/fallback | 是 |

`quote_date_source` 枚举：

1. `title`：标题包含完整年月日。
2. `body`：正文报价上下文包含完整年月日。
3. `publish_date_fallback`：没有明确报价日期，回退文章发布日期。
4. `collect_date_fallback`：没有文章发布时间，回退采集日期。
5. `unknown`：以上均不可用。

`quote_date_confidence` 建议值：

1. `1.0`：标题命中完整日期。
2. `0.9`：正文报价上下文命中完整日期。
3. `0.6`：回退文章发布日期。
4. `0.4`：回退采集日期。
5. `0.0`：无法判断。

## 后端实现方案

1. 新增轻量日期抽取函数，输入标题、正文、文章发布时间、采集时间，输出 `QuoteDateInfo`。
2. 日期正则支持 `2026年07月09日`、`2026年7月9日`、`2026-07-09`、`2026/07/09`。
3. 标题中有多个日期时取第一个完整年月日；正文仅在包含 `报价`、`价格`、`参考`、`行情` 等关键词附近的文本中识别。
4. `analyze_clean_article` 先计算 `quote_date_info`，再传给 egg price extractor，确保 analysis 和 item 使用同一个业务日期。
5. 每条 `EggPriceItem` 保存 `quote_date`、`quote_date_source`、`quote_date_confidence`，同时保留 `publish_time/publish_date/collect_time`。
6. 日报任务创建改为 `analysis.quote_date or analysis.publish_date`。
7. 日报统计查询改为 `COALESCE(quote_date, publish_date) = :report_date`。

## 前端影响

当前没有前端页面变更。后续如果展示报价明细，应展示三类时间：

1. 报价日期：`quote_date`
2. 微信发布时间：`publish_time`
3. 采集时间：`collect_time`

## 测试与回归方案

1. 单元测试覆盖福建闽融场景：发布时间 `2026-07-08 21:59`，标题 `2026年07月09日｜福建闽融平台`，期望 `quote_date=2026-07-09` 且 `publish_date=2026-07-08`。
2. 解析测试覆盖微信可见 `#publish_time`。
3. SQL schema 测试覆盖新字段、索引和迁移隔离。
4. Repo 测试覆盖 analysis 与 egg price item 新字段落库。
5. Service 测试覆盖日报任务按 `quote_date` 创建。
6. 真实 MySQL 验证：在 `config/config.dev.yaml` 的 `3307` 端口执行迁移，并查询字段存在性。

## 风险与分阶段落地建议

风险：

1. 历史数据没有 `quote_date`，需要查询层兼容。
2. 标题日期可能是文章排版日期而非报价日期，但对当前报价公众号样本是最高置信来源。
3. 正文多日期场景可能误判，首版仅做保守抽取并记录来源置信度。

分阶段落地：

1. 第一阶段新增字段、抽取规则、落库和日报日期口径。
2. 第二阶段对历史已分析文章补跑 analyze，重建 `quote_date` 和明细。
3. 第三阶段在规格转换表中统一使用 `quote_date` 作为业务分区键。
