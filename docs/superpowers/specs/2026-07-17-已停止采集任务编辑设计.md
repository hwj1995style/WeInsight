# 已停止采集任务编辑设计

## 背景与现状

采集任务目前支持创建、停止、重新启动和软删除。重新启动会沿用原任务名称、采集目标和调度计划，因此任务有效期已经结束或配置需要调整时，管理员只能重新创建任务。

任务运行历史通过 `wechat_collection_job_target_run.job_target_id` 引用任务目标。已执行过的目标不能物理删除，否则会破坏历史运行记录；因此编辑采集目标不能采用“删除全部目标后重建”的方式。

## 目标与非目标

### 目标

- 仅允许编辑状态为 `stopped` 的人工任务。
- 支持修改任务名称、采集目标、整体有效期、每日窗口和采集频率。
- 编辑后任务仍保持 `stopped`，由管理员确认后再启动。
- 保留历史目标和运行记录，同时确保新运行只读取当前启用目标。
- 使用 CSRF、严格表单字段、乐观锁版本和审计事件保护写操作。

### 非目标

- 不允许编辑 `scheduled`、`active`、`stop_requested`、`completed` 或 `deleted` 任务。
- 不允许编辑系统管理任务 `managed_key=article_global`。
- 不允许修改任务采集链路 `pipeline_type`。
- 保存编辑不会自动启动任务，也不会立即创建运行实例。
- 本次不提供运行中热更新、批量编辑或历史运行配置回写。

## 影响范围

- 数据模型：任务目标增加当前启用标记。
- 后端：新增任务更新命令、服务方法、仓储事务和审计事件。
- 前端：新增编辑页及列表、详情页编辑入口。
- 调度：认领任务时只读取当前启用目标。
- 权限：Web 数据库角色增加任务目标表 `UPDATE` 权限。
- 测试：覆盖状态、权限、冲突、历史目标保留、表单安全和真实浏览器流程。

## 数据模型与权限模型

### 任务目标启用标记

为 `wechat_collection_job_target` 增加：

```sql
is_active TINYINT(1) NOT NULL DEFAULT 1 COMMENT '是否为任务当前启用目标'
```

并增加索引 `idx_job_target_active (job_id, is_active)`。

- 现有目标迁移后默认全部启用，保持当前行为。
- 编辑时先将该任务所有目标标记为未启用，再对本次选择目标执行插入或更新并标记为启用。
- 已有目标行复用原主键，更新名称、优先级和配置快照；历史运行引用保持有效。
- 被移除目标保留为未启用状态，不再进入目标计数、重叠校验、来源引用和新运行。
- 历史运行与事件查询仍可通过目标主键读取未启用目标快照。

### 操作权限

- `stopped` 且 `managed_key IS NULL`：允许打开编辑页和提交编辑。
- 其他状态：返回安全的 `409` 提示并刷新当前详情。
- 系统管理任务：返回 `403`，页面不渲染编辑入口。
- Web 数据库角色对 `wechat_collection_job_target` 增加 `UPDATE`；不增加物理删除权限。

## 后端实现方案

### 服务层

新增 `UpdateCollectionJobCommand` 和 `CollectionJobService.update_job(...)`：

1. 校验任务 ID、版本、操作者和带时区当前时间。
2. 读取任务并拒绝系统任务、非停止状态和链路篡改。
3. 复用创建任务的名称、目标、频率和调度校验。
4. 确认更新后的计划从当前时间起至少存在一个未来运行网格；过期计划不允许保存。
5. 将规范化命令交给仓储层原子更新。

### 仓储层

在单个数据库事务内：

1. `FOR UPDATE` 锁定任务并再次校验 `managed_key`、`status`、`version` 和 `pipeline_type`。
2. 按稳定顺序锁定所选来源，校验来源存在、启用且链路一致，并生成新快照。
3. 查询其他 `scheduled/active/stop_requested` 任务的启用目标，执行调度重叠校验。
4. 更新任务名称和调度字段，保持 `status=stopped`、`next_run_at=NULL`，版本加一。
5. 将旧目标标记为未启用，并插入或重新启用当前目标。
6. 写入 `job_updated` 管理员审计事件。

任一步骤失败则整笔事务回滚。

### 查询兼容调整

以下“当前配置”查询增加 `is_active=1`：

- 任务列表目标数。
- 任务详情目标名称和 ID。
- 任务重叠校验。
- 调度认领的目标总数和目标快照。
- 来源配置的任务引用检查。
- 系统公众号任务当前目标汇总。

历史运行、历史事件和目标运行详情查询不增加该条件。

## 前端影响

- 新增 `GET /jobs/{job_id}/edit` 和 `POST /jobs/{job_id}/edit`。
- 编辑页复用创建表单布局，标题、提交地址和按钮文案切换为编辑模式。
- 表单预填任务名称、当前启用目标、有效期、每日窗口和频率。
- 表单提交必须包含 CSRF、版本和固定链路；拒绝未知、重复或缺失字段。
- 已停止任务的列表操作区显示“编辑”和“启动”；详情页显示“编辑任务”“启动任务”“删除任务”。
- 系统任务和非停止状态不显示编辑入口。
- 保存成功后以 `303` 返回任务详情；失败时保留安全的表单回显。

## 测试与回归方案

### 服务与仓储

- 只有停止状态可以更新，系统任务和其他状态被拒绝。
- 更新校验 ID、版本、操作者、调度和未来运行网格。
- 更新目标时保留旧目标行并切换 `is_active`，不执行物理删除。
- 新增、保留和移除目标的快照均正确。
- 来源缺失、停用、链路错误、任务重叠和版本冲突全部回滚。
- 编辑后保持停止状态、清空下次运行时间、版本加一并记录 `job_updated`。

### Web 与安全

- 编辑页只在合法状态渲染，字段预填正确。
- GET 不执行写操作；POST 强制 CSRF 和乐观锁版本。
- 重复字段、未知字段、链路篡改和直接访问非法状态被安全拒绝。
- 列表与详情页入口按状态和系统任务权限显示。
- 保存成功跳回详情，冲突时展示安全中文提示且不泄露内部异常。

### 数据模型与调度回归

- 初始化 SQL 与增量迁移包含 `is_active` 和索引。
- Web 数据库角色具备目标表更新权限但没有删除权限。
- 调度目标总数和目标加载只计算启用目标。
- 历史目标运行仍可读取已停用目标。
- 全量 `pytest` 回归通过。

### 浏览器验收目标

验收流程：`/jobs` → 已停止任务“编辑” → 修改名称、有效期和目标 → 保存后仍为“已停止” → 点击“启动任务” → 恢复合法下次运行时间。

## 风险与分阶段落地建议

- 风险：遗漏某个当前目标查询的 `is_active` 条件会导致已移除目标再次执行。通过仓储 SQL 契约测试和真实调度认领测试控制。
- 风险：迁移未执行时应用查询新字段会失败。部署顺序必须为先迁移、再发布应用。
- 风险：编辑与启动并发可能造成旧版本写入。任务行锁和版本条件保证只有一个操作成功。
- 风险：已停用来源不会出现在编辑候选项中。管理员必须选择至少一个当前启用来源后才能保存。
- 建议一次性发布迁移、权限脚本和应用代码；发布后先编辑未来时间窗口的停止任务进行冒烟验证，再开放常规使用。

## 验收标准

- 管理员可编辑已停止人工任务的名称、启用目标和完整调度配置。
- 编辑不会删除或改写历史运行引用。
- 保存后任务仍停止，启动后按新计划生成下次运行时间。
- 非法状态、系统任务、过期计划、冲突和并发写入均被安全拒绝。
- 自动化测试和真实浏览器验收通过，页面无相关控制台错误。
