# WeRSS 公众号只读状态与增量处理设计

## 1. 背景与现状

WeRSS 已成为公众号来源配置和上游文章更新的唯一管理端，WeInsight 通过 RSS/Atom 获取文章并执行后续处理。当前 WeInsight 管理后台仍提供“新增公众号”、名称与 Feed URL 编辑、本地启停、公众号级采集间隔、优先级和处理策略等入口，造成两个系统都像公众号配置主数据源，容易出现清单、状态和调度策略漂移。

当前固定 WeRSS 镜像提供 `GET /api/v1/wx/mps`，支持 `AK-SK` 专用凭据、分页和启用状态筛选，返回公众号稳定 ID、名称和状态。该接口不提供公众号最近同步时间，因此 WeRSS 启用状态与 WeInsight 数据新鲜度必须作为两个不同维度展示。

公众号旧 RPA 已明确废弃，不得恢复、重新运行或作为回退方案。“一箱蛋”不在采集范围内，必须在服务端同步边界强制排除。

## 2. 目标与非目标

### 2.1 目标

1. 将 WeRSS 确立为公众号新增、删除、启停和上游采集调度的唯一管理端。
2. 将 WeInsight 的公众号页面改为只读状态页面，不再提供公众号级人工配置。
3. WeInsight 通过本机 WeRSS API 和专用 AK/SK 同步必要的来源映射，不直连 WeRSS 数据库。
4. 公众号链路使用一个全局执行周期，默认每 10 分钟运行一次，并允许通过部署配置调整。
5. 每轮仅处理新增或待重试文章，自动完成增量采集、清洗、分析和报价提取。
6. 保留历史文章、任务引用和处理证据；上游停用或删除不得触发历史数据删除。
7. 保证公众号链路不获取微信 UI 锁，不影响微信群采集与处理链路。

### 2.2 非目标

1. 不从 WeInsight 创建、修改或删除 WeRSS 公众号。
2. 不从 WeInsight 创建、修改或启动 WeRSS 消息任务。
3. 不直连或查询 WeRSS 私有数据库表。
4. 不恢复公众号旧 RPA 或其命令、调度、探针和 UI 入口。
5. 不在本次改造中引入公众号级采集间隔、优先级、窗口或处理开关。
6. 不实现后台可视化凭据管理；AK/SK 只通过环境变量注入。

## 3. 方案比较与选择

### 3.1 已选择方案：全局周期自动同步与只读展示

WeInsight 在公众号全局周期的每轮开始时，通过 WeRSS API 读取完整公众号清单，更新本地最小来源映射，然后采集启用来源的 Feed。管理页面只读取 WeInsight 的状态聚合，不触发写入或采集。

该方案使页面职责简单、状态刷新稳定，并确保无人访问页面时来源同步和增量处理仍会继续。

### 3.2 未选择：打开页面时同步

页面会依赖 WeRSS 实时可用性，接口超时可能拖慢管理后台，并让只读访问产生隐式写入，因此不采用。

### 3.3 未选择：人工预览并确认同步

该方案适合人工管理本地来源，但与“公众号配置只在 WeRSS 管理、WeInsight 仅展示”的最终职责不一致，因此不采用。

## 4. 影响范围

- 数据模型：公众号配置增加 WeRSS 稳定 ID和上游状态字段；移除公众号级策略对运行时的影响。
- 后端：增加只读 WeRSS 清单客户端、来源同步服务和全局公众号处理编排。
- 前端：公众号管理页面改为只读状态页面，移除所有写操作和表单入口。
- 配置与部署：增加 WeRSS API 端点、AK/SK 环境变量名和全局执行周期配置。
- 调度：公众号来源同步、Feed 采集、增量清洗、分析和报价提取形成连续的异步状态链路。
- 运维文档：修订“不得读取 WeRSS 管理 API”的旧边界，继续保留禁止跨库访问和禁止写 WeRSS 的边界。
- 验收：本改造改变来源同步与调度行为，当前连续 24 小时观察窗口失效；发布并稳定后必须重新计时。

微信群配置、微信群 RPA、已有结果查询与日报能力不在本次改造范围内。

## 5. 数据模型与状态语义

### 5.1 公众号来源映射

在 `wechat_public_account_config` 增加：

- `werss_source_id VARCHAR(200) NULL`：WeRSS 公众号稳定 ID；建立唯一索引。旧记录迁移期间允许为空，新同步记录必须有值。
- `upstream_status VARCHAR(20) NOT NULL DEFAULT 'unknown'`：`active`、`disabled`、`missing`、`excluded` 或 `unknown`。
- `upstream_last_seen_at DATETIME NULL`：最近一次在完整 WeRSS 清单中确认存在的时间。
- `upstream_missing_at DATETIME NULL`：首次在成功读取的完整清单中确认缺失的时间；恢复后清空。

现有 `account_name` 与 `feed_url` 继续保留，以支持采集快照、历史审计和现有任务引用，但只允许同步服务写入。Feed URL 根据经过校验的 `werss_source_id` 和固定本机端点模板生成，不接受 WeRSS 响应中的任意 URL，也不允许管理员编辑。

现有公众号级 `enabled`、`priority`、`poll_interval_minutes`、`daily_window_start`、`daily_window_end`、`request_timeout_seconds`、`collect_today_only` 等字段先保留以兼容现有代码和历史记录，但不再作为页面可编辑策略。实现阶段采用分阶段兼容，不在同一版本直接删除字段。

### 5.2 状态含义

- 上游正常：`upstream_status=active`。
- 上游停用：WeRSS 清单中存在且 `status=0`。
- 上游缺失：只有在成功读取全部分页且通过完整性校验后，已绑定来源未出现时才标记。
- 已排除：来源命中服务端排除规则，不参与任何新采集或处理。
- 采集异常：最近一轮 Feed 获取失败。
- 陈旧：依据 WeInsight 最近成功采集时间、最近 Feed 文章时间和全局周期计算；不使用 WeRSS 清单接口无法提供的同步时间。

WeRSS API 请求失败、鉴权失败、分页不完整或响应非法时，不改变任何来源的上游状态，避免把系统故障误判为全部删除。

### 5.3 旧数据绑定

首次同步按以下顺序绑定旧记录：

1. 已有 `werss_source_id` 时按稳定 ID 匹配。
2. 未绑定记录按规范化后的固定 WeRSS Feed URL 精确匹配。
3. 仍未绑定时按完全相同的公众号名称匹配。
4. 不进行模糊匹配；冲突记录标记为需运维处理，并保持不变。

标准化名称为“一箱蛋”的来源在同步服务中强制排除。若本地存在历史记录，则保留历史数据但标记为排除且不得进入新的采集轮次。

## 6. 权限与安全模型

1. WeInsight 只访问精确端点 `http://127.0.0.1:8001/api/v1/wx/mps`，禁止修改主机、端口、协议和路径后访问其他接口。
2. 使用 WeRSS 专用 AK/SK，凭据由 `WEINSIGHT_WERSS_ACCESS_KEY` 和 `WEINSIGHT_WERSS_SECRET_KEY` 环境变量提供。
3. AK/SK 不写入 YAML、数据库、日志、错误响应、HTML、截图或测试夹具。
4. HTTP 客户端禁止跨源重定向，限制连接/读取超时、响应体大小、分页数量和最大公众号数量。
5. WeInsight 不调用 WeRSS 的 POST、PUT、PATCH 或 DELETE 接口。
6. 当前 WeRSS 镜像虽然支持 AK 权限字段，但公众号清单端点只验证 AK/SK、未对细粒度权限做强制判断。因此安全边界同时依赖：本机监听、精确出站端点、WeInsight 客户端仅实现 GET、专用凭据定期轮换。
7. 管理后台继续使用现有登录、会话和 CSRF 保护；只读公众号页面不新增写请求。

## 7. 后端实现方案

### 7.1 WeRSS 清单客户端

新增独立 `WeRSSCatalogClient`，职责仅包括：

- 使用 `Authorization: AK-SK <AK>:<SK>` 请求清单接口。
- 按 `limit=100` 和 `offset` 拉取全部分页。
- 校验响应结构、稳定 ID、名称、状态、总数一致性和重复 ID。
- 将结果转换为不包含凭据和响应正文的领域对象。
- 将 401/403、超时、连接失败、非法响应和分页不完整映射为稳定错误码。

客户端不生成数据库写入，也不实现任何 WeRSS 写接口。

### 7.2 来源同步服务

同步服务接收完整清单并在一个数据库事务中：

1. 排除“一箱蛋”。
2. 绑定兼容旧记录。
3. 新增必要的本地来源映射。
4. 更新名称、固定 Feed URL、上游状态和最近见到时间。
5. 将完整清单中不存在的已绑定来源标记为 `missing`。
6. 保留所有历史文章和任务引用。
7. 写入现有配置变更审计记录，记录新增、更新、停用、缺失、恢复、排除和冲突数量，不记录凭据或原始响应。

同步必须幂等；同一清单重复执行不得产生重复来源或重复审计噪声。

### 7.3 全局公众号周期

新增或收敛为一个全局配置 `pipelines.article.sync_interval_minutes`：

- 默认值：`10`。
- 最小值：`10`。
- 通过部署 YAML 或等价环境变量配置，不在公众号页面编辑。

每轮执行顺序：

1. 尝试同步 WeRSS 清单。
2. 同步成功后使用最新 `active` 来源；同步失败时允许使用上一次成功清单继续读取已知 `active` Feed，但不得改变上游状态。
3. 按现有并发上限读取 Feed，并通过文章唯一标识与哈希幂等写入原始表。
4. 为新增或待重试文章创建清洗任务。
5. 清洗成功后创建分析任务。
6. 分析流程继续执行现有报价提取并写入结果表。

每个阶段均使用现有任务状态推进水位。无新增文章时不重复清洗、分析或报价提取；单条失败进入可重试状态，不推进该条水位，也不阻塞其他来源和文章。

### 7.4 并发与恢复

- 来源同步使用数据库级互斥或等价的单执行者保护。
- 同一轮同步和来源映射更新在单事务内完成，异常时整批回滚。
- 采集、清洗、分析继续使用现有幂等任务和租约机制。
- WeRSS 恢复后下一轮自动重试；来源重新出现时从 `missing` 恢复为实际状态并清空 `upstream_missing_at`。
- 公众号链路不得申请 `wechat_ui_lock`。

## 8. 前端影响

“公众号名单”更名为“公众号状态”，删除：

- 新增公众号按钮。
- 编辑、启用、停用、删除操作。
- 公众号新增与编辑表单入口。
- 公众号级间隔、优先级、时间窗口、超时和处理开关。

只读页面展示：

- 公众号名称。
- WeRSS 上游状态。
- WeRSS 来源 ID。
- WeInsight 采集状态。
- 最近文章时间与最近成功采集时间。
- 文章总数。
- 待清洗、待分析和失败数量。
- 最近结构化错误。
- 状态更新时间。

页面状态标签包括“正常”“陈旧”“上游停用”“上游缺失”“采集异常”。保留“刷新”按钮，但刷新只重新查询 WeInsight 状态，不同步来源、不触发采集，也不修改 WeRSS。

运行配置页面只读展示当前全局周期，例如“公众号增量处理：每 10 分钟”。实际值由部署配置控制。

## 9. 测试与回归方案

### 9.1 单元与契约测试

- AK/SK 请求头生成且日志不含凭据。
- 清单分页、空清单、重复 ID、总数变化、非法字段和响应过大。
- 401/403、连接失败、超时和非法 JSON 的稳定错误码。
- 精确端点限制和重定向拦截。
- “一箱蛋”服务端排除。
- 旧数据按 ID、Feed URL、精确名称绑定，冲突不模糊合并。
- 新增、重复同步幂等、停用、缺失、恢复及失败不改状态。
- 全局周期默认 10 分钟且拒绝小于 10 的配置。

### 9.2 服务与数据测试

- 完整清单同步在单事务内提交，任一异常整批回滚。
- 新文章只入库一次。
- 无新增文章时不重复创建清洗、分析和报价任务。
- 单条失败可重试且不阻塞其他记录。
- 清洗成功自动进入现有增量分析与报价提取。
- 上游停用或缺失不删除历史文章和任务引用。
- 公众号任务不获取微信 UI 锁，微信群链路正常运行。

### 9.3 Web 测试

- 公众号页面不存在新增、编辑、启停和删除写入口。
- 直接访问旧表单 URL 返回 404 或重定向到只读状态页。
- 状态标签、时间、计数和错误码正确展示。
- 刷新操作不产生来源变更或采集任务。
- 桌面和窄屏布局可用，表格可访问且不会遮挡关键状态。

### 9.4 真实联调与视觉验收

1. 创建并启用专用 WeRSS AK/SK，写入本机环境变量。
2. 使用当前 9 个范围内公众号验证清单同步，“一箱蛋”计数为 0。
3. 验证 WeRSS 停用、恢复及临时不可用场景。
4. 验证一个真实新增文章完成采集、清洗、分析和报价提取。
5. 启动正式开发服务，实际登录后分别截取桌面与窄屏页面，由实施者检查状态层级、密度、溢出和空态。
6. 运行公众号、任务、结果、报告、微信群隔离和旧 RPA 退役回归测试。

## 10. 风险与缓解

| 风险 | 缓解措施 |
|---|---|
| WeRSS 清单接口版本变化 | 固定镜像摘要、严格契约校验、结构化失败且保留上次成功状态 |
| WeRSS AK 缺少端点级权限强制 | 本机监听、精确端点客户端、仅实现 GET、专用凭据与轮换 |
| 旧记录错误绑定 | ID 和 URL 优先、只允许精确名称匹配、冲突不自动合并 |
| 空清单误判全部删除 | 只有完整分页且通过总数校验后才允许标记缺失 |
| 两个 10 分钟周期相位接近导致延迟 | 接受最多一个额外周期的可见延迟；通过最近采集与文章时间监控，不建立跨系统事务 |
| 保留旧策略字段造成语义混淆 | UI 移除，运行时不读取；后续稳定版本再单独设计字段清理迁移 |
| 改造期间重复调度 | 单执行者保护、任务幂等、发布时先停旧公众号调度再切换 |

## 11. 分阶段落地与回滚

### 11.1 落地阶段

1. 数据库迁移与旧数据兼容读取。
2. WeRSS 只读客户端及来源同步服务。
3. 全局周期编排和增量链路回归。
4. 公众号只读状态页面。
5. 专用凭据配置与真实数据联调。
6. 停止旧公众号级调度入口，切换到全局周期。
7. 重新启动连续 24 小时观察。

### 11.2 回滚

若新同步链路异常，停止新的全局公众号周期并回滚 WeInsight 应用版本与对应数据库迁移备份。WeRSS 继续独立运行，微信群链路继续运行。不得恢复公众号旧 RPA，也不得通过直连 WeRSS 数据库临时绕过 API。

## 12. 验收标准

1. WeInsight 公众号页面完全只读，不存在公众号配置写入口。
2. WeRSS 是公众号新增、删除、启停和上游调度的唯一管理端。
3. WeInsight 使用专用 AK/SK 通过固定本机 GET 接口同步，未访问 WeRSS 私有表或写接口。
4. 全局周期默认 10 分钟且可部署配置，公众号级策略不再影响运行。
5. 新文章自动完成幂等采集、增量清洗、增量分析和报价提取。
6. WeRSS 临时不可用不造成来源误删除或历史数据丢失。
7. “一箱蛋”无新采集、清洗、分析和报价任务。
8. 公众号链路 UI 锁计数为 0，微信群链路无回归。
9. 桌面和窄屏真实截图通过视觉检查。
10. 发布后新的连续 24 小时观察通过；此前窗口不计入验收。
