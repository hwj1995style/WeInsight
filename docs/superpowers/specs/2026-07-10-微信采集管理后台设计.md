# WeInsight 微信采集管理后台设计

## 1. 背景与现状

WeInsight 当前是运行在 Windows 交互式桌面会话中的 Python CLI/RPA 系统，使用 MySQL、SQLAlchemy、PyMySQL、APScheduler、Playwright、pywinauto 和 wxauto4。项目没有现成 Web 服务、HTTP API 或前端构建链。

现有能力已经覆盖：

1. 微信群和公众号/订阅号两条独立采集链路。
2. `wechat_group_config` 和 `wechat_public_account_config` 两套采集名单配置。
3. `GroupPollingRunner`、`ArticlePollingRunner` 和全局 `wechat_ui_lock`。
4. 群消息 raw、clean、analysis、日报以及公众号文章 raw、clean、analysis、报价明细、日报。
5. 两条链路独立的 process task、采集日志、游标、进度、运行指标和失败补偿能力。
6. 群日报、文章日报和汇总日报的查询与 Markdown 导出能力。
7. 微信进程、版本、登录状态和 RPA 适配器的基础健康检查。

当前主要缺口是：

1. 没有登录页面和 Web 鉴权。
2. 名单配置只能通过 CLI 或数据库完成。
3. 没有面向管理员的持久化采集任务模型，现有 `wechat_*_process_task` 只表示清洗、分析和日报等后处理任务，不能承担采集任务的创建、停止和删除。
4. 没有独立常驻的公众号调度控制层，也没有安全停止、运行租约和漏跑合并语义。
5. 运行状态、日志、脱敏明细和日报没有统一可视化入口。
6. 当前 group scheduler 是同步循环，Web 请求不能直接复用这种进程模型来启动或强杀 RPA。

现有测试基线为 421 项测试通过。管理后台必须在保持现有测试全部通过的前提下增量建设。

## 2. 技术可行性评估

结论：技术上可行，属于中等规模改造，不需要重写现有采集、清洗、分析和日报链路。

可直接复用的能力：

| 需求 | 现有可复用能力 | 新增工作量判断 |
| --- | --- | --- |
| 群和公众号名单配置 | 两套配置表和 Repository | 低 |
| 采集执行 | 两套 PollingRunner、RPA Client、UI 锁 | 中 |
| 后处理 | 两套 process task、Service 和 Repo | 低到中 |
| 运行指标 | runtime metrics、collect log、cursor/progress | 中 |
| 日报 | 群日报、文章日报、汇总日报查询与生成服务 | 低到中 |
| 登录与 Session | 当前不存在 | 中 |
| 任务创建、停止、删除 | 当前不存在持久化采集作业模型 | 中到高 |
| 实时日志页面 | 当前缺少统一结构化事件和 SSE | 中 |
| 脱敏明细页面 | clean/analysis 已存在，缺少只读查询层 | 中 |

主要技术限制：

1. 微信 UI 操作不能并行。多个任务可以排队，但所有真实微信操作必须继续通过 `wechat_ui_lock` 串行执行。
2. 采集 Worker 必须运行在已登录的 Windows 交互式桌面会话中，不能作为 Session 0 Windows 服务运行。
3. “停止任务”只能在安全检查点协作停止，不能强杀 RPA 线程，否则可能留下 UI 锁或半完成状态。
4. 公众号任务必须保持微信群优先，并在核心群到期时主动让出微信 UI。
5. Web 只能展示脱敏后的群 clean/analysis 数据，不允许读取或返回群 raw 原文。

## 3. 目标与非目标

### 3.1 目标

1. 提供局域网内单管理员登录页面，默认账号 `admin`、默认密码 `admin123456`。
2. 首次登录不强制修改默认密码，但持续显示默认密码风险提醒，并支持管理员主动改密。
3. 在前端配置微信群和公众号/订阅号采集名单，支持新增、编辑、启用、停用和安全删除。
4. 手动创建单链路多目标采集任务，配置整体有效期、每日运行窗口和执行频率。
5. 公众号采集频率最小间隔为 10 分钟；微信群首版最小间隔为 30 秒。
6. 支持停止和软删除采集任务，停止必须等待安全检查点，任务历史和采集结果保留。
7. 展示 Worker、微信客户端、任务、运行实例、目标执行进度、UI 锁和结构化日志。
8. 定时探测微信客户端运行、版本、窗口、登录和 RPA 可用状态；异常时暂停领取新采集运行，恢复后自动继续。
9. 展示脱敏群消息、文章分析、报价明细、群日报、文章日报和汇总日报。
10. 采集后自动执行清洗和分析；支持手动生成当日临时日报，并在次日 `00:10` 自动补偿生成前一自然日最终版日报。
11. 尽量复用现有 Python、MySQL、SQLAlchemy、APScheduler、Runner、Repository 和 Service，不引入 Redis、Celery 和 Node 构建链。

### 3.2 非目标

1. 不提供公网访问。
2. 不支持用户注册、多管理员、角色权限、多租户或第三方单点登录。
3. 不允许一个采集任务同时包含微信群和公众号。
4. 不实现微信 UI 并行采集。
5. 不在 Web 中展示群 raw 原文、文章正文、HTML、原始文章链接或故障截图内容。
6. 不允许 Web 请求线程直接操作微信 UI、运行长耗时后处理或强杀 Worker。
7. 不提供业务数据清空、raw 数据下载或日志物理删除页面。
8. 不改变当前核心群不超过 5 个、公众号/订阅号不超过 20 个的小规模运行边界。

## 4. 方案比较与选型

### 4.1 方案一：Web 与调度器同进程

FastAPI Web 进程内运行 APScheduler 和 RPA。优点是部署单元少；缺点是 Web 重启影响调度，HTTP 和 RPA 故障互相传播，且不适合安全停止和交互式桌面隔离。

### 4.2 方案二：Web、采集 Worker、后处理 Worker 分进程，共用 MySQL

Web 只处理登录、配置、控制指令和查询；采集 Worker 处理调度与微信 RPA；后处理 Worker 处理清洗、解析、分析和日报。三个进程以 MySQL 作为任务与运行状态的唯一事实来源。

优点：

1. 最大限度复用当前技术栈。
2. Web 重启不影响 Worker。
3. 微信 UI 和非 UI 后处理隔离。
4. 不需要 Redis、Celery 或独立消息队列。
5. 能保留现有 group/article 双链路隔离与 UI 锁规则。

缺点是需要维护三个 Python 进程和数据库运行租约。

### 4.3 方案三：前后端分离与分布式任务队列

Vue/React + FastAPI + Celery/Redis。扩展能力强，但会引入 Node、Redis、队列运维和更多部署单元，对局域网单管理员、单机微信 UI 串行场景过重。

### 4.4 选型

采用方案二。Web 与 Worker 之间不直接远程调用，统一通过 MySQL 中的任务、运行、事件、请求和心跳记录协作。

## 5. 总体架构

### 5.1 Web 服务

技术组成：

1. FastAPI：路由、中间件、Session、表单和 JSON 接口。
2. Jinja2：服务端页面模板。
3. HTMX：局部刷新、表单提交和任务操作。
4. ECharts：运行指标、结果趋势和 TOP 图表。
5. FastAPI `StreamingResponse`：SSE 日志和状态增量推送。
6. HTMX、ECharts 等前端资源本地托管，不依赖公网 CDN。

Web 服务不导入或调用真实 RPA Client，不获取 `wechat_ui_lock`。

### 5.2 采集 Worker

采集 Worker 运行在已登录微信的 Windows 交互式会话，负责：

1. APScheduler 周期唤醒数据库调度检查。
2. 领取到期采集任务并创建运行实例。
3. 复用 `GroupPollingRunner` 和 `ArticlePollingRunner` 逐目标执行。
4. 维持微信群优先、公众号可中断恢复和 `wechat_ui_lock` 互斥。
5. 响应停止请求、刷新运行租约、写目标运行和结构化事件。
6. 定时执行微信客户端健康探测。

### 5.3 后处理 Worker

后处理 Worker 不操作微信 UI，负责：

1. 群 raw → clean → analysis。
2. 文章 raw → parse/clean → analysis/报价明细。
3. 消费手动日报生成请求。
4. 次日 `00:10` 生成前一自然日最终版群日报、文章日报和汇总日报状态。
5. 写 Worker 心跳、处理计数、失败摘要和结构化事件。

### 5.4 MySQL 控制面

MySQL 保存名单、任务定义、任务目标、运行实例、目标运行、结构化事件、健康检查、Worker 心跳、日报请求和现有业务结果。数据库状态是调度和页面展示的唯一事实来源，浏览器或 Web 进程状态不能决定任务是否运行。

### 7.1 微信群

支持新增、编辑、启用、停用和删除，配置字段复用现有：

- 群名称
- 是否核心群
- 优先级
- 默认轮询间隔
- 普通回溯屏数
- 额外回溯屏数
- 备注

### 7.2 公众号/订阅号

支持新增、编辑、启用、停用和删除，配置字段复用现有：

- 名称
- 类型
- 优先级
- 默认轮询间隔
- 每日窗口
- 每轮文章上限
- 是否只采当天
- 备注

### 7.3 停用、删除和改名约束

1. 被活动或待开始任务引用时，不能停用或删除；接口返回引用它的任务清单。
2. 被 stopped、completed 或 deleted 任务引用时，可以停用，但不能删除。
3. 删除前必须已停用，且不存在任何 `wechat_collection_job_target` 引用。
4. 删除只移除配置行，不删除 raw、clean、analysis、日志和日报。
5. 后端在删除事务中重新检查引用关系，数据库 `RESTRICT` 外键是最终保护。
6. 配置名称已经被现有业务表作为业务键。一旦产生采集数据或被任务引用，首版禁止直接改名；需要改名时停用旧配置并新建配置。

## 8. 调度、停止和异常恢复

### 8.1 调度参数

每个任务同时配置：

1. 整体有效期 `effective_start_at/effective_end_at`。
2. 每日运行窗口 `daily_window_start/daily_window_end`，支持跨午夜。
3. 任务级频率 `interval_seconds`。
4. 一个或多个同链路采集对象。

时区固定使用现有 `Asia/Shanghai`。公众号间隔不得小于 600 秒；微信群间隔不得小于 30 秒。

任务创建后，任务级 `interval_seconds` 是管理后台调度的唯一频率口径。名单配置中的默认轮询间隔只用于创建任务时预填表单，以及保留现有 CLI 兼容性；采集 Worker 不再同时叠加名单默认间隔做第二次到期判断。

同一采集对象不能同时加入整体有效期和每日窗口均重叠的活动任务。前端提前提示，后端事务校验为最终判断。

### 8.2 数据库调度

APScheduler 仅每约 5 秒唤醒一次调度检查，任务定义、下一次运行和运行租约全部保存到 MySQL。

调度步骤：

1. 查询当前到期任务。
2. 在事务中领取任务并创建唯一运行实例。
3. 全局优先领取微信群运行，再领取公众号运行。
4. 同一任务最多存在一个 `queued/running` 实例。
5. 多目标按优先级逐个执行。
6. 完成后按原计划时间计算下一次运行，避免执行耗时导致持续漂移。

Worker 离线或进程重启造成多个周期漏跑时，只在当前仍处于任务有效期和每日窗口内立即合并补跑一次。其余遗漏周期记录 `misfire` 事件，不集中执行历史 RPA。

### 8.3 安全停止

1. 尚未运行的 scheduled 任务可直接转为 stopped。
2. active 任务先转为 stop_requested，页面显示“停止中”。
3. 采集 Worker 在领取前、目标之间和公众号阶段检查点检查停止状态。
4. 微信群当前目标完成并释放 UI 锁后停止。
5. 公众号复用现有状态机检查点，保存进度后中断。
6. 不强杀 RPA 线程，不在单个 UI 动作中间中断。
7. 任务停止不删除历史运行、结构化事件、采集结果或日报。

### 8.4 Worker 租约与恢复

Worker 每 10 秒刷新心跳和运行租约。租约超时后：

1. 当前运行标记为 `aborted`。
2. 过期 `wechat_ui_lock` 按现有租约规则恢复。
3. 写入异常恢复事件。
4. 任务是否继续下一周期由任务状态、有效期和健康状态决定。

数据库不可用时禁止开始新的 RPA，Worker 采用退避方式重试数据库连接。

### 8.5 托管模式与现有 CLI 共存规则

管理后台采集 Worker 启用后进入托管模式，不能与现有长期 group scheduler 或人工启动的 article scheduler 并行运行：

1. 部署采集 Worker 前停止并注销现有长期 `run-group-scheduler` Windows 计划任务。
2. `run-group-scheduler` 和 `run-article-scheduler` 在发现采集 Worker 心跳有效时拒绝启动真实调度。
3. `collect-group-once` 和 `collect-article-once` 继续保留为本机故障恢复入口，但必须获取同一 `wechat_ui_lock`，采集 Worker 正在执行时拒绝操作微信。
4. Fake RPA、配置检查、微信健康检查和纯数据库后处理 CLI 不受托管模式限制。
5. 采集 Worker 启动时检查同主机是否已有有效 collector 心跳；重复实例不领取任务，并在健康页面记录重复实例错误。

数据库运行唯一键、运行租约和 `wechat_ui_lock` 仍是并发保护的最终边界，不能只依赖启动脚本或进程检查。

## 9. 微信客户端健康探测

采集 Worker 定时复用 `WechatDesktopProbe` 检查：

1. 微信进程是否存在。
2. 客户端版本是否与配置一致。
3. 主窗口是否可访问。
4. 微信是否已登录。
5. RPA 适配器是否可用。
6. 最近成功探测时间和连续失败次数。

任意一次健康检查失败即暂停领取新的群和公众号采集运行，当前任务在安全检查点完成或中断。连续失败在总览页升级为红色告警。恢复为 `ok` 后自动恢复调度，并继续遵守只合并补跑一次的规则。

健康探测本身不得主动打开群或公众号，不得与真实采集争抢微信窗口。

## 10. 自动后处理与日报

### 10.1 自动后处理

采集成功写入现有 raw 和 process task 后，后处理 Worker 自动执行：

1. 群 clean。
2. 群 analysis。
3. 文章 parse/clean。
4. 文章 analysis 和报价明细。

继续复用现有每阶段最多三次重试、独立任务表和失败隔离。后处理失败不反向停止采集任务，也不能修改另一链路状态。

分析完成后只将对应日期的日报任务标记为数据有变化，不立即生成未完结的当日日报。

### 10.2 手动生成当日日报

管理员可选择今日及可选采集对象提交日报请求。Web 只写 `wechat_report_generation_request`，由后处理 Worker 异步生成。

当日手动日报规则：

1. 状态为 `provisional`。
2. `data_cutoff_time` 等于管理员提交请求的时间。
3. 当天重复生成使用 upsert 更新同一份日报，不产生重复记录。
4. 页面明确展示“临时版”和数据截止时间。

### 10.3 次日自动补偿

次日 `00:10` 自动创建前一自然日 `all` 补偿请求，顺序执行：

1. 群日报。
2. 文章日报。
3. 汇总日报状态复核。

补偿结果标记为 `final`。某个对象失败不阻止其余对象，失败对象保留请求、错误摘要和重试入口。手动生成历史日期时直接生成最终版。

## 11. 后端接口与服务边界

主要接口域：

- `/auth/*`：登录、退出、主动改密。
- `/sources/groups/*`：微信群名单查询、新增、编辑、启用、停用、删除。
- `/sources/articles/*`：公众号名单查询、新增、编辑、启用、停用、删除。
- `/jobs/*`：任务创建、列表、详情、停止、软删除。
- `/runs/*`：运行实例、目标运行和结构化事件。
- `/workers/*`：Worker 心跳、微信健康和 UI 锁状态。
- `/results/groups/*`：脱敏群消息和分析明细。
- `/results/articles/*`：文章分析和报价明细。
- `/reports/*`：群日报、文章日报、汇总日报和生成请求。
- `/events/stream`：SSE 增量事件。

路由只处理身份、参数和响应映射，业务规则放在 Service，SQL 放在 Repository。Web 路由不能导入真实 RPA Client。

关键后端校验：

1. 任务整体开始时间早于结束时间。
2. 至少选择一个已启用、未删除对象。
3. 同一任务目标属于同一链路。
4. 公众号间隔不少于 600 秒。
5. 微信群间隔不少于 30 秒。
6. 同一对象不存在重叠活动任务。
7. 更新、停止和删除使用 `version` 乐观锁。
8. 重复停止、删除或日报请求保持幂等。
9. 所有列表执行服务端分页并限制最大页容量。

## 12. 权限与安全模型

### 12.1 登录安全

1. 密码使用 Argon2id 哈希，不保存明文。
2. 默认账号为 `admin/admin123456`，首次登录不强制改密。
3. 仍使用默认密码时持续显示安全提醒。
4. 连续登录失败 5 次锁定 15 分钟。
5. Session 令牌使用高强度随机值，数据库只保存令牌哈希。
6. Cookie 设置 `HttpOnly`、`Secure`、`SameSite=Strict`。
7. 所有写请求校验 CSRF Token。
8. 管理员忘记密码时只允许在采集机本地通过 CLI 受控重置。

### 12.2 网络边界

1. 生产局域网使用 HTTPS。
2. Web 只绑定指定内网地址，不监听公网网卡。
3. Windows 防火墙只允许指定局域网网段访问。
4. 采集 Worker 和后处理 Worker 不开放网络端口。

### 12.3 数据库最小权限

生产建议使用独立 MySQL 账号：

1. Web 账号：可读写管理员、Session、名单、采集任务、日报请求；可读安全结果、运行、事件、健康和日报；禁止 SELECT 群 raw 表，禁止写 UI 锁和业务结果表。
2. 采集 Worker 账号：可读任务与名单，可写运行、事件、健康、raw、cursor/progress、collect log 和 UI 锁；禁止访问管理员密码和 Session。
3. 后处理 Worker 账号：可读写 process task、clean、analysis、报价、日报、日报请求和事件；禁止访问管理员密码和 Session，禁止写 UI 锁。

开发环境可以继续共用现有开发账号，生产部署必须按上述权限拆分。

### 12.4 输出安全

1. 群明细只查询 clean/analysis，Web 数据库账号无权读取群 raw。
2. 文章页面不返回正文、HTML、原始文章链接和运行时临时正文。
3. Markdown 禁用原始 HTML，并执行 XSS 清理。
4. 错误摘要继续执行 URL、正文和敏感信息脱敏。
5. 故障截图不通过 Web 预览、下载或静态文件服务。
6. 页面只显示采集主机名和经校验的本机绝对截图路径，并支持复制路径文本。
7. 截图路径必须位于配置的 `runtime.screenshot_dir` 下；页面提示路径只能在采集机本地打开。

不强制修改默认密码会增加局域网内被猜测登录的风险。该风险由 HTTPS、防火墙、登录锁定、默认密码持续提醒和主动改密入口共同缓解，但不能完全消除。

## 13. 前端页面与交互

首版桌面端优先，支持 1366×768 及以上分辨率，不单独建设移动端任务编辑流程。

### 13.1 登录页

用户名、密码、失败提示和锁定提示。默认密码未修改时，登录后在所有页面顶部持续显示风险提醒。

### 13.2 总览看板

展示：

1. 采集 Worker、后处理 Worker、微信健康和 UI 锁状态。
2. 活动、停止中和失败任务数量。
3. 今日成功/失败运行次数。
4. 今日群消息、文章入库量、任务积压和日报生成数。
5. 最近 24 小时成功/失败趋势。
6. 最近异常、主机名和本机截图路径。

### 13.3 采集名单

微信群和公众号独立页签，支持新增、编辑、启用、停用和删除。停用或删除被阻止时，页面展示引用它的任务。

### 13.4 采集任务

任务列表支持链路、状态、日期和名称筛选。创建表单包含名称、链路、多目标、整体有效期、每日窗口和频率。公众号低于 10 分钟时前端即时提示，但后端校验是最终判断。

任务详情展示下一次运行、当前状态、目标、运行历史和操作记录。仅在合法状态显示停止或删除操作。

### 13.5 运行监控与日志

运行实例展示排队、运行中、部分成功、成功、失败、取消和异常终止。目标级展示阶段、批次号、读取量、写入量、重复量、跳过量和失败摘要。

SSE 增量追加结构化事件；断线后使用最后事件 ID 补读。日志支持任务、运行、目标、链路、级别和时间筛选。

### 13.6 结果明细

群明细展示：

- 时间
- 群名
- 脱敏发送人
- 脱敏内容
- 供需类型
- 地区
- 品类
- 关键词
- 商机评分
- 联系方式标记

文章明细展示：

- 公众号
- 标题
- 发布时间
- 报价日期
- 采集时间
- 摘要
- 标签
- 内容长度
- 分析版本

报价明细展示：

- 地区和市场
- 产品和规格
- 原始报价
- 统一价格和单位
- 涨跌
- 换算方法与置信度

图表包括群供需分布、小时分布、地区/品类/关键词 TOP、文章数量趋势、标签 TOP 和统一报价趋势。

### 13.7 日报

群日报、文章日报和汇总日报独立页签。支持日期和对象筛选、安全渲染 Markdown、指标图表和下载脱敏 Markdown。

管理员可手动生成今日临时日报；页面显示生成状态和数据截止时间。次日补偿完成后前一日报告显示最终版。

## 14. 影响范围与文件组织建议

新增或重点调整范围：

1. `requirements.txt`：增加 FastAPI、Uvicorn、Jinja2、表单解析、Argon2id 和安全 Markdown 渲染依赖。
2. `app/core/config.py`：增加 Web、Session、Worker、健康探测和日报补偿配置。
3. `app/web/`：应用入口、中间件、路由、模板和静态资源。
4. `app/services/`：身份、名单、任务、运行、结果和日报请求服务。
5. `app/workers/`：采集 Worker、后处理 Worker、调度计算、心跳和恢复。
6. `app/storage/`：管理员、Session、任务、运行、事件、健康、日报请求和安全结果查询 Repository。
7. `sql/migrations/`：只增加新表、索引和日报生命周期字段。
8. `scripts/windows/`：Web、采集 Worker、后处理 Worker 的启动、注册和注销脚本。
9. `tests/`：单元、Repository、API、Worker、浏览器和文档测试。

不在 Web 模块继续堆叠 `app/main.py`；现有 CLI 保留用于本机运维和故障恢复。

建议迁移拆分：

1. 管理员与 Session 表。
2. 采集任务、目标、运行和目标运行表。
3. 结构化事件、Worker 心跳和微信健康表。
4. 日报请求表。
5. 群日报和文章日报生命周期字段。

迁移全部采用增量 DDL，不删除、清空或重建现有业务表。

3. 重叠任务检测、任务状态机、运行状态机和软删除。
4. 漏跑合并、租约、心跳超时和异常恢复。
5. 默认管理员初始化、Argon2id、登录锁定、Session 和 CSRF。
6. 名单停用、引用检查、安全删除和禁止改名。
7. 日报临时版、最终版、截止时间和幂等生成。
8. 微信健康状态转换、暂停和恢复。
9. 日志、Markdown、错误摘要和截图路径安全处理。
10. 名单默认间隔只作为表单默认值，任务级频率是托管调度唯一口径。
11. 托管模式下长期 scheduler 和人工真实采集 CLI 的互斥检查。

### 15.2 MySQL 集成测试

覆盖：

1. 全新库和历史库增量迁移。
2. 两个 Worker 并发领取时只能一个成功。
3. `(job_id, scheduled_at)` 防重复。
4. 租约未过期不能抢占，过期后可恢复。
5. 被任意任务引用的名单无法删除。
6. 手动日报请求幂等。
7. 分页、筛选和安全结果查询不访问群 raw。

### 15.3 API 与浏览器测试

覆盖：

1. 登录、退出、默认密码提醒和主动改密。
2. 名单新增、编辑、启用、停用和删除。
3. 任务创建、停止中、停止完成和软删除。
4. 公众号低于 10 分钟时前后端均拒绝。
5. SSE 断线重连和按事件 ID 补读。
6. 当日日报手动生成、临时版和次日补偿最终版。
7. Markdown XSS、CSRF、路径穿越和未登录访问。
8. Web 无法取得群 raw、文章正文、原始文章链接或截图内容。
9. 1366×768 下主要页面无横向溢出。

### 15.4 Worker 与 RPA 回归

先用 Fake RPA 覆盖：

1. 微信群优先和公众号主动让锁。
2. 任意时刻只有一个微信 UI 操作。
3. 停止只在安全检查点生效。
4. 微信异常时不启动采集，恢复后只补跑一次。
5. Worker 崩溃和租约恢复。
6. 后处理失败不停止采集、不影响另一链路。
7. 次日 `00:10` 自动补偿。

然后在人工值守下执行一个核心群和一个公众号的真实小窗口 POC。

### 15.5 回归基线

每个阶段必须保持当前 421 项测试全部通过，再增加本功能测试。任何现有 group/article 隔离、UI 锁、输出安全或日报测试失败都不能进入真实 RPA 验证。

## 16. 验收标准

1. 可使用 `admin/admin123456` 登录，首次登录不被强制改密，页面持续显示默认密码提醒。
2. 群和公众号名单可新增、编辑、启用、停用；仅未被任务引用且已停用的配置可删除。
3. 可创建同链路多目标任务，并同时配置整体有效期、每日窗口和频率。
4. 公众号低于 10 分钟的任务创建成功数为 0。
5. 重复运行实例数为 0。
6. 并发持有微信 UI 锁的运行数为 0。
7. 停止请求在当前目标的 RPA 超时上限内进入 stopped。
8. 微信异常在一个探测周期内显示到总览，并阻止新采集运行。
9. Worker 重启后 60 秒内恢复数据库调度检查。
10. 当日手动日报可重复更新并显示数据截止时间；次日补偿后转为最终版。
11. Web 返回群 raw 原文、文章正文、原始文章链接或截图内容的数量为 0。
12. 单链路失败不得修改另一链路的任务状态、游标和失败计数。
13. 当前 421 项测试和新增测试全部通过。
14. 管理后台采集 Worker 运行期间，现有长期 scheduler 和未持有 UI 锁的人工真实采集命令成功启动数为 0。

## 17. 风险与应对

| 风险 | 等级 | 应对 |
| --- | --- | --- |
| 微信 UI 或客户端升级导致 RPA 失效 | 高 | 定时健康探测、版本固定、异常时阻断新运行、升级前回归 |
| 锁屏、休眠或 Session 0 导致窗口不可访问 | 高 | 采集 Worker 仅在交互式会话启动，启动时检查桌面环境 |
| 默认密码长期不修改 | 高 | HTTPS、防火墙、失败锁定、持续提醒、主动改密；接受仍存在剩余风险 |
| Web 误触发并行 RPA | 高 | Web 只写任务，采集 Worker 单通道领取，UI 锁和唯一运行约束双重保护 |
| 现有 scheduler 或人工 CLI 与托管 Worker 并行 | 高 | 托管模式互斥检查、注销旧计划任务、所有真实采集统一获取 UI 锁 |
| 停止时强杀导致状态损坏 | 高 | 只允许 stop_requested 和安全检查点协作停止 |
| Worker 重启产生重复运行 | 高 | 数据库唯一键、运行租约、心跳和漏跑合并 |
| 健康探测瞬时误报 | 中 | 一次失败暂停新运行，连续失败升级告警，恢复后自动继续 |
| 名单删除破坏任务引用 | 中 | 先停用、事务复核、RESTRICT 外键、历史任务保留引用 |
| 名称修改破坏历史关联 | 中 | 产生数据或被引用后禁止改名，使用停用旧配置并新建配置 |
| 结构化事件持续增长 | 中 | 建立按时间与运行的索引，运行稳定后制定归档和保留策略 |
| 日报手动生成与自动补偿并发 | 中 | 日报请求幂等、同日期对象串行、upsert 和 provisional/final 状态 |
| clean 脱敏规则漏掉敏感内容 | 高 | Web 禁止 raw 权限，继续扩充脱敏回归，真实 POC 人工抽查 |
| 局域网 HTTPS 证书维护 | 中 | 部署手册记录证书签发、续期和回滚，开发仅 localhost 可用 HTTP |

## 18. 分阶段落地建议

### 阶段一：数据与只读后台

交付管理员、Session、增量迁移、Web 骨架、Worker/微信健康、总览、脱敏明细和日报只读页面。此阶段不允许前端创建真实采集任务。

### 阶段二：名单配置管理

开放微信群和公众号的新增、编辑、启用、停用和安全删除，补齐引用检查、禁止改名和数据库外键回归。

### 阶段三：Fake RPA 任务闭环

开放任务创建、停止、软删除、运行实例、结构化事件、SSE 和手动日报。使用 Fake RPA 完成调度、停止、租约、漏跑合并和日报双触发测试。

### 阶段四：单目标受控真实 POC

在人工值守下启用一个核心群和一个公众号，验证微信健康、UI 锁、任务停止、自动后处理、日志和本机截图路径。

### 阶段五：自动调度与日报补偿

启用整体有效期、每日窗口、频率调度和次日 `00:10` 补偿。观察 Worker 重启、微信掉线和数据库短暂不可用恢复。

### 阶段六：小规模扩容

验收通过后再扩到现有设计上限：核心群不超过 5 个、公众号不超过 20 个。任一 UI 锁、健康、失败率或核心群等待指标不达标时回退到上一阶段。

## 19. 设计结论

管理后台采用 FastAPI + Jinja2 + HTMX + ECharts，Web、采集 Worker、后处理 Worker 三进程共用 MySQL。该方案保留现有 Python/MySQL/RPA 技术栈和双链路隔离，不引入分布式队列，并通过数据库任务状态、运行租约、结构化事件、微信健康探测和安全检查点实现可控的创建、停止、删除、监控和恢复。

前端的核心安全边界是：不直接控制微信 UI、不读取群 raw、不展示文章正文或故障截图内容。真实 RPA 必须继续遵守交互式桌面、微信群优先、UI 锁串行和人工值守分阶段扩容原则。
