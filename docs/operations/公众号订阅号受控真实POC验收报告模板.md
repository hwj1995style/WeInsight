## 4. 通过条件

满足以下条件，才允许给出 Go 判断：

```text
1 个授权公众号/订阅号闭环通过后，才进入 3 个账号人工值守 POC
3 个授权公众号/订阅号人工值守 POC 通过
每个授权账号在窗口内按每小时 1 次执行
只采集当天发布数据
重复文章被去重，不产生重复入库
article 链路失败可查看、可重试、可回滚
微信群链路配置、任务、日志和日报未被 article 链路修改
微信 UI 锁没有长时间残留
CLI、日志和报告不展示文章正文或具体文章链接
```

## 5. 停止条件

出现以下任一情况，必须给出 No-Go 或延后判断：

```text
任一授权账号连续失败 3 次
核心群等待超过 max_core_group_block_seconds
微信 UI 卡死、锁屏、掉线或长时间不释放
wechat_ui_lock 残留且无法自动恢复
article 链路写入或污染微信群链路状态
采集结果出现非当天发布数据且无法解释
去重失效导致重复入库
CLI、日志或报告出现文章正文或具体文章链接
```

## 6. 单机串行交错限制

本次 POC 必须单独说明单机串行交错限制：

```text
微信群链路和公众号/订阅号链路可以独立调度、独立入库、独立清洗、独立分析。
只要涉及同一个微信 PC UI，RPA 操作必须通过 wechat_ui_lock 串行交错执行。
article 链路不是 UI 并行执行，只是低优先级让路执行。
如果核心群轮询变密、微信 UI 卡顿或锁屏，两个链路都会受到同一个微信窗口影响。
如果该限制不可接受，应拆分采集机或使用独立桌面。
```

结论填写规则：

```text
Go：所有通过条件满足，停止条件未触发，核心群等待影响可接受。
No-Go：触发停止条件，或单机串行交错限制已经影响核心群链路。
延后：缺少足够样本、失败原因未定位，或仍需补充观察窗口。
```

No-Go 或延后时，必须回滚到手动单账号模式，继续优先观察微信群链路。
