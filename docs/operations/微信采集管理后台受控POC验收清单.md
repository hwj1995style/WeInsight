# 微信采集管理后台受控 POC 验收清单

## 范围与硬门禁

- 基准客户端：微信 PC 4.1.8.107；版本不一致时停止真实 POC 并重新评估。
- 第一轮只允许 1 个核心群和 1 个公众号，必须由管理员在采集机前人工值守。
- Fake RPA 未全部通过、微信健康异常、UI lock 为 held、旧 scheduler 存在或回滚负责人不在场时，禁止真实采集。
- 故障截图不得通过 Web 展示；只记录经过校验的本机截图路径。
- 本清单不会授权生产 SQL、计划任务、防火墙或真实微信操作。

## 固定执行顺序

### 1. 完整自动化回归

- [ ] 运行完整 pytest，记录提交号、通过数和 skip 原因。
- [ ] 运行敏感输出、UTF-8 和差异检查，不记录密码、Session、raw 正文或截图。

### 2. 只读上线前检查

- [ ] 复核三任务、HTTPS、MySQL 最小权限、证书和旧 scheduler。
- [ ] 微信健康为 ok；UI lock 独立诊断只接受 free_no_row 或 free_expired。

### 3. Fake RPA 全量验收

- [ ] 验证多目标、协作停止、断线、心跳过期、健康门禁和 SSE 重连。
- [ ] 验证手动当日日报临时版和次日补偿路径，不伪造次日 00:10 真实结果。

### 4. 单核心群真实 POC

- [ ] 仅一个获批核心群，有效窗口 30 分钟，间隔不少于 30 秒。
- [ ] 记录 run ID、UI lock 等待、健康探测、结果和停止耗时。

### 5. 单公众号真实 POC

- [ ] 仅一个获批公众号，有效窗口 30 分钟，采集频率至少 10 分钟，每轮最多 1 篇。

### 6. 双链路交错运行

- [ ] 群优先，单群与单公众号交错运行 1 到 2 小时。
- [ ] 验证同一时刻只有一个有效 UI owner，核心群等待未超阈值。

### 7. 次日 00:10 补偿验收

- [ ] 复核前一自然日群、文章和汇总日报为 final，记录 compensation all 与 cutoff。

### 8. Go / Watch / No-Go 与回滚

- [ ] 由业务、运维和开发共同签署，不得由执行人预填 Go。
- [ ] 重复运行、并发 UI owner、连续健康失败、日报未 final 或敏感泄漏均触发 No-Go。
- [ ] 按部署手册选择回滚路径并记录恢复点。
