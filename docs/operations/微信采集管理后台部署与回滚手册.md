# 微信采集管理后台部署与回滚手册

> 状态说明：本文遗留的公众号 article 手动 CLI 仅用于审计历史，不再可执行，也不得作为故障回退。公众号现行操作以 `docs/operations/公众号RSS采集运行手册.md` 为唯一准则；本文微信群操作仍有效。

## 1. 适用范围与执行原则

本文用于在一台受控 Windows 10/11 采集机上部署 WeInsight 管理后台、采集 Worker 和后处理 Worker。所有命令均为人工执行模板；本文本身不会执行 SQL、计划任务、防火墙或 ACL 修改。

部署必须遵守以下原则：

1. 先备份、后变更；每一步验收通过后再进入下一步。
2. 所有 `<...>` 均为占位符，执行前由管理员在受控副本中替换并复核，禁止把真实 IP、账号、密码或私钥提交到仓库。
3. 三个进程使用三个独立 MySQL 用户和三份配置，不得用一个高权账号替代。
4. 采集 Worker 只能运行在当前登录用户的交互式桌面会话，禁止 SYSTEM、服务账号和 Session 0。
5. Fake RPA 和单目标人工值守验收通过前，不启用无人值守真实采集或扩容。

## 2. 前置检查与备份

在维护窗口记录并复核：

- 项目版本、提交号、项目绝对目录和配置文件清单；
- Windows、Windows PowerShell 5.1、Python、MySQL 版本；
- 微信 PC 4.1.8.107，自动更新已关闭；
- 当前用户为实际登录的交互用户，锁屏、休眠已关闭；
- MySQL 全量或一致性备份、现有任务清单、现有配置和证书 ACL 导出；
- `runtime` 目录剩余空间和三个日志目录可写；
- 回滚负责人、维护窗口和变更单号。

只读记录示例：

```powershell
git rev-parse HEAD
$PSVersionTable.PSVersion
python --version
Get-ScheduledTask | Where-Object { $_.TaskName -like "*WeInsight*" } |
    Select-Object TaskName, State
```

数据库备份由组织批准的备份工具完成。不要把备份文件放入 Web 静态目录或仓库。

## 3. TLS 文件与 ACL

在采集机本地准备以下占位路径：

```text
<TLS_CERT_PATH>  对应 WEINSIGHT_TLS_CERTFILE
<TLS_KEY_PATH>   对应 WEINSIGHT_TLS_KEYFILE
```

私钥不得进入仓库、命令历史、应用日志、工单正文或聊天记录。证书名称必须覆盖管理员实际访问的内网主机名或 IP，并由客户端系统信任链验证；不得跳过证书校验。

由管理员在维护窗口执行并逐条复核以下 ACL 模板：

```powershell
$TLS_CERT_PATH = "<TLS_CERT_PATH>"
$TLS_KEY_PATH = "<TLS_KEY_PATH>"
$RUN_AS_USER = "<DOMAIN_OR_HOST>\<RUN_AS_USER>"

Test-Path -LiteralPath $TLS_CERT_PATH -PathType Leaf
Test-Path -LiteralPath $TLS_KEY_PATH -PathType Leaf
icacls $TLS_KEY_PATH /inheritance:r
icacls $TLS_KEY_PATH /grant:r "${RUN_AS_USER}:(R)"
icacls $TLS_KEY_PATH /remove:g "Users" "Authenticated Users" "Everyone"
icacls $TLS_CERT_PATH /grant:r "${RUN_AS_USER}:(R)"
Get-Acl -LiteralPath $TLS_KEY_PATH | Format-List Owner, AccessToString
Get-Acl -LiteralPath $TLS_CERT_PATH | Format-List Owner, AccessToString
```

验收标准：私钥只授运行账号所需的读取权限；普通用户组没有读取权限；Web 进程只收到路径，不在启动日志中输出文件内容。组织若要求管理员组保留应急访问，必须由安全负责人批准并记录。

## 4. 创建独立 MySQL 用户并分配 Role

### 4.1 确认固定 schema 模板

`sql/operations/grant_admin_stack_roles.sql` 固定引用 `weinsight_prod`。DBA 必须先确认生产 schema 正是该名称。若名称不同，只能在受控副本中逐项修改每个显式 schema 标识并重新审计，禁止直接在生产会话做未复核的全局替换。

使用交互密码提示登录 DBA 会话，密码不会出现在命令行：

```powershell
mysql --user <DBA_USER> --password
```

在确认备份和 schema 后，由 DBA 在该交互会话中执行 role 模板。模板只创建三个 role 和逐表 GRANT，不创建用户、不包含用户凭据。

### 4.2 创建三个用户

优先使用 MySQL 随机密码，由 DBA 将一次性输出直接录入组织 secret manager，不复制到本文、终端日志或变更单：

```sql
CREATE USER '<WEB_DB_USER>'@'<APP_HOST>' IDENTIFIED BY RANDOM PASSWORD;
CREATE USER '<COLLECTOR_DB_USER>'@'<APP_HOST>' IDENTIFIED BY RANDOM PASSWORD;
CREATE USER '<PIPELINE_DB_USER>'@'<APP_HOST>' IDENTIFIED BY RANDOM PASSWORD;
```

若组织不允许数据库生成随机密码，使用组织批准的交互式 secret 流程创建用户；禁止在 SQL 文件、PowerShell 命令、配置文件或环境变量设置命令中写明文值。

### 4.3 分配、默认启用并审计 Role

```sql
GRANT 'weinsight_web_role' TO '<WEB_DB_USER>'@'<APP_HOST>';
GRANT 'weinsight_collector_role' TO '<COLLECTOR_DB_USER>'@'<APP_HOST>';
GRANT 'weinsight_pipeline_role' TO '<PIPELINE_DB_USER>'@'<APP_HOST>';

SET DEFAULT ROLE 'weinsight_web_role' TO '<WEB_DB_USER>'@'<APP_HOST>';
SET DEFAULT ROLE 'weinsight_collector_role' TO '<COLLECTOR_DB_USER>'@'<APP_HOST>';
SET DEFAULT ROLE 'weinsight_pipeline_role' TO '<PIPELINE_DB_USER>'@'<APP_HOST>';

SHOW GRANTS FOR '<WEB_DB_USER>'@'<APP_HOST>' USING 'weinsight_web_role';
SHOW GRANTS FOR '<COLLECTOR_DB_USER>'@'<APP_HOST>' USING 'weinsight_collector_role';
SHOW GRANTS FOR '<PIPELINE_DB_USER>'@'<APP_HOST>' USING 'weinsight_pipeline_role';
```

审计必须确认：Web 无群/文章 raw 和 UI lock；Collector 无管理员、Session 和日报生命周期写权限；Pipeline 无管理员、UI lock 和 job run 写权限。任何额外直接授权都必须先移除或单独审批。

## 5. 三份配置与持久环境变量

从生产模板分别创建三份不入仓库的配置，例如：

```text
<WEB_CONFIG_PATH>
<COLLECTOR_CONFIG_PATH>
<PIPELINE_CONFIG_PATH>
```

三份配置分别填写 `<WEB_DB_USER>`、`<COLLECTOR_DB_USER>`、`<PIPELINE_DB_USER>`；数据库密码字段继续使用 `${WEINSIGHT_MYSQL_PASSWORD}`，由各 start 脚本在当前子进程内映射角色密码。

通过 Windows 受控凭据工具、组织 secret manager 或“用户环境变量”图形界面设置以下 User 级值，不在命令行输入真实值：

```text
WEINSIGHT_WEB_MYSQL_PASSWORD
WEINSIGHT_COLLECTOR_MYSQL_PASSWORD
WEINSIGHT_PIPELINE_MYSQL_PASSWORD
WEINSIGHT_WEB_HOST
WEINSIGHT_TLS_CERTFILE
WEINSIGHT_TLS_KEYFILE
```

只验证“存在”，不要回显值：

```powershell
@(
    "WEINSIGHT_WEB_MYSQL_PASSWORD",
    "WEINSIGHT_COLLECTOR_MYSQL_PASSWORD",
    "WEINSIGHT_PIPELINE_MYSQL_PASSWORD",
    "WEINSIGHT_WEB_HOST",
    "WEINSIGHT_TLS_CERTFILE",
    "WEINSIGHT_TLS_KEYFILE"
) | ForEach-Object {
    $present = -not [string]::IsNullOrWhiteSpace(
        [Environment]::GetEnvironmentVariable($_, "User")
    )
    "{0}_present={1}" -f $_, $present
}
```

生产计划任务不启用共享数据库密码 fallback。

生产最小权限模式禁止名单改名。当前改名历史守卫会检查 raw/cursor/progress 等表，而 Web role 按安全边界无权读取这些表；因此改名请求必须 fail closed。操作方式固定为“停用旧配置并新建替代配置”：先确认历史与任务引用保留，再重新绑定新任务。后续只有在实现不暴露 raw 的安全投影或引用表后，才可重新评估改名能力。

## 6. 防火墙与 HTTPS 边界

Web 只绑定明确私网地址和端口 8848。`<ADMIN_LAN_CIDR>` 必须替换为经批准的管理员局域网 CIDR；禁止使用任意来源或公网地址。

```powershell
New-NetFirewallRule `
    -DisplayName "WeInsight Admin HTTPS 8848" `
    -Direction Inbound `
    -Protocol TCP `
    -LocalPort 8848 `
    -RemoteAddress <ADMIN_LAN_CIDR> `
    -Action Allow
```

变更后从批准网段使用系统信任链验证 HTTPS；从非批准网段确认连接被拒绝。不要使用跳过证书验证的参数或全局回调。

## 7. 切换旧 Scheduler 并注册新管理栈

必须先检查两个历史名称；任一存在时，停止并注销后才允许注册新 stack：

```powershell
$legacyNames = @("WeInsight-Group-Scheduler", "WeInsight Group Scheduler")
foreach ($name in $legacyNames) {
    $task = Get-ScheduledTask -TaskName $name -ErrorAction SilentlyContinue
    if ($null -ne $task) {
        Stop-ScheduledTask -TaskName $name -ErrorAction SilentlyContinue
        Unregister-ScheduledTask -TaskName $name -Confirm:$false
    }
}
```

确认两个旧名称均不存在、数据库备份完成、TLS/角色/防火墙已验收后，在目标交互用户的 PowerShell 中注册：

```powershell
powershell -NoProfile -File scripts\windows\register_admin_stack.ps1 `
    -ProjectRoot "<PROJECT_ROOT>" `
    -ConfigPath "<WEB_CONFIG_PATH>" `
    -WebConfigPath "<WEB_CONFIG_PATH>" `
    -CollectorConfigPath "<COLLECTOR_CONFIG_PATH>" `
    -PipelineConfigPath "<PIPELINE_CONFIG_PATH>"
```

采集 Worker 必须使用当前登录用户的 `-LogonType Interactive` 和 Limited 权限。注册脚本还会拒绝 SYSTEM、`NT AUTHORITY\*`、`NT SERVICE\*` 和非交互身份。审核三个任务的 Principal、Trigger、Action、WorkingDirectory、IgnoreNew 和 restart 设置，不要只看任务名称。

## 8. 启动与验收

在维护窗口重新登录目标用户，或经批准后启动三个固定任务。执行只读检查：

```powershell
powershell -NoProfile -File scripts\windows\check_admin_stack.ps1 `
    -ProjectRoot "<PROJECT_ROOT>" `
    -ConfigPath "<WEB_CONFIG_PATH>" `
    -WebConfigPath "<WEB_CONFIG_PATH>" `
    -CollectorConfigPath "<COLLECTOR_CONFIG_PATH>" `
    -PipelineConfigPath "<PIPELINE_CONFIG_PATH>"
```

验收七个固定键、三个任务状态、两个 legacy 名称、HTTPS、三份配置和微信健康。七键检查不查询 `wechat_ui_lock`，且生产 Web 角色无该表权限，所以七键通过不代表 UI lock 空闲。检查：

- `runtime\logs\admin_web`、`collector_worker`、`pipeline_worker` 按日期生成且不含密码、私钥或 traceback；
- Collector heartbeat 与微信健康在约定周期内更新；
- Web 只返回 clean/analysis、日报和安全运行摘要，不返回 raw、正文或截图内容；
- 故障页面只显示经校验的本机截图路径，该路径只能在采集机本机打开；
- 次日 00:10 后，前一自然日群和文章两类子日报均为 final，compensation all 请求成功，并通过只读 summary 页面或下载复核汇总结果；
- 任务重复实例为 0。

### 8.1 UI lock 独立只读诊断

停止 Collector、启动或恢复 Collector、切换旧手动 CLI、执行 Fake RPA 前，必须由 DBA 或使用 Collector 数据库账号的本机管理员独立执行 `sql\operations\check_wechat_ui_lock.sql`。不得使用 Web 账号，不得给 Web 角色增加 `wechat_ui_lock` 权限。

```powershell
Get-Content -LiteralPath "sql\operations\check_wechat_ui_lock.sql" -Raw |
    & mysql.exe --user "<COLLECTOR_DB_USER>" --password --database "weinsight_prod"
```

`--password` 只触发交互式密码提示，禁止把密码拼入命令或写入手册。查询是单条只读 `SELECT`，不会删除过期行。以数据库返回的 CURRENT_TIMESTAMP（`database_time`）比较 `lease_until`，不要使用采集机本地时钟自行推断：

- `free_no_row`：没有锁记录，可通过 UI lock 门禁；
- `free_expired`：锁记录存在，但 `lease_until <= database_time`，没有有效持有者，可通过门禁；
- `held`：`lease_until > database_time`，仍有有效持有者，必须停止后续停机、启动、恢复、Fake RPA 或手动 CLI 操作，等待协作释放或 lease 到期后重新查询。

每次门禁都要保存 `database_time`、`lease_until`、`ui_lock_state` 和操作时间到变更单，但不得记录数据库密码。空闲判定只能接受 `free_no_row` 或 `free_expired`，不得从七键检查、Web 页面或“进程看起来已退出”推断。

真实采集仍需在 Fake RPA 和单群单公众号人工值守闸门通过后单独批准。

## 9. 回滚路径

四种路径可以独立选择。所有回滚先记录时间、操作人、任务状态、heartbeat，并按 8.1 节独立查询 UI lock；数据库迁移不回退删除数据。

### 回滚一：Web 只读查询

当 Web 写操作不稳定但查询仍可用时，DBA 临时从 `weinsight_web_role` 回收业务控制面 DML，仅保留认证/Session 维持登录所需写权限和所有既有 SELECT。需要回收的对象包括名单、job、target、report request 和 Web 事件 INSERT/UPDATE/DELETE 权限。使用 `SHOW GRANTS` 复核后，只允许查看总览、运行、结果和日报；禁止创建、停止、删除任务或生成日报。

```sql
REVOKE INSERT, UPDATE, DELETE ON `weinsight_prod`.`wechat_group_config` FROM 'weinsight_web_role';
REVOKE INSERT, UPDATE, DELETE ON `weinsight_prod`.`wechat_public_account_config` FROM 'weinsight_web_role';
REVOKE INSERT, UPDATE ON `weinsight_prod`.`wechat_collection_job` FROM 'weinsight_web_role';
REVOKE INSERT ON `weinsight_prod`.`wechat_collection_job_target` FROM 'weinsight_web_role';
REVOKE INSERT ON `weinsight_prod`.`wechat_report_generation_request` FROM 'weinsight_web_role';
REVOKE INSERT ON `weinsight_prod`.`wechat_collection_job_event` FROM 'weinsight_web_role';
SHOW GRANTS FOR '<WEB_DB_USER>'@'<APP_HOST>' USING 'weinsight_web_role';
```

恢复时由 DBA 重新审计并执行 `grant_admin_stack_roles.sql` 中 Web role 的逐表授权，再用三项只读查询和一项受控 Fake 写请求验证。不要为方便恢复授数据库级通配权限。

### 回滚二：暂停 article 单链路

在 Web 中逐个请求停止 article 任务，等待当前目标到达安全检查点并进入 stopped；群任务保持运行。确认 article 不再领取新运行、article UI slice 为 0、核心群等待恢复正常。不要强杀微信或 Collector 进程。

恢复时先安排无其他 UI 采集的受控窗口，执行 `sql\operations\check_wechat_ui_lock.sql` 并得到 `free_no_row` 或 `free_expired`，再通过 `wechat-health` 和单账号 Fake RPA，最后逐个恢复 article 任务；若返回 `held`，不得执行 Fake RPA。公众号频率仍不得低于 10 分钟。

### 回滚三：暂停全部托管采集

先在 Web 对所有 scheduled/active 托管 job 请求停止全部采集，等待 run 和 target-run 进入终态。随后由 DBA 或 Collector 账号执行 `sql\operations\check_wechat_ui_lock.sql`；只有返回 `free_no_row` 或 `free_expired`，才视为 UI lock 已释放并允许停止 `WeInsight-Collector-Worker` 固定任务。返回 `held` 时必须等待后重新查询，不得仅凭七键或 Web 页面继续。协作停止条件全部满足并确认 heartbeat 过期后，Web 可继续查询，Pipeline 可完成已落库的非 UI 后处理；不得启动第二个 Collector 或长期旧 scheduler。

```powershell
Stop-ScheduledTask -TaskName "WeInsight-Collector-Worker"
Get-ScheduledTask -TaskName "WeInsight-Collector-Worker" |
    Select-Object TaskName, State
```

只有主机失控且无法完成协作停止时才允许紧急强停。紧急强停后不得立即重启或运行手动 CLI；必须等待 run lease 和 UI lock lease 过期，核对数据库运行状态并按恢复流程处理遗留实例。

恢复前执行 `check_admin_stack.ps1`，确认两个 legacy 名称不存在且微信健康正常；再独立执行 `sql\operations\check_wechat_ui_lock.sql`。只有返回 `free_no_row` 或 `free_expired` 才能启动唯一 Collector 任务，返回 `held` 时停止恢复并排查持有者和 lease。

### 回滚四：恢复旧手动 CLI

仅在全部托管采集已暂停、Collector heartbeat 已过期，并执行 `sql\operations\check_wechat_ui_lock.sql` 得到 `free_no_row` 或 `free_expired` 后，由管理员在采集机本机人工值守执行旧的 `collect-group-once` 或 `collect-article-once`。返回 `held` 时禁止进入手动模式。手动 CLI 使用 Collector 配置和 Collector 角色凭据，仍必须获取同一 UI lock；不得恢复长期 scheduler，不得与托管 Collector 并行。

```powershell
python -m app.main collect-group-once `
    --config "<COLLECTOR_CONFIG_PATH>" `
    --group-name "<AUTHORIZED_GROUP_NAME>"

python -m app.main collect-article-once `
    --config "<COLLECTOR_CONFIG_PATH>" `
    --account-name "<AUTHORIZED_ACCOUNT_NAME>"
```

完成故障处理后停止手动入口，重新执行 `sql\operations\check_wechat_ui_lock.sql`、Fake RPA、微信健康和七键只读检查，再按回滚三的恢复条件启用托管 Collector。

## 10. 默认管理员密码的剩余风险

首次登录不强制改密，因此默认凭据长期保留会产生局域网内猜测和共享风险。即使系统具备 HTTPS、指定 CIDR 防火墙、登录失败锁定、最小权限角色、默认密码持续提醒和审计日志，仍存在剩余风险。

生产负责人应在首次部署后通过主动改密入口设置组织批准的随机密码并存入 secret manager；若业务决定暂不修改，必须记录风险接受人、期限和补偿控制，且不得扩大 Web 可达网段。

## 11. 变更完成记录

在变更单中记录但不粘贴敏感值：提交号、三任务状态、三角色 `SHOW GRANTS` 审计结论、证书指纹和有效期、防火墙规则名称、七键检查结果、独立 UI lock 诊断结果、00:10 补偿结果、回滚选择和遗留风险。私钥内容、数据库密码、Session token、raw 消息和截图不得进入记录。
