# 公众号订阅号链路启动前设计复核结论

## 1. 复核结论

本次复核结论是：暂不启动公众号/订阅号真实采集，允许进入受控 POC 准备。

理由如下：

1. 现有设计仍满足同一台主机、同一个微信 PC UI 下的串行交错模型：所有微信 UI 操作必须持有 `wechat_ui_lock`，同一时刻只能由微信群链路或公众号/订阅号链路中的一个持有。
2. 当前微信群链路已经完成采集、清洗、分析、日报、失败任务、监控指标、安全回归和长稳脚本增强，具备作为高优先级链路继续试运行的基础。
3. 公众号/订阅号链路已有配置、任务表、进度表、状态机和空窗判断设计，但真实 RPA 操作、文章链接复制、正文解析、失败恢复和监控指标尚未完成实机验收。
4. 生产模板中 `article.enabled=false`，符合“先完成启动前检查，再开启真实采集”的落地顺序。

因此，下一阶段可以准备公众号/订阅号 POC，但不应直接在当前生产模板中打开真实采集开关。

## 2. UI 锁交错规则复核

当前规则仍然成立：

1. 微信群链路优先级高，负责核心群短事务、频繁轮询。
2. 公众号/订阅号链路优先级低，只在低峰执行。
3. 微信 UI 操作必须通过 `wechat_ui_lock` 串行化。
4. 每次 UI 占用必须有超时，超时后释放锁并记录失败。
5. 公众号/订阅号采集不能阻塞核心群超过 `max_core_group_block_seconds`。

生产初版容量边界保持不变：

```text
核心群不超过 5 个
公众号/订阅号不超过 20 个
文章链路每个 UI 分片默认只处理 1 个账号
```

该容量下可以在同一台主机上做交错执行，但不是 UI 并行执行。只要使用同一个微信 PC UI，真实 RPA 操作就必须串行。

## 3. 低峰执行与可中断重试复核

公众号/订阅号链路应保持以下执行模型：

1. 默认执行窗口：公众号/订阅号链路默认在 `07:30-19:30` 内调度。
2. 账号级频率：每个公众号/订阅号默认每小时执行 1 次，不能只按全局批次粗略触发。
3. 当天发布过滤：只采集当天发布的数据，非当天文章不入库、不创建后续处理任务。
4. 去重入库：按 `article_hash` 去重，重复文章不重复创建 `wechat_article_process_task`。
5. 空窗判断：如果距离下一次核心群采集窗口小于 `min_article_ui_window_seconds`，文章链路不获取 UI 锁。
6. 分片执行：每次持有 UI 锁只处理 1 个公众号/订阅号。
7. 可中断重试：每个安全阶段结束后检查核心群是否到期；如果到期，写入 `wechat_article_collect_progress`，释放 UI 锁，后续从中断点恢复。
8. 链接优先：微信 UI 只负责打开订阅号、复制文章链接；正文解析在释放 UI 锁后执行。

不允许的行为：

1. 一次性持有微信 UI 锁处理全部公众号/订阅号。
2. 在鼠标点击、列表加载、剪贴板复制等单个 UI 动作中间强制中断。
3. Playwright 正文解析期间继续持有 `wechat_ui_lock`。
4. 文章链路失败后修改微信群链路 cursor、任务、日志或日报状态。

## 4. 独立链路复核

公众号/订阅号链路必须和微信群链路保持以下隔离：

```text
独立调度
独立入库
独立清洗
独立分析
独立任务表
独立采集日志
独立进度恢复
独立启停开关
```

允许共用：

1. 同一个 MySQL database。
2. 同一个微信 PC UI。
3. 同一张 `wechat_ui_lock`。
4. 同一套配置加载和运行日志基础设施。

不允许共用：

1. 不共用 `wechat_group_process_task` 和 `wechat_article_process_task`。
2. 不共用群游标和文章进度。
3. 不共用失败计数。
4. 不用文章链路补偿命令修改群链路任务，反向也一样。

## 5. 启动真实 POC 的前置条件

如果后续要启动公众号/订阅号真实采集，至少先完成以下前置条件：

1. 明确授权账号清单，生产初版不超过 20 个公众号/订阅号。
2. 新增 article 配置管理与账号启停能力，不直接写死在 RPA 代码中。
3. 完成微信 PC 4.1.8.107 下订阅号入口、账号搜索、文章列表定位、链接复制的 RPA 适配验证。
4. 完成 article 采集日志、截图路径、错误码和 UI 锁持有耗时记录。
5. 完成 `wechat_article_collect_progress` 的实机中断恢复验证。
6. 完成“核心群到期打断文章链路”的集成测试，证明阻塞核心群不超过 `max_core_group_block_seconds`。
7. 完成 Playwright 链接解析链路，并验证正文解析不持有 `wechat_ui_lock`。
8. 完成文章链路安全输出回归，默认不在 CLI、日志和日报中输出大段原文。
9. 完成账号级每小时调度验证，确保每个公众号/订阅号在 `07:30-19:30` 窗口内每小时最多执行 1 次且应执行时不漏调度。
10. 完成当天发布过滤验证，确保非当天文章不会写入 article raw 或创建后续任务。
11. 完成 article 去重验证，确保重复 URL 或重复 `article_hash` 不重复入库。
12. 完成小规模人工值守 POC：先 1 个公众号、每小时 1 次、每次 1 篇当天文章，再扩大到 3 个公众号。
13. 通过回滚检查：关闭 `article.enabled` 后，微信群链路仍可继续运行，文章链路不再占用微信 UI。

## 6. 建议推进顺序

建议下一阶段不要直接进入全量公众号/订阅号采集，而是按以下顺序推进：

```text
1 个授权公众号 fake RPA 验证每小时调度、当天过滤和去重
1 个授权公众号真实 UI 拿当天文章链接
释放 UI 后 Playwright 解析
中断恢复和群优先打断测试
3 个授权公众号人工值守 POC
再评估是否开启最多 20 个账号的小规模试运行
```

## 7. 最终判断

当前不启动公众号/订阅号真实采集。

可以进入受控 POC 准备阶段，目标是先证明文章链路在微信 PC 4.1.8.107 上能稳定完成“低峰拿链接、释放 UI、浏览器解析、失败可恢复、群链路优先”的最小闭环。只有这些前置条件通过后，才建议把生产模板中的 `article.enabled` 从 `false` 改为 `true`。
