# 汇总日报只读聚合层设计

## 1. 背景与现状

当前系统已经形成两条独立链路：

1. 微信群链路：采集、清洗、分析后生成 `wechat_group_daily_report`。
2. 公众号/订阅号链路：采集链接、解析、分析后生成 `wechat_article_daily_report`。

两条链路在调度、任务表、入库、清洗、分析和日报生成上保持隔离。第六阶段后续需要一份面向人工查看的汇总日报，用于把群日报和文章日报放在同一份 Markdown 中做阅读入口，但不能让汇总日报反向影响任一业务链路。

## 2. 目标与非目标

目标：

1. 建立汇总日报只读聚合层。
2. 明确汇总日报只允许读取 `wechat_group_daily_report` 和 `wechat_article_daily_report`。
3. 明确汇总日报可以生成独立 Markdown 文件。
4. 明确汇总日报失败不得回写 group/article 链路状态。
5. 明确汇总日报不得占用微信 PC UI，不获取 `wechat_ui_lock`。

非目标：

1. 不重新采集微信群消息。
2. 不重新采集公众号/订阅号文章链接。
3. 不重新解析文章页面。
4. 不重新执行群消息或文章分析。
5. 不修改 group/article 已生成日报内容。
6. 不在本任务实现汇总日报生成代码，只定义设计边界。

## 3. 影响范围

只读输入：

1. 读取 `wechat_group_daily_report`。
2. 读取 `wechat_article_daily_report`。

允许输出：

1. 生成汇总日报 Markdown 文件，建议目录为 `runtime/reports/summary/YYYY-MM-DD/summary.md`。
2. 后续如需落表，应使用独立汇总日报表，例如 `wechat_summary_daily_report`，不得复用 group/article 源表。

禁止影响：

1. 不得更新 `wechat_group_process_task`。
2. 不得更新 `wechat_article_process_task`。
3. 不得更新群链路游标、采集日志、清洗表、分析表或群日报源表。
4. 不得更新文章链路游标、采集日志、清洗表、分析表或文章日报源表。

硬性约束：不得更新 wechat_group_process_task；不得更新 wechat_article_process_task。

## 4. 数据模型和权限模型

### 4.1 读取模型

汇总日报只读聚合层只读取 `wechat_group_daily_report` 和 `wechat_article_daily_report`。

读取建议：

```text
group_daily_reports = 读取 wechat_group_daily_report
article_daily_reports = 读取 wechat_article_daily_report
summary_markdown = 按日期拼装 group/article 日报摘要和链接入口
```

### 4.2 写入模型

第一版只生成 Markdown 文件，不新增数据库表：

```text
runtime/reports/summary/YYYY-MM-DD/summary.md
```

如果后续需要落库，必须新增独立结果表，不得把汇总状态写回 group/article 链路表。

### 4.3 权限模型

生产环境建议使用独立只读账号：

```text
wechat_summary_reader:
  SELECT wechat_group_daily_report
  SELECT wechat_article_daily_report
```

如需写文件，由应用本地文件权限控制。该账号不授予 group/article 任务表、游标表、采集日志表、清洗表、分析表的 UPDATE 权限。

## 5. 后端实现方案

后续实现建议拆为三层：

1. `SummaryDailyReportQueryRepo`：只读 group/article 日报表。
2. `SummaryDailyReportService`：按日期组装汇总 Markdown。
3. CLI：`summary-daily-report-show/export --date YYYY-MM-DD`。

核心规则：

1. 汇总日报失败不得回写 group/article 链路状态。
2. 汇总日报失败只能返回错误码、错误摘要或写入独立汇总层日志。
3. 汇总日报不得创建、重置或标记 group/article 处理任务。
4. 汇总日报不得回查 raw、clean、analysis 明细表。
5. 汇总日报不得输出文章链接、HTML 或正文原文；文章部分只使用已生成文章日报草稿中的安全内容。

## 6. 前端和 CLI 影响

当前项目暂无独立前端页面。CLI 建议保持只读命令形态：

```powershell
python -m app.main summary-daily-report-show --config config/config.dev.yaml --date 2026-07-06
python -m app.main summary-daily-report-export --config config/config.dev.yaml --date 2026-07-06 --output runtime/reports/summary
```

CLI 输出约束：

1. show 可以输出汇总 Markdown。
2. export 只输出 `export_path`、`bytes_written` 和独立汇总错误键。
3. 不输出 group/article 原始明细字段。
4. 不输出具体文章链接或正文。

## 7. 测试与回归方案

设计阶段回归：

```powershell
pytest tests/test_phase_six_plan_docs.py tests/test_sensitive_output_guard.py -q
```

后续实现阶段建议增加：

1. repo 测试：SQL 只包含 `wechat_group_daily_report` 和 `wechat_article_daily_report`。
2. service 测试：汇总日报可合并两类日报摘要。
3. CLI 测试：show/export 不触发微信 UI、不访问 group/article 任务表。
4. 敏感输出测试：汇总日报命令输出 allowlist 不包含正文或内部明细字段。
5. 隔离测试：汇总日报失败时 group/article 任务状态不变化。

## 8. 风险与分阶段落地建议

风险：

1. 如果汇总日报直接读取明细表，容易扩大敏感数据暴露面。
2. 如果汇总日报复用 group/article 任务表，失败重试会污染原链路状态。
3. 如果汇总日报包含过多正文内容，后续导出文件会变成新的敏感数据副本。

落地建议：

1. 第一阶段只做设计文档和测试约束。
2. 第二阶段实现只读 repo 和 Markdown 组装，不新增数据库表。
3. 第三阶段再评估是否需要独立汇总日报表或独立汇总运行日志。
4. 任何阶段都必须保持：汇总日报失败不得回写 group/article 链路状态。
