# 微信信息采集分析系统落地方案设计

## 1. 背景与现状

本项目目标是建设一套轻量级微信信息采集与分析系统，用于采集指定微信公众号文章和已授权微信群消息，并完成清洗、去重、入库、分析和日报生成。

前置设计已经明确采用 Windows 微信 PC 客户端、RPA/UI 自动化、Python 采集服务、MySQL、规则清洗和批量 AI 分析的单机轻量架构。结合技术可行性分析，本轮落地方案聚焦第一阶段可验证、可运维、可回退的实现，不把普通微信 RPA 采集定义为强可靠消息流。

开发环境数据库使用 deploy 容器中的 MySQL 8.4，已为本项目单独创建开发库：

```yaml
mysql:
  host: 127.0.0.1
  port: 3307
  database: weinsight_dev
  username: weinsight
  password: weinsight_dev
```

微信 PC 客户端版本固定为：

```text
4.1.8.107
```

该版本号必须进入配置、日志和采集批次记录。生产和测试期间禁止自动升级微信客户端；如需升级，应先在测试环境重新完成 RPA 兼容性验证。

## 2. 技术可行性结论

系统可以落地为内部授权场景下的轻量采集分析工具，适合小规模试点和业务辅助分析。数据库、调度、清洗、分析、日报链路技术风险较低；核心风险集中在微信 PC 端 UI 自动化和微信公众号内容采集合规边界。

落地定位：

```text
内部授权信息采集
+ 准实时业务监控
+ 去重入库
+ 脱敏清洗
+ 辅助分析日报
```

不定位为：

```text
强审计归档系统
100% 不漏采消息流
秒级实时系统
面向公众的公众号内容聚合或替代平台
```

## 3. 目标与非目标

### 3.1 目标

1. 固定微信 PC 版本 4.1.8.107，完成稳定 RPA 采集基线。
2. 支持指定微信公众号文章的定时采集，默认每天 08:30 执行。
3. 支持指定授权微信群消息的周期轮询采集。
4. 每个微信群维护独立采集游标，采用回溯、锚点和唯一键去重降低漏采。
5. 原始数据先入库，清洗、脱敏、分析后置。
6. 开发环境使用 deploy 容器内 MySQL 的独立库 `weinsight_dev`。
7. 生产环境预留独立 MySQL 配置，不使用 Docker MySQL 作为生产库。
8. 采集异常必须记录日志、失败原因和截图路径。
9. AI 分析只处理脱敏后的必要字段，不直接发送完整原始群消息。

### 3.2 非目标

1. 不使用 Hook、抓包、逆向、破解微信本地数据库。
2. 不引入 Kafka、Flink、Doris。
3. 不承诺普通微信群消息 100% 不漏采。
4. 不采集私人聊天、非授权群、群成员列表、头像、语音、图片和文件内容。
5. 不做微信公众号内容公开搜索、再分发或替代原平台访问。
6. 第一阶段不建设复杂前端后台，仅提供配置文件、日志、数据库表和必要的命令行验证能力。

## 4. 影响范围

### 4.1 运行环境

采集程序运行在 Windows 10/11 交互式桌面会话中，微信 PC 4.1.8.107 必须保持已登录、窗口可访问、显示缩放固定为 100%。电脑应关闭自动休眠和自动锁屏。

不建议把直接操作微信 UI 的采集进程作为普通 Windows 服务运行。Windows 服务通常运行在 Session 0，与用户桌面隔离，会影响微信窗口识别和鼠标键盘模拟。后续如需服务化，应拆成：

```text
交互式桌面采集进程：负责微信 RPA
后台调度/分析进程：负责任务调度、清洗、分析、日报
```

### 4.2 数据库

第一阶段只影响 `weinsight_dev` 开发库。生产环境需要单独创建生产库和最小权限账号。

### 4.3 文件目录

项目初始目录建议：

```text
WeInsight/
├── app/
├── config/
├── docs/
├── logs/
├── sql/
├── tests/
└── README.md
```

### 4.4 外部系统

1. 微信 PC 客户端：作为 RPA 目标，不做协议层侵入。
2. MySQL：保存配置、原始数据、清洗结果、游标、日志和分析结果。
3. 大模型 API 或私有模型：仅用于脱敏后的批量分析。

### 4.5 生产容量假设

生产初版按以下规模设计：

```text
核心微信群：不超过 5 个
公众号/订阅号：不超过 20 个
部署方式：优先同一台 Windows 主机交错执行
微信 PC：同一个微信 PC 4.1.8.107 客户端
```

在上述规模内，允许采用单机共用微信 UI 的交错执行方案作为生产初版，但必须满足以下前提：

1. 核心群采集优先级始终高于公众号/订阅号采集。
2. 公众号/订阅号采集必须拆成小批次，不允许一次持有微信 UI 锁处理全部 20 个账号。
3. 每个核心群单次 UI 采集耗时需要持续记录，建议 P95 控制在 10 秒以内。
4. 5 个核心群一轮采集建议 P95 控制在 60 秒以内。
5. 公众号/订阅号采集可以延迟、可以中断、可以重试，不能反向挤占核心群采集窗口。

如果核心群数量超过 5 个，或公众号/订阅号超过 20 个，或核心群要求更高实时性，应重新评估拆分采集机或独立桌面。

## 5. 总体架构

推荐采用“双链路隔离 + 基础设施共享”的分层架构。微信群消息采集分析链路和公众号/订阅号信息采集分析链路必须分开设计、分开调度、分开失败处理，任何一条链路失败都不能阻断另一条链路继续运行。

```text
Windows 交互式桌面
│
├── 微信 PC 4.1.8.107
│
├── 微信群消息链路
│   ├── group_collector
│   ├── group_cleaner
│   ├── group_analyzer
│   ├── group_reporter
│   └── group_monitor
│
├── 公众号/订阅号链路
│   ├── article_collector
│   ├── article_cleaner
│   ├── article_analyzer
│   ├── article_reporter
│   └── article_monitor
│
└── MySQL
    ├── group_* 表
    ├── article_* 表
    └── shared_* 基础表
```

核心原则：

1. RPA 只负责操作微信和读取可见信息。
2. 采集层只负责生成结构化原始记录并入库。
3. 清洗层只处理已入库 raw 数据。
4. 分析层只处理 clean 数据和必要上下文。
5. 游标必须在原始数据入库成功后更新。
6. 所有异常先落日志，不吞异常、不静默跳过。
7. 微信群链路和公众号/订阅号链路不能共用任务队列、任务状态、游标状态和失败计数。
8. 两条链路允许共享 MySQL 实例、配置加载、日志框架、通用工具函数，但共享层不得保存某一条业务链路的运行状态。

### 5.1 双链路隔离边界

微信群消息链路负责：

```text
微信群配置
微信群 RPA 打开与回溯
群消息 raw 入库
群消息清洗
群消息商机分析
群消息日报
群消息采集日志与游标
```

公众号/订阅号链路负责：

```text
公众号/订阅号配置
订阅号入口 RPA 操作
文章链接获取
文章 raw 入库
文章清洗摘要
文章主题分析
文章日报
文章采集日志与游标
```

两条链路禁止直接调用对方模块，禁止读取或更新对方任务表，禁止因为对方任务积压、失败、重试而暂停自身调度。

### 5.2 微信 PC UI 资源隔离说明

如果微信群采集和公众号/订阅号采集共用同一个 Windows 桌面和同一个微信 PC 客户端，则 UI 操作天然存在物理争用。若生产要求 UI 操作层也完全互不影响，推荐采用硬隔离：

```text
采集桌面 A / 采集机 A：只运行微信群消息链路
采集桌面 B / 采集机 B：只运行公众号/订阅号链路
```

如果第一阶段只能使用单台机器和单个微信 PC 客户端，则必须采用降级隔离：

```text
两条链路独立调度
+ 独立任务表
+ 独立游标
+ 独立日志
+ 微信 UI 互斥锁
+ 单次 UI 占用超时
+ 微信群链路优先级高于文章链路
```

降级隔离只能保证错误状态、任务积压和数据处理不互相污染，不能保证 UI 操作时间完全互不影响。该限制必须在 POC 验收报告中单独说明。

### 5.3 单机共用微信 UI 调度规则

如果第一阶段共用同一个微信 PC UI，必须执行以下硬性规则：

```text
微信群链路优先
公众号/订阅号链路低峰执行
微信 UI 操作必须加互斥锁
每次 UI 占用必须有超时
超时后释放锁并记录失败
公众号采集不能阻塞核心群超过阈值
```

规则解释：

1. 微信群链路优先：核心群轮询优先级高于公众号/订阅号采集。微信群到达采集窗口时，文章链路不得新抢占微信 UI。
2. 公众号/订阅号链路低峰执行：文章采集优先安排在微信群消息低峰期，默认每天 08:30 触发前需检查核心群是否处于可延迟窗口。
3. 微信 UI 操作必须加互斥锁：所有打开群聊、滚动消息、打开订阅号、复制文章链接等 UI 操作必须先获取 `wechat_ui_lock`。
4. 每次 UI 占用必须有超时：群链路和文章链路分别配置 `rpa_timeout_seconds`，禁止无期限占用微信窗口。
5. 超时后释放锁并记录失败：超时任务必须释放 `wechat_ui_lock`，写入对应链路日志，记录截图路径和错误码。
6. 公众号采集不能阻塞核心群超过阈值：如果文章链路持有 UI 锁导致核心群等待超过 `max_core_group_block_seconds`，文章任务应中断、标记可重试，并释放 UI 锁。

单机共用微信 UI 时，`wechat_ui_lock` 只允许包裹最小 UI 操作区间。数据库读写、文本清洗、AI 分析、日报生成不得放入 UI 锁内。

## 6. 方案路线对比

### 6.1 方案 A：单进程轻量版

一个 Python 进程同时负责调度、RPA、清洗和分析。

优点：

1. 实现最快。
2. 部署简单。
3. 适合第一周 POC。

缺点：

1. RPA 卡住会影响清洗和分析。
2. 错误隔离差。
3. 后续扩展成本较高。

### 6.2 方案 B：采集与后处理拆分

交互式采集进程负责微信 RPA 和 raw 入库；后台任务进程负责清洗、分析和日报。

优点：

1. RPA 风险与后处理隔离。
2. 更容易重试和补偿。
3. 更接近生产落地形态。

缺点：

1. 初始工程量略高。
2. 需要任务表或状态字段协调。

### 6.3 方案 C：双链路硬隔离

微信群消息链路和公众号/订阅号链路分别运行在独立采集桌面、独立进程和独立任务队列中。

优点：

1. 两条链路故障隔离最彻底。
2. 公众号采集不会占用微信群采集窗口。
3. 更符合生产环境“互不影响”的要求。

缺点：

1. 需要更多采集资源或至少两个独立桌面会话。
2. 运维配置略复杂。
3. 如果使用两个微信采集账号，需要额外管理授权和登录状态。

### 6.4 方案 D：企业微信会话存档优先

如果业务允许迁移到企业微信，使用官方会话存档能力获取聊天内容。

优点：

1. 可靠性和合规性更强。
2. 有更明确的服务端数据来源。
3. 更适合审计归档。

缺点：

1. 不适用于当前普通微信群场景。
2. 需要企业微信管理后台、授权、付费和成员/客户同意链路。
3. 迁移成本高。

### 6.5 推荐路线

第一阶段采用方案 C 的逻辑隔离形态。在核心群不超过 5 个、公众号/订阅号不超过 20 个的生产初版规模内，默认采用单机交错执行；如果 POC 显示核心群轮询延迟超过阈值，或后续规模超过该容量，再升级为硬隔离部署。

```text
微信群消息链路先落地
+ group raw 入库
+ group 游标维护
+ group 日志截图
+ group 最小清洗任务

公众号/订阅号链路独立落地
+ article raw 入库
+ article 文章游标维护
+ article 日志截图
+ article 最小清洗任务
```

AI 分析和日报在各自采集链路稳定后分别启用。微信群链路稳定不能自动代表公众号/订阅号链路稳定，反之亦然；两条链路必须分别验收、分别开关、分别回退。

## 7. 数据模型设计

### 7.1 配置表

`wechat_public_account_config` 保存公众号采集配置。

`wechat_group_config` 保存微信群采集配置。

两类配置表独立维护。禁用公众号/订阅号链路不得影响微信群配置生效；禁用微信群链路也不得影响公众号/订阅号定时采集。

第四阶段先落地 `wechat_public_account_config`，用于受控 POC 的账号启停和账号级每小时调度：

```sql
CREATE TABLE wechat_public_account_config (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    account_name VARCHAR(200) NOT NULL COMMENT '公众号或订阅号名称',
    account_type VARCHAR(50) NOT NULL DEFAULT 'subscription' COMMENT 'official/subscription',
    enabled TINYINT DEFAULT 1 COMMENT '是否启用',
    priority INT DEFAULT 5 COMMENT '优先级，数字越小优先级越高',
    poll_interval_minutes INT DEFAULT 60 COMMENT '账号级轮询间隔，默认每小时一次',
    daily_window_start TIME NOT NULL DEFAULT '07:30:00' COMMENT '每日采集窗口开始',
    daily_window_end TIME NOT NULL DEFAULT '19:30:00' COMMENT '每日采集窗口结束',
    max_articles_per_round INT DEFAULT 5 COMMENT '每轮最多采集文章数',
    collect_today_only TINYINT DEFAULT 1 COMMENT '是否只采集当天发布文章',
    dedup_key VARCHAR(50) DEFAULT 'article_hash' COMMENT '去重键',
    last_success_collect_time DATETIME NULL COMMENT '最近成功采集时间',
    remark VARCHAR(500) NULL COMMENT '备注',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_public_account_name (account_name),
    KEY idx_public_account_due (enabled, priority, last_success_collect_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

配置命令边界：

```text
article-account-list:
  只读 wechat_public_account_config
  输出账号名、账号类型、启停、优先级、轮询间隔、每日窗口、每轮文章数和最近成功时间

article-account-upsert:
  创建或更新单个公众号/订阅号账号配置
  默认 poll_interval_minutes=60
  默认 daily_window_start=07:30
  默认 daily_window_end=19:30
  默认 max_articles_per_round=5
  默认 collect_today_only=1
  默认 dedup_key=article_hash

article-account-disable:
  禁用单个公众号/订阅号账号配置
```

以上命令不占用微信 UI，不获取 `wechat_ui_lock`，不读写 `wechat_group_config`、`wechat_group_process_task`、微信群 cursor、微信群采集日志和群日报。

建议新增以下字段：

```sql
collect_account VARCHAR(100) COMMENT '采集微信账号标识',
wechat_pc_version VARCHAR(50) COMMENT '验证通过的微信PC版本',
last_validate_time DATETIME NULL COMMENT '最近一次人工或自动验证时间'
```

### 7.2 原始消息表

`wechat_group_msg_raw` 保存微信群原始消息。

建议在原设计基础上补充：

```sql
collect_account VARCHAR(100) COMMENT '采集微信账号标识',
wechat_pc_version VARCHAR(50) COMMENT '采集时微信PC版本',
hash_version VARCHAR(20) DEFAULT 'v1' COMMENT '消息哈希规则版本',
first_seen_time DATETIME NULL COMMENT '首次采集看到该消息的时间',
last_seen_time DATETIME NULL COMMENT '最近一次采集看到该消息的时间',
observed_order INT NULL COMMENT '本轮采集中消息的可见顺序',
parse_status VARCHAR(50) DEFAULT 'parsed' COMMENT '消息解析状态',
raw_payload_json JSON NULL COMMENT 'RPA原始解析结构'
```

第一版消息唯一键：

```text
msg_hash = sha256(
  group_name
  + sender_name
  + msg_time_display
  + msg_content
  + msg_type
)
```

后续如果 RPA 能读取更稳定的消息节点属性，可以升级 `hash_version`。

### 7.3 清洗消息表

`wechat_group_msg_clean` 保存脱敏后的业务字段。

重点约束：

1. 发送人只保留 hash 或脱敏名。
2. 手机号、微信号、地址等敏感字段必须脱敏。
3. AI 分析默认只读取 clean 表。
4. clean 表不保存 raw 原文；如需排查原文，只能在授权范围内从 raw 表按 hash 回查。

第一版 clean 表字段：

```sql
CREATE TABLE wechat_group_msg_clean (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    msg_hash VARCHAR(64) NOT NULL COMMENT '消息唯一hash，与raw表一致',
    group_name VARCHAR(200) NOT NULL COMMENT '微信群名称',
    sender_hash VARCHAR(64) NULL COMMENT '发送人脱敏hash',
    sender_display VARCHAR(200) NULL COMMENT '发送人脱敏展示名',
    msg_time_display VARCHAR(100) NULL COMMENT '微信界面显示时间',
    msg_time_inferred DATETIME NULL COMMENT '推断消息时间',
    msg_type VARCHAR(50) NULL COMMENT '消息类型',
    clean_content TEXT NULL COMMENT '脱敏后的消息内容',
    content_length INT DEFAULT 0 COMMENT '脱敏后内容长度',
    is_empty TINYINT DEFAULT 0 COMMENT '是否空消息或无业务内容',
    has_phone TINYINT DEFAULT 0 COMMENT '原文是否包含手机号',
    has_wechat_id TINYINT DEFAULT 0 COMMENT '原文是否包含微信号',
    clean_version VARCHAR(20) DEFAULT 'v1' COMMENT '清洗规则版本',
```

第一版清洗规则：


6. 每个账号分片必须先获取 `wechat_ui_lock(owner_pipeline='article')`；成功、失败或异常都必须释放锁。
7. 调度结果写入 `wechat_article_collect_log`，成功记录 `link_count` 和 `insert_count`，锁超时记录 `WECHAT_UI_LOCK_TIMEOUT`，RPA 异常记录 `WECHAT_ARTICLE_RPA_ERROR` 和截图路径。
8. Task 28 不启动真实公众号/订阅号采集，不解析正文，不注册 Windows 计划任务，不读写微信群配置、微信群任务、微信群日志或群日报。

第四阶段 Task 29 增加真实 UI 拿链接 POC，仍然是人工触发的受控命令，不进入后台自动调度：

1. 命令名为 `collect-article-once`，必须显式传入 `--account-name`，禁止不带账号名自动扫描全部公众号/订阅号。
2. 命令只构造 1 个 `ArticlePollingTarget`，单次最多采集 `--max-articles-per-round` 指定数量，默认沿用配置中的 `max_articles_per_account`。
3. 命令执行前做微信健康检查，并通过 `ArticlePollingRunner` 获取 `wechat_ui_lock(owner_pipeline='article')`；拿到链接或失败后立即释放锁。
4. 真实 UI 适配器 `WxautoArticleRpaClient` 先采用最小能力：打开指定公众号/订阅号会话，从当前可见消息或卡片文本中提取 `mp.weixin.qq.com/s/` 文章链接。
5. Task 29 只验证“打开指定账号并拿到链接”的最小 POC；完整发布时间、标题和正文解析放到 Task 30 的 Playwright 链路中完成。
6. 失败时写入 `wechat_article_collect_log`，错误码为 `WECHAT_ARTICLE_RPA_ERROR`，并保存截图路径；不做长时间重试，不阻塞微信群链路。
7. CLI 输出只包含账号名、attempt/success/failed/lock timeout 等计数，不输出文章链接、正文或 raw 字段名。

第四阶段 Task 30 增加 Playwright 文章解析链路：

1. `parse-article-once` 只消费 `wechat_article_process_task(task_type='clean_article')` 的 pending 任务，不获取 `wechat_ui_lock`，不打开微信 UI。
2. 解析器使用浏览器打开 `wechat_article_raw.article_url`，提取标题、发布时间、作者、摘要和正文长度。
3. 解析结果写入 `wechat_article_clean`，不保存 HTML 或正文原文。
4. 解析成功后创建 `wechat_article_process_task(task_type='analyze_article', ref_type='article')`，并将原 `clean_article` 任务标记为 success。
5. 解析失败只更新 article 任务的 retry/status/error，不修改微信群配置、任务、游标、日志或日报。
6. CLI 输出只包含 read/success/failed 计数和错误摘要，不输出文章链接、HTML、正文或内部正文列名。

公众号/订阅号文章分析采用中间方案：运行时临时读取正文，落库只保存摘要和结构化特征，不保存正文或 HTML。正文允许在 Playwright 解析器或 article 分析服务的内存变量中短暂存在，用于生成摘要、主题标签、关键词命中和正文长度；处理完成后立即丢弃，不写入 MySQL、日志、截图、CLI 输出、日报或汇总日报。

第四阶段 Task 31 增加 article 中断恢复与群优先验证：

1. article UI 分片的安全检查点为 `open_account`、`copy_links`、`save_links` 阶段结束后；不得在单个 UI 动作中间强行中断。
2. 检查点发现核心群已到期时，article 链路必须写入 `wechat_article_collect_progress(status='interrupted')`，记录 `crawl_date`、`account_name`、`stage` 和 `last_article_url`。
3. 中断后 runner 必须释放 `wechat_ui_lock`，并写入 `wechat_article_collect_log(status='interrupted', error_code='ARTICLE_INTERRUPTED_FOR_CORE_GROUP')`。
4. 恢复时先读取 `wechat_article_collect_progress`，如果存在同账号同日期 interrupted 记录，则从记录的 `stage` 和 `last_article_url` 后继续，不从头处理已保存链接。
5. `max_core_group_block_seconds` 是 article 检查点允许核心群等待的最大阈值；只要核心群到期，article 在最近一个安全检查点中断，测试需覆盖阻塞时间不超过该阈值。
6. 中断恢复逻辑只读写 article progress、article log 和 article raw，不访问微信群任务、游标、日志或日报。

第四阶段 Task 32 增加 article 运行监控指标和安全回归：

1. 命令名为 `article-runtime-metrics`，默认查看最近 24 小时，可通过 `--hours` 指定窗口。
2. 指标只读 `wechat_public_account_config`、`wechat_article_collect_log` 和 `wechat_article_process_task`，不获取 `wechat_ui_lock`，不打开微信 UI。
3. 输出账号总数、启用账号数、窗口内采集成功数、失败数、跳过数、采集总数、当前 article 任务积压和最近错误摘要。
4. 跳过数包含 `skipped` 和 `interrupted`，用于观察低优先级 article 链路被核心群打断或主动跳过的情况。
5. 任务积压按 `task_type/status` 聚合，只统计 article 任务表中的 `pending`、`running` 和 `failed` 状态。
6. 最近错误摘要只输出错误码和截断后的错误说明，不输出文章链接、文章正文、HTML 或内部正文列名。
7. 该命令不得读取微信群配置、微信群任务、微信群日志、群日报或任意群消息内容表，避免监控命令反向耦合两条链路。
8. README 和运行手册必须补充 `article-runtime-metrics` 巡检命令，安全输出白名单必须覆盖该命令。

## 8. 权限模型

### 8.1 开发环境

开发环境使用 `weinsight` 账号访问 `weinsight_dev`，允许建表、读写和调试。

### 8.2 生产环境

生产环境至少拆分五类账号：

1. `wechat_group_collector`：读取群配置，写入 group raw、group cursor、group collect_log、group process_task。
2. `wechat_article_collector`：读取公众号/订阅号配置，写入 article raw、article cursor、article collect_log、article process_task。
3. `wechat_group_analyzer`：读取 group raw 和 group clean，写入 group clean、group analysis。
4. `wechat_article_analyzer`：读取 article raw 和 article clean，写入 article clean、article analysis。
5. `wechat_reader`：只读取 clean 和 analysis，不允许读取 raw。

raw 表包含原始群消息，默认只允许采集程序、分析程序和管理员访问。

权限禁止跨链路授权。例如 `wechat_article_collector` 不得更新 `wechat_group_collect_cursor`，`wechat_group_analyzer` 不得消费 `wechat_article_process_task`。

### 9.1 配置管理

```text
config/config.prod.example.yaml
config/config.prod.yaml
```

开发默认密码可写入本地 `.env`，但不提交生产密码。

`pipelines.article.browser_executable_path` 用于文章解析阶段的 Playwright 浏览器选择，不占用微信 UI。默认值 `auto` 表示优先复用本机已存在的 Playwright Chromium 缓存目录，例如 `chromium-*` 下的 `chrome.exe`；找不到可复用浏览器时回退到 Playwright 默认 bundled browser 解析。也可配置为明确的 `chrome.exe` 绝对路径，显式路径不存在时应阻断解析并返回清晰错误，避免静默使用错误浏览器。

生产配置模板要求：

1. `config/config.prod.example.yaml` 可被 `load_config` 和 `check-config` 解析。
2. 模板允许写入非敏感示例值，例如 MySQL host、database、username，但 `mysql.password` 必须使用 `${WEINSIGHT_MYSQL_PASSWORD}`。
3. 模板必须明确 `runtime.log_dir`、`runtime.screenshot_dir`、`runtime.report_dir`，默认仍使用 `runtime/logs`、`runtime/screenshots`、`runtime/reports`，便于脚本和运维手册统一。
4. 生产模板必须保持核心群上限 `core_group_limit <= 5`、公众号/订阅号上限 `account_limit <= 20`、`max_accounts_per_ui_slice = 1`。
5. 生产模板不得包含开发库名 `weinsight_dev`、开发密码 `weinsight_dev`、Docker 容器名或任何真实生产密码。

`pipelines.group.enabled` 和 `pipelines.article.enabled` 必须独立生效。关闭文章链路不能停止群消息采集，关闭群消息链路也不能停止文章采集。

`pipelines.ui_resource.max_core_group_block_seconds` 用于约束公众号/订阅号采集对核心群采集的阻塞时间。超过阈值时，文章链路必须主动释放微信 UI 锁并记录可重试失败。

`pipelines.article.max_accounts_per_ui_slice` 默认设置为 1，表示文章链路每次拿到微信 UI 锁只处理 1 个公众号/订阅号，然后释放锁并检查核心群是否到期。即使配置了 20 个公众号/订阅号，也不能在一次 UI 锁持有周期内连续处理全部账号。

### 9.2 数据库初始化与升级脚本

数据库脚本按用途分层管理：

```text
sql/init.sql
sql/migrations/
```

`sql/init.sql` 只用于全新数据库初始化，必须包含当前版本完整表结构。它可以使用 `CREATE TABLE IF NOT EXISTS`，但不负责给历史库补列或修改已有表。

`sql/migrations/` 保存从某个已部署版本升级到新版本所需的增量 DDL。迁移文件命名规范：

```text
YYYYMMDD_NNN_简短英文说明.sql
```

约束：

1. `YYYYMMDD` 使用提交或任务落地日期。
2. `NNN` 从 `001` 开始递增，保证同一天内按文件名排序即可执行。
3. 迁移脚本必须幂等；新增列需要先查 `information_schema.COLUMNS`，不存在才执行 `ALTER TABLE`。
4. 迁移脚本禁止 `DROP TABLE`、`TRUNCATE TABLE`、无条件删除数据或重建业务表。
5. 增量 DDL 必须和 `sql/init.sql` 保持一致；新库直接执行 `init.sql`，旧库按文件名顺序执行未应用迁移。
6. 开发库辅助执行命令只写在 README/运行手册，不和生产迁移脚本混在一起。

当前第一批迁移用于补齐 Task 20 的日报质量字段：

```text
sql/migrations/20260703_001_add_group_analysis_quality_fields.sql
```

RPA 依赖必须按微信大版本选择：

1. 微信 3.9 系列才允许使用 `wxauto` 开源包。
2. 微信 4.x 系列不得直接使用 `wxauto` 3.9 适配层，因为该包默认查找 `WeChatMainWndForPC` 窗口类，和 4.x 的 Qt/mmui 窗口结构不一致。
3. 微信 4.x POC 优先尝试 `wxauto4` 或 `wxautox4` 适配层；如果目标版本高于适配层明确兼容范围，启动检查必须给出 `WECHAT_RPA_ADAPTER_UNSUPPORTED`，不进入真实采集。
4. 如果 4.1.10.31 在当前主机只暴露 `Qt51514QWindowIcon` 且 UIA/Win32 均无法遍历搜索框和消息列表，则该主机只能做键鼠级探索，不作为稳定 POC 验收环境。

历史开发机验证结论：

```text
微信 PC 版本：4.1.10.31
已安装 wxauto 包版本目标：3.9.11.17
wxauto 查找窗口类：WeChatMainWndForPC
实际可见主窗口类：Qt51514QWindowIcon
结果：wxauto 初始化失败，底层错误为 SetWindowPos 无效窗口句柄
已安装 wxauto4 版本：41.1.2
wxauto4 初始化结果：未找到已登录的客户端主窗口
Win32 子窗口枚举结果：当前主窗口未暴露 wxauto4 期望的 mmui::MainWindow 子窗口
```

因此第二阶段真实 RPA POC 必须先完成适配器可用性检查，再进入“打开测试群”和“读取消息”验证。

如果微信 PC 高版本不能降级，则优先尝试恢复 UIA 控件树，而不是直接进入 OCR 路线。恢复流程必须可回退：

1. 记录基线：显示缩放、采集进程权限、微信进程路径、主窗口类、UIA 树是否存在 `mmui::MainWindow` 或 `消息` 列表。
2. 保持显示缩放 100%，避免多屏混合 DPI；采集进程和微信进程权限保持一致。
3. 使用 Qt 软件渲染环境变量重启微信：

```text
QT_ANGLE_PLATFORM=software
QT_OPENGL=software
QT_QUICK_BACKEND=software
QSG_RHI_BACKEND=software
```

4. 重启后重新枚举顶层窗口和 UIA 树。如果能看到 `mmui::MainWindow` 或 `ListControl(Name="消息")`，则走 `pywinauto`/`uiautomation` 自研适配器；如果仍只看到 `Qt51514QWindowIcon`，则记录失败并进入剪贴板探针或 OCR 键鼠路线。
5. 任何环境调整都不得跳过 `wechat_ui_lock`、超时释放和截图留证规则。

如果采用单机降级隔离，两个 RPA 客户端必须通过 `wechat_ui_lock` 互斥访问微信 UI。UI 锁只保护窗口操作，不允许把数据库任务执行、清洗、分析放进锁内。

`wechat_ui_lock` 必须记录以下运行信息：

```text
lock_owner_pipeline
lock_owner_task_id
lock_acquire_time
lock_heartbeat_time
lock_expire_time
lock_release_time
lock_wait_seconds
lock_lease_seconds
stale_recovered_by
```

`wechat_ui_lock` 必须采用租约机制。持锁进程需要按 `lock_heartbeat_interval_seconds` 刷新心跳和过期时间；如果锁超过 `stale_lock_recover_seconds` 未刷新，调度器可以判定为僵死锁，先记录 `stale_lock_recovered_count`，再接管或释放该锁。僵死锁恢复必须写入对应链路日志，避免静默覆盖真实故障。

当文章链路等待或持有 UI 锁期间发现核心群采集已到期，且核心群预计等待时间超过 `max_core_group_block_seconds` 时，文章链路必须中断当前 UI 操作，释放锁并写入 `wechat_article_collect_log`，错误码建议为 `ARTICLE_INTERRUPTED_BY_GROUP_PRIORITY`。

当任一链路 UI 操作超过自身 `rpa_timeout_seconds` 时，必须释放锁、保存截图、写入对应链路日志，错误码建议为 `WECHAT_UI_LOCK_TIMEOUT` 或 `WECHAT_RPA_TIMEOUT`。

### 9.4 双链路调度隔离

运行入口拆分为两个独立调度器：

```text
group_scheduler
article_scheduler
```

两个调度器分别读取自己的配置、任务表、游标和日志表。一个调度器异常退出时，另一个调度器必须继续运行。

推荐进程形态：

```text
python -m app.group_pipeline
python -m app.article_pipeline
```

开发阶段可由一个 supervisor 拉起两个进程；生产阶段建议分别注册或托管，便于单独重启和单独回滚。

### 9.5 微信群采集

微信群链路定时轮询以 `wechat_group_config` 为唯一群配置来源。每轮调度只选择 `enabled=1` 且到期的核心群，按 `priority ASC`、上次成功采集时间 ASC 排序，单轮最多处理 `pipelines.group.max_group_per_round` 个群；生产初版核心群总数不超过 5 个。

采集流程：

```text
读取 due 群配置
生成 batch_id
获取 wechat_ui_lock
打开微信群
读取当前可见消息
按配置回溯 N 屏
查找 last_msg_hash 锚点
找不到则进入额外回溯
生成 msg_hash
事务写入 group raw 表、group 任务表、group 日志表、group 游标表
释放 wechat_ui_lock
```

事务边界：

```text
BEGIN
INSERT raw
INSERT group_process_task
UPDATE cursor
INSERT group_collect_log
COMMIT
```

只有 raw 入库成功后才允许更新游标。

失败路径必须遵循以下顺序：

```text
捕获异常
如果已持有 UI 锁，先保存截图
写入 failed collect_log，包含 error_code、error_msg、screenshot_path
释放 wechat_ui_lock
更新 cursor 连续失败计数和 error_msg
```

锁等待超过 `pipelines.ui_resource.lock_acquire_timeout_seconds` 时，本轮群采集记为 `WECHAT_UI_LOCK_TIMEOUT`，不打开微信窗口。RPA 操作超过 `pipelines.group.rpa_timeout_seconds` 时，本轮记为 `WECHAT_RPA_TIMEOUT`，保存截图并释放锁。第一阶段 Python POC 先采用同步单进程轮询；后续如果拆分多进程，必须复用同一张 `wechat_ui_lock`。

截图保存目录按日期分层：

```text
runtime/screenshots/group/YYYYMMDD/<batch_id>.png
```

截图文件只用于故障排查，不进入 AI 分析链路，不对外分发。

### 9.6 公众号/订阅号采集

公众号/订阅号链路默认可执行窗口为 07:30-19:30。第一阶段只支持配置中明确指定的公众号或订阅号，每个账号默认每小时到期执行 1 次，每次默认采集最近 1 到 5 篇文章。

文章链路调度不再只有单一 `crawl_time`，而是采用账号级到期模型：

```text
if now not in 07:30-19:30:
    defer article account
if account_last_success_at within 60 minutes:
    defer article account
if next_core_group_due_in_seconds < min_article_ui_window_seconds:
    defer article account
else:
    acquire wechat_ui_lock and process one account slice
```

`crawl_time` 仅表示默认窗口起点，默认值为 `07:30`。真实调度以 `low_peak_windows` 和 `account_poll_interval_minutes` 为准。

采集范围必须限制为当天发布的数据。文章列表中如果出现非当天文章，应停止继续向历史页回溯；如果发布时间无法识别，本轮该文章应记为 skipped 或 failed，不能按当天文章入库。

文章去重必须在入库前执行。默认去重键为 `article_hash`，由账号名、标题、发布时间和文章 URL 生成；后续 article raw 表落地时必须对 `article_hash` 建唯一约束，重复文章只更新采集日志和统计，不重复创建 `wechat_article_process_task`。

在生产初版上限 20 个公众号/订阅号、每个 UI 分片 30 秒的配置下，文章链路理论 UI 占用约 10 分钟/小时。若中途被核心群打断，账号级到期任务应保留 pending 状态并在下一次可用窗口内继续补跑；不得因为一次打断直接跳过该账号当小时的采集。

生产初版公众号/订阅号数量不超过 20 个。文章链路必须按账号分片执行，每个 UI 分片默认只处理 1 个账号：

```text
获取 wechat_ui_lock
处理 1 个公众号/订阅号
释放 wechat_ui_lock
检查核心群是否到期
若核心群到期，暂停文章链路并让出 UI
若核心群未到期，继续下一个公众号/订阅号
```

禁止文章链路一次性持有 UI 锁连续处理全部公众号/订阅号。

文章链路必须按状态机执行，只允许在安全检查点之间被核心群打断：

```text
open_account
list_articles
copy_links
save_links
release_ui
parse_article_in_browser
write_article_raw
done
```

安全检查点为每个阶段结束后。禁止在鼠标点击、列表加载、剪贴板复制等单个 UI 动作中间强行中断；如果核心群到期，应在当前阶段完成后立即释放 `wechat_ui_lock`，将进度写入 `wechat_article_collect_progress`，再让群链路执行。

文章链路必须“先拿链接，后解析”：

1. 微信 UI 只用于打开订阅号、定位文章列表、复制文章链接。
2. 链接保存到数据库后立即释放 `wechat_ui_lock`。
3. 正文解析、HTML 清洗、摘要提取交给 Playwright/浏览器执行，不再占用微信 UI。
4. Playwright 解析失败只影响文章链路，不允许重新占用微信 UI 反复尝试，除非进入下一轮文章采集任务。

文章链路启动前还必须做空窗判断：

```text
if next_core_group_due_in_seconds < min_article_ui_window_seconds:
    defer article task
```

如果距离下一次核心群采集窗口过近，文章链路不应获取微信 UI 锁，而是延后到下一次调度。

公众号/订阅号链路启动前检查：

本阶段复核结论见 `docs/design/公众号订阅号链路启动前设计复核结论.md`。当前结论是：暂不启动公众号/订阅号真实采集，允许进入受控 POC 准备。

真实 POC 启动前必须满足：

1. 授权账号清单已确认，生产初版公众号/订阅号不超过 20 个。
2. `article.enabled` 仍默认关闭，只有人工值守 POC 前才允许临时开启。
3. 文章链路每次 UI 分片只处理 1 个账号，并在分片结束后释放 `wechat_ui_lock`。
4. 核心群到期时，文章链路能在安全检查点中断、记录进度并释放 UI 锁。
5. `wechat_article_collect_progress` 能支持从中断阶段恢复，不从头扫描全部公众号/订阅号。
6. Playwright 正文解析必须在释放微信 UI 后执行。
7. 文章链路失败不得修改微信群链路任务、游标、日志和日报状态。
8. CLI、日志和日报输出通过敏感输出回归，不暴露正文原文或内部正文列名。

采集流程：

```text
打开订阅号入口
进入指定公众号
复制最新文章链接
Playwright 打开链接
解析标题、作者、发布时间、正文
生成 article_hash
写入 article_raw
生成 article_process_task
```

合规边界：

1. 只采集公司明确授权关注和分析的公众号。
2. 不绕过登录、验证码、访问限制或反爬机制。
3. 不将文章内容公开再分发。
4. 日报中优先保存摘要、链接和出处，不复制大段原文。

### 9.7 清洗与分析

群消息清洗任务只从 `wechat_group_process_task` 读取 pending 任务，只写入 `wechat_group_msg_clean` 和群分析表。

公众号/订阅号清洗任务只从 `wechat_article_process_task` 读取 pending 任务，只写入 `wechat_article_clean` 和文章分析表。文章分析可在运行时临时读取正文，落库只保存摘要和结构化特征，不保存正文或 HTML；正文不得进入日志、CLI 输出、日报或汇总日报。

群消息清洗包括：

1. 空白和换行归一化。
2. 系统消息过滤。
3. 手机号、微信号脱敏。
4. 供需、报价、地区、品类、数量识别。
5. 写入 clean 表。

AI 分析任务只读取 clean 表和必要上下文。默认配置：

```yaml
ai:
  enabled: false
  batch_size: 50
  send_raw_content_to_llm: false
```

第一阶段 AI 默认关闭。后续启用时也必须按链路分别启用：

```yaml
ai:
  group_enabled: false
  article_enabled: false
```

群消息 AI 分析失败不得重试文章 AI 任务，文章 AI 分析失败也不得改变群消息任务状态。

## 10. 前端影响

第一阶段不建设 Web 前端。

必要的人机界面为：

1. YAML 配置文件。
2. MySQL 表数据。
3. 运行日志。
4. 异常截图目录。
5. 命令行健康检查。

后续如需前端后台，建议优先提供以下能力：

1. 群配置管理和公众号/订阅号配置管理分区展示。
2. 微信群链路状态看板和公众号/订阅号链路状态看板分开展示。
3. 两条链路的失败批次和截图分开查看。
4. 群消息清洗结果和文章清洗结果分开检索。
5. 群日报、文章日报和最终汇总日报分开查看和导出。

前端不得提供一个会同时启停两条链路的模糊开关。所有启停、重试、清空任务、重跑日报操作必须明确链路类型。

9. 如果窗口类为 `Qt51514QWindowIcon` 且控件树不可遍历，采集命令必须失败在前置检查阶段，并给出可读错误，不允许继续执行真实采集。

### 11.4 POC 验收测试

第一阶段建议设置 2 周 POC。

验收指标：

| 验证项 | 验收标准 |
| --- | --- |
| 测试群采集 | 连续 2 小时受控发送 200 条文本消息，捕获率不低于 98% |
| 去重 | 重复轮询不重复入库 |
| 游标 | 入库失败时不更新游标 |
| 异常处理 | 微信退出、未登录、窗口遮挡时有日志和截图 |
| 公众号 | 3 个指定公众号各采集最近 3 篇文章 |
| 生产规模模拟 | 5 个核心群和 20 个公众号/订阅号配置下，单机交错执行无状态互相污染 |
| 群采集耗时 | 单核心群 UI 采集 P95 不超过 10 秒，5 个核心群一轮 P95 不超过 60 秒 |
| 文章分片 | 文章链路每个 UI 分片默认只处理 1 个公众号/订阅号，分片后释放锁 |
| 脱敏 | clean 表不出现完整手机号和微信号 |
| AI 安全 | 默认不向外部模型发送 raw 原文 |
| 链路隔离 | 群链路失败不影响文章链路，文章链路失败不影响群链路 |
| UI 互斥 | 所有微信 UI 操作均经过 `wechat_ui_lock`，锁等待和持有耗时可追踪 |
| 群优先 | 公众号/订阅号采集阻塞核心群不超过 `max_core_group_block_seconds` |
| UI 超时 | 任一 UI 操作超时后释放锁、记录失败、保存截图 |
| 锁租约 | 持锁进程异常退出后，过期锁可被识别和恢复 |
| 文章状态机 | 文章链路只在阶段边界被打断，不在单个 UI 动作中间强制中断 |
| 链接优先 | 文章链路拿到链接后释放微信 UI，正文解析不占用 `wechat_ui_lock` |
| 进度恢复 | 文章链路中断后可从 `wechat_article_collect_progress` 继续 |
| 空窗判断 | 距离核心群下一次采集过近时，文章链路延后而不是抢锁 |

## 12. 运维与监控

### 12.1 日志

日志分为：

```text
logs/app/
logs/screenshots/
```

日志中禁止打印完整原始消息内容，只允许打印 hash、群名、批次、数量、状态和截断后的摘要。

### 12.2 健康检查

健康检查包括：

1. MySQL 连接状态。
2. 微信进程是否存在。
3. 微信 PC 版本是否为 4.1.8.107。
4. 微信是否已登录。
5. 微信群链路最近一次成功采集时间。
6. 公众号/订阅号链路最近一次成功采集时间。
7. 微信群链路连续失败次数。
8. 公众号/订阅号链路连续失败次数。
9. 两条链路各自的 pending/running/failed 任务数量。
10. `wechat_ui_lock` 当前持有者、持有时长、租约过期时间。
11. 群链路 UI 等待时间 `group_ui_wait_seconds`。
12. 文章链路 UI 等待时间 `article_ui_wait_seconds`。
13. UI 锁持有时间 `ui_lock_hold_seconds`。
14. 文章链路被核心群打断次数 `article_interrupted_count`。
15. 核心群一轮采集耗时 `group_round_duration_seconds`。
16. 僵死锁恢复次数 `stale_lock_recovered_count`。

### 12.3 告警条件

第一阶段可先写入日志和数据库，后续再接入企业微信、邮件或短信告警。

建议告警条件：

1. 微信未登录超过 5 分钟。
2. 任一核心群连续失败 3 次。
3. 任一核心群超过 10 分钟无成功采集。
4. 找不到锚点并标记 `possible_lost = 1`。
5. MySQL 连接失败。
6. 公众号/订阅号链路连续失败 3 次。
7. 任一链路任务积压超过阈值。
8. 公众号/订阅号链路阻塞核心群超过 `max_core_group_block_seconds`。
9. 任一链路持有 `wechat_ui_lock` 超过自身 `rpa_timeout_seconds`。
10. `wechat_ui_lock` 超过 `stale_lock_recover_seconds` 未刷新心跳。
11. 文章链路连续被核心群打断超过阈值，导致当天公众号/订阅号采集无法完成。
12. Playwright 解析积压超过阈值，但不得因此抢占微信 UI。

告警必须带上 `pipeline=group` 或 `pipeline=article`，避免排查时混淆。

## 13. 风险与应对

| 风险 | 等级 | 应对 |
| --- | --- | --- |
| 微信 UI 结构变化 | 高 | 固定 4.1.8.107，升级前跑 RPA 回归 |
| 微信 4.1.10.31 超出当前 RPA 适配器明确兼容范围 | 高 | 启动前校验依赖和窗口类；优先验证 `wxauto4`/`wxautox4` 对 4.1.10.31 的支持；不兼容时阻断真实采集并保留 fake RPA 和数据库链路 |
| 电脑锁屏或休眠 | 高 | 关闭锁屏休眠，采集进程运行在交互式桌面 |
| 普通群消息漏采 | 高 | 回溯、深度回溯、锚点、唯一键、possible_lost 标记 |
| 微信掉线 | 高 | 登录状态检测、阻断采集、截图、告警 |
| 公众号采集合规风险 | 高 | 只采授权账号，不突破限制，不公开再分发 |
| AI 数据泄露 | 高 | 默认关闭 AI，启用前只发送脱敏字段 |
| RPA 卡死影响后处理 | 中 | 采集和后处理拆分，任务表解耦 |
| 两条链路互相影响 | 高 | 独立进程、独立调度、独立任务表、独立日志；生产优先硬隔离采集桌面 |
| 单机共用微信 UI 导致互斥等待 | 中 | 在核心群不超过 5 个、公众号/订阅号不超过 20 个时允许生产初版交错执行；超过容量或实时性不达标时拆采集机或桌面 |
| 文章链路一次占用 UI 过久 | 高 | 按公众号/订阅号分片执行，默认每次 UI 锁只处理 1 个账号，分片间检查核心群是否到期 |
| UI 锁僵死 | 高 | `wechat_ui_lock` 使用租约、心跳和过期接管机制，恢复动作写入日志 |
| 文章链路中断后重复扫号 | 中 | 使用 `wechat_article_collect_progress` 保存账号阶段和最后文章链接，恢复时从中断点继续 |
| 文章正文解析占用微信 UI | 中 | 微信 UI 只负责拿链接，Playwright 正文解析必须在释放 UI 锁后执行 |
| MySQL 开发库误用为生产 | 中 | env 明确区分，生产配置独立，禁止 Docker MySQL 存生产数据 |
| 重复消息误判 | 中 | hash_version、raw_payload_json、first_seen/last_seen 辅助排查 |

## 15. 第一阶段推荐验收口径

第一阶段只验收系统是否具备继续投入价值，不验收完整生产能力。

通过条件：

1. 微信 PC 4.1.8.107 下 RPA 可稳定读取测试群文本消息。
2. 原始消息可去重入库。
3. 游标机制在正常和异常场景下表现正确。
4. 异常日志和截图可用于排查。
5. 开发库、配置、日志和测试脚本可复现。
6. 微信群链路和公众号/订阅号链路具备独立配置、独立任务表、独立日志和独立启停能力。
7. 任一链路失败不会修改另一链路的任务状态、游标状态和失败计数。
8. 单机共用微信 UI 时，公众号/订阅号采集不会阻塞核心群超过配置阈值，超时后能释放锁并记录失败。
9. `wechat_ui_lock` 具备租约和僵死锁恢复能力。
10. 文章链路能在被核心群打断后从采集进度表恢复。
11. 文章正文解析不占用微信 UI 锁。

不通过条件：

1. 受控测试捕获率低于 98%。
2. 微信窗口或版本检测不可控。
3. 锁屏、掉线、窗口遮挡后没有明确失败状态。
4. 重复消息大量入库。
5. raw 数据未经授权被导出或发送到外部模型。
6. 群任务积压导致文章任务不能执行，或文章任务积压导致群任务不能执行。
7. 单一全局任务表、单一全局失败计数或单一全局开关导致两条链路互相误伤。
8. 公众号/订阅号链路长时间占用微信 UI，导致核心群超过阈值无法采集且没有失败记录。
9. 持锁进程异常退出后无法自动识别和恢复僵死锁。
10. 文章链路每次被打断后都从头扫描全部公众号/订阅号。
11. Playwright 正文解析仍然持有 `wechat_ui_lock`。

## 16. 设计边界确认

本方案确认以下边界：

1. 普通微信 RPA 只能做低漏采采集，不能提供强一致消息流。
2. 微信 PC 4.1.8.107 是第一阶段唯一验证版本。
3. 生产数据不进入 Docker MySQL。
4. 第一阶段不建设前端后台。
5. AI 默认关闭，待采集和脱敏链路稳定后灰度启用。
6. 微信群消息链路和公众号/订阅号链路是两个独立业务管道，不能共用任务状态、游标状态、失败计数和启停开关。
7. 在核心群不超过 5 个、公众号/订阅号不超过 20 个的生产初版规模内，可以采用同一台主机、同一个微信 PC UI 的交错执行方案。
8. 单机交错执行不是 UI 并行执行，所有微信 UI 操作仍必须通过 `wechat_ui_lock` 串行化。
9. 如果生产要求两条链路在 UI 操作层完全互不影响，或容量超过上述规模，应拆成两个采集桌面或两台采集机。
10. 如业务未来要求审计级归档，应优先评估企业微信会话内容存档或其他官方合规接口。
