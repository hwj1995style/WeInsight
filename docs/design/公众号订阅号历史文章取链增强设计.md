# 公众号订阅号历史文章取链增强设计

## 1. 背景与技术验证结论

当前 article 链路的真实 POC 已验证出一个明确问题：`WxautoArticleRpaClient.copy_latest_article_links()` 只从 `GetAllMessage()` 返回的可见消息文本中正则提取 `mp.weixin.qq.com/s/...` 链接。对“一箱蛋”、“贵阳鸡蛋价格”、“家美鲜鸡蛋 佳美鲜”、“河北馆陶鸡蛋报价”这类公众号/订阅号，窗口可以打开，但可见消息里往往是图片、文章卡片或菜单入口，不直接暴露文本链接，因此现有实现会返回 `link_count=0`。

本次真实微信 UI 技术验证得到以下结论：

1. 环境不是 Chromium 问题：本机已有 Playwright Chromium，文章解析链路可复用本机浏览器；取链失败发生在微信 RPA 取链接阶段。
2. 账号窗口可以打开：`article-rpa-probe` 能打开“一箱蛋”，账号可识别，旧逻辑因可见消息不含文本 URL 返回 0 条链接。
3. `GetAllMessage()` 对目标账号只暴露 `FriendImageMessage` 等对象，没有可用 URL 字段。
4. “一箱蛋”底部菜单“蛋价资讯”可以进入历史文章列表；点击列表项可打开微信内置浏览器文章详情页。
5. 文章详情页右上角 `...` 菜单存在“复制链接”入口，人工截图已确认；自动化时应优先走菜单复制链接。
6. 如果菜单复制链接失败，微信内置浏览器 UIA `Value` 字段能稳定暴露 `mp.weixin.qq.com/s` 文章详情 URL；同一页面可同时出现 `/mp/profile_ext` 历史页 URL 和 `/s` 文章详情 URL，需要筛选 `/s` 且包含 `__biz/mid/idx/sn` 等文章参数的 URL。
7. 不同公众号的历史文章详情入口不一致，不能把“一箱蛋”的底部菜单坐标或菜单名写死。系统需要多策略探测、账号级路由缓存和缓存失效后自动重新探测。

结论：article 链路真实 POC 可以继续推进。增强重点不是解析 Chromium，而是增强微信内公众号/订阅号历史文章取链方式。

## 2. 目标与非目标

### 2.1 目标

1. 支持不同公众号/订阅号入口形态下的历史文章取链。
2. 本轮聚焦“真实取链增强”，让 `copy_latest_article_links(max_articles)` 能返回真实 `mp.weixin.qq.com/s` 链接，并闭环到现有 `raw -> Playwright parse -> clean -> analysis -> daily report`。
3. 对每个账号自动学习可用打开方式，并将成功路由持久化为账号级能力缓存。
4. 缓存丢失或缓存失效时可以重新 probe，不让缓存成为单点故障。
5. 继续遵守既有安全边界：真实微信 UI 操作必须持有 `wechat_ui_lock`，CLI 和日志不输出具体文章链接或正文，Playwright 解析阶段不持有微信 UI 锁。

### 2.2 非目标

1. 不做协议层抓包、逆向微信接口或绕过微信限制。
2. 不长期保存文章正文、HTML 原文、截图 OCR 原文或复制到的完整链接日志。
3. 不建设前端后台。
4. 不把所有公众号一次性扩到 20 个；仍按 1 个账号闭环、再 3 到 4 个账号小规模扩容的节奏推进。
5. 不承诺普通微信 UI RPA 达到强一致消息流，只做到小规模可观测、可重试、可回滚。
6. 本轮不实现文章内容结构化增强，不新增真实 HTML 表格抽取、图片 OCR 表格抽取或报价明细提取；这些能力作为下一阶段独立设计与任务推进。

## 4. 数据模型与权限模型

### 4.1 新增账号路由缓存表

新增 `wechat_article_route_cache`，只保存账号取链能力和路由元数据，不保存文章 URL、正文或截图内容。

```sql
CREATE TABLE IF NOT EXISTS wechat_article_route_cache (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    account_name VARCHAR(200) NOT NULL COMMENT '公众号或订阅号名称',
    route_type VARCHAR(50) NOT NULL COMMENT 'visible_text/chat_card/bottom_menu/profile_home/profile_ext',
    entry_label VARCHAR(100) NULL COMMENT '入口标签，如历史消息、全部消息、蛋价资讯',
    entry_index INT NULL COMMENT '入口序号，用于同名或无文本入口兜底',
    link_extract_type VARCHAR(50) NOT NULL COMMENT 'copy_link_menu/uia_value/visible_text',
    cache_status VARCHAR(50) NOT NULL DEFAULT 'active' COMMENT 'active/invalid/probing/failed',
    last_success_at DATETIME NULL COMMENT '最近一次按该路由成功取到文章详情链接的时间',
    last_failure_at DATETIME NULL COMMENT '最近一次按该路由失败的时间',
    failure_count INT NOT NULL DEFAULT 0 COMMENT '连续失败次数',
    last_error_code VARCHAR(100) NULL COMMENT '最近失败错误码',
    last_error_msg VARCHAR(500) NULL COMMENT '最近失败摘要，不包含文章链接或正文',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_article_route_account (account_name),
    KEY idx_article_route_status (cache_status, failure_count),
    KEY idx_article_route_success (last_success_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

缓存消失或查不到记录时，系统进入自动 probe；缓存存在但连续失败达到阈值时，标记为 `invalid` 并重新 probe。缓存只用于加速，不作为唯一事实来源。

### 4.2 静态 hint 配置

配置用于给已知账号提供优先入口，不替代自动探测。

```yaml
pipelines:
  article:
    route_cache_enabled: true
    route_probe_enabled: true
    route_probe_failure_threshold: 3
    route_entry_labels:
      - 历史消息
      - 全部消息
      - 往期文章
      - 文章
      - 资讯
      - 蛋价资讯
    link_extract_methods:
      - copy_link_menu
      - uia_value
      - visible_text
```

如后续发现个别账号必须特例，可在账号配置 `remark` 或单独配置块中加入 hint；第一阶段不把账号名写成代码分支。

### 4.3 权限与安全边界

1. 只有 article 链路在持有 `wechat_ui_lock` 时可以操作微信 UI。
2. route cache 只记录路由和状态，不记录文章具体 URL。
3. 剪贴板复制链接时必须保存用户原剪贴板，读取并校验 URL 后恢复原剪贴板。
4. CLI、日志、错误摘要、文档和验收报告不得输出完整文章 URL、正文、HTML 或截图 OCR 原文。

## 5. 后端实现方案

### 5.1 取链总流程

```text
open_public_account(account_name)
  ↓
copy_latest_article_links(max_articles)
  ↓
读取 route cache
  ↓
缓存存在且 active → 按缓存路由尝试
  ↓
成功取到 /s 链接 → 刷新 cache success → 返回链接
  ↓
缓存不存在/失败/失效 → 进入 probe
  ↓
按策略顺序尝试入口
  ↓
成功取到 /s 链接 → 写入 cache → 返回链接
  ↓
全部失败 → 返回空列表或抛出可诊断错误，由现有 collect log 记录
```

### 5.2 导航策略

策略不按账号名分支，而按入口形态分支：

1. `visible_text`
   - 复用现有 `GetAllMessage()` 正则提取。
   - 适用于聊天可见文本直接包含 `mp.weixin.qq.com/s` 的账号。

2. `chat_card`
   - 从可见消息 UIA 元素中识别文章卡片，点击进入详情。
   - 若卡片点击后进入内置浏览器，则进入统一详情取链流程。
   - 首阶段可作为低优先级策略，因为前次验证中图片消息和卡片没有稳定 URL。

3. `bottom_menu`
   - 识别公众号窗口底部菜单。
   - 依次尝试配置的入口标签，如“历史消息”、“全部消息”、“往期文章”、“蛋价资讯”等。
   - 进入历史列表后点击文章列表项，再进入详情取链流程。

4. `profile_home`
   - 点击公众号主页，再查找“全部消息/历史消息/更多文章”等入口。
   - 适用于底部没有历史入口但主页暴露历史列表的账号。

5. `profile_ext`
   - 如果当前或新打开页面已经是 `/mp/profile_ext` 历史页，直接枚举可见文章列表并进入详情。

### 5.3 详情页取链策略

详情页取链统一封装为 `extract_article_detail_url()`：

1. `copy_link_menu`
   - 点击微信内置浏览器右上角 `...`。
   - 定位菜单项“复制链接”。
   - 保存原剪贴板，执行复制，读取剪贴板，恢复原剪贴板。
   - 只接受 `mp.weixin.qq.com/s` 且包含文章参数的 URL。

2. `uia_value`
   - 枚举微信内置浏览器可访问性树。
   - 从 `Value.CurrentValue` 中提取 `mp.weixin.qq.com` URL。
   - 过滤 `/mp/profile_ext`，只保留 `/s` 详情链接。
   - 该路径不依赖剪贴板，是复制菜单失败时的稳定兜底。

3. `visible_text`
   - 如果详情页或消息对象中直接出现 URL，则正则提取。
   - 作为兼容兜底。

### 5.4 多文章采集

首轮真实 POC 以 `max_articles_per_round=1` 为主。实现结构需支持 `max_articles > 1`：

1. 历史列表页按可见顺序点击文章项。
2. 每进入一篇详情页，提取一个 `/s` URL。
3. 返回历史列表或关闭当前文章 tab，再处理下一篇。
4. 达到 `max_articles` 后立即停止，释放微信 UI。
5. 如果回退失败，不继续乱点，记录失败并释放 UI。

## 6. 前端影响

无前端变更。当前项目没有前端后台，运维仍通过 CLI、Markdown runbook 和 MySQL 指标查询完成。

## 7. 测试与回归方案

### 7.1 单元测试

1. URL 筛选：
   - 输入同时包含 `/mp/profile_ext` 和 `/s` 的 UIA Value 文本。
   - 期望只返回 `/s` 且带 `__biz/mid/idx/sn` 的文章详情 URL。

2. 复制链接菜单：
   - 使用 fake clipboard 验证复制前保存原剪贴板、读取后恢复原剪贴板。
   - URL 不合法时返回空并恢复剪贴板。

3. 路由缓存 repo：
   - `upsert_success()` 写入 active route。
   - `mark_failure()` 增加 failure_count。
   - 达到阈值后标记 invalid。
   - SQL 不访问 group 表。

4. RPA 策略顺序：
   - 有 active cache 时优先走缓存。
   - 缓存失败后进入 probe。
   - probe 成功后写回 cache。
   - 缓存不存在时直接 probe。

5. 敏感输出：
   - `article-rpa-probe` 输出不包含 `mp.weixin.qq.com`、文章正文、HTML 或剪贴板内容。

### 7.2 集成与真实 POC 验证

1. fake RPA 集成测试覆盖 `ArticleCollectService` 不变接口。
2. 本机真实微信人工值守验证顺序：
   - `wechat-health`
   - `article-rpa-probe --account-name "一箱蛋"`
   - `collect-article-once --account-name "一箱蛋" --max-articles-per-round 1`
   - `parse-article-once --limit 5`
   - `article-runtime-metrics --hours 24`
3. 单账号通过后，再验证：
   - “贵阳鸡蛋价格”
   - “家美鲜鸡蛋 佳美鲜”
   - “河北馆陶鸡蛋报价”
4. 每个账号记录 route type、link extract type、是否命中 cache、是否重新 probe。

### 7.3 回归重点

1. 群链路命令和状态不受 article route cache 影响。
2. `wechat_ui_lock` 仍能释放，核心群到期时 article 在安全检查点中断。
3. Playwright 解析继续在释放微信 UI 后执行。
4. CLI 和日志继续不输出真实文章 URL。
5. 缓存表丢失或清空后，下一轮可重新 probe 并恢复。

### 7.4 真实 POC 成功判定收紧

真实 POC 不能只以 `collect-article-once` 命令退出码或 `success_count=1` 作为闭环依据。单账号单篇文章验收必须同时满足：

1. `article-rpa-probe` 能打开账号窗口，并且取链路径没有返回失败状态。
2. `collect-article-once` 输出 `link_count >= 1`。
3. `collect-article-once` 输出 `raw_insert_count + duplicate_count >= 1`，证明真实链接已经进入 raw 入库或命中去重。
4. `task_created_count >= 1` 或后续 `parse-article-once` 有可处理任务；如果只命中 duplicate，需要明确说明是重复链接导致没有新任务。
5. `parse-article-once`、`article-runtime-metrics` 能继续运行，且 CLI 输出不包含具体文章 URL、正文、HTML 或 OCR 原文。

当 `link_count=0` 或 `raw_insert_count + duplicate_count=0` 时，即使 RPA 过程没有抛异常，也必须按采集失败记录，不得把本轮真实 POC 判为通过。

### 7.5 公众号窗口打开稳定性回归

2026-07-11 管理后台首次真实 POC 表明，公众号窗口打开和文章详情窗口打开是两个独立检查点：首次任务可以识别账号但未打开文章详情；后续独立探针又出现账号窗口未稳定可见。为避免把 `ChatWith()` 的同步返回误判为 UI 已就绪，本轮补充以下回归：

1. `ChatWith()` 返回后，客户端必须在有限等待窗口内轮询目标公众号窗口；窗口变为可见时不得触发主窗口搜索兜底。
2. 等待窗口超时后，才执行已有的主窗口本地搜索兜底；兜底成功后同样必须等待目标窗口可见。
3. 当前微信客户端显示“搜索网络结果”入口，但其稳定选择器尚未在受控 POC 中验证；本轮不把点击网络搜索入口写入自动化逻辑，也不将其作为真实 POC 通过条件。
4. 探针和采集任务继续只输出安全状态、计数和错误码；不得输出搜索结果、文章链接、正文或截图内容。

## 8. 风险与分阶段落地建议

### 8.1 主要风险

| 风险 | 等级 | 缓解 |
| --- | --- | --- |
| 不同公众号入口差异大 | 高 | 多策略 probe + 账号级 route cache + 静态 hint 兜底 |
| 微信 UIA 树或菜单文本抖动 | 高 | copy_link_menu 和 uia_value 双路径；失败后重新 probe |
| 公众号不在本地会话且仅显示网络搜索入口 | 高 | 先等待 `ChatWith()` 对应窗口；仅保留已验证的本地搜索兜底；未验证的网络搜索入口须单独受控 POC 后再接入 |
| 剪贴板污染用户内容 | 中 | 复制前保存、读取后恢复；异常 finally 中恢复 |
| 文章链路持有 UI 过久 | 高 | 默认单账号、单篇文章；达到 max 后立即释放；核心群安全检查点中断 |
| 缓存丢失 | 中 | 缓存不是事实源；缓存缺失触发自动 probe |
| CLI 泄露真实链接 | 高 | 输出只显示计数、route 类型和状态，不输出 URL |

### 8.2 分阶段落地

第一阶段：设计与测试骨架

1. 落地本设计文档。
2. 新增 SQL migration 和 schema 测试。
3. 新增 route cache repo 测试。
4. 新增 URL 筛选和剪贴板保护单元测试。

第二阶段：策略框架

1. 在 RPA 层引入路由策略抽象。
2. 保留现有 visible text 策略。
3. 加入 cache 命中、失败、重新 probe 的控制流。
4. fake 测试验证接口不影响 `ArticleCollectService`。

第三阶段：真实 UI 能力

1. 实现 copy_link_menu。
2. 实现 uia_value URL 兜底。
3. 实现 bottom_menu/profile_ext 的最小可用路径。
4. 在真实微信中验证“一箱蛋”单篇文章闭环。

第四阶段：小规模扩容

1. 对 4 个授权账号逐一运行 `article-rpa-probe`。
2. 对每个账号记录 route cache。
3. 成功后执行单账号 `collect-article-once + parse-article-once`。
4. 通过后再考虑 `run-article-scheduler --once` 的人工值守小窗口。

第五阶段：内容结构化增强，另行设计

1. 基于已稳定入库的真实文章链接，单独设计 HTML 表格抽取、图片 OCR 表格抽取和报价明细结构化。
2. 只在运行时短暂读取正文、HTML、图片和 OCR 原文；落库仍只保存摘要、结构化表格、价格项和质量指标。
3. 该阶段不与真实取链增强混在同一次 POC 验收中，避免扩大本轮微信 UI 风险。

## 9. 设计确认点

本方案建议确认以下边界后再进入编码：

1. 新增 `wechat_article_route_cache` 表，只保存路由能力，不保存文章链接。
2. 默认启用 copy_link_menu，失败后用 uia_value 兜底。
3. route cache 丢失或失效后自动 probe，不要求人工重建。
4. 首轮真实落地仍只跑单账号、单篇文章；4 个账号逐个扩容。
5. 采用两步走：本轮只增强真实取链并闭环现有后续链路；`raw`、Playwright parse、`clean`、`analysis`、`daily report` 的职责和主流程不重构，仅允许真实链接入库所需的最小兼容调整。HTML 表格、图片 OCR 表格和报价明细结构化作为下一阶段独立设计。
